# Mirage Project 二元架构校准文档

## 架构定位

Mirage Project 采用**二元架构**：

- **Mirage-Gateway**（手脚）：前线战斗单元，负责流量伪装、威胁检测、多路径传输
- **Mirage-OS**（大脑）：控制中心，负责全球调度、生死裁决、威胁情报聚合

---

| 冲突维度 | 架构文档（蓝图）定义 | 实现文档（战报）现状 | 校准结论（以此为准） |

## 三大核心对齐动作

### 1. 协议栈的"名实对齐"

我们将协议功能进行物理整合，避免代码中出现重叠逻辑：

| 协议 | 维度 | 核心职责 | 实现位置 | 状态 |
|------|------|---------|---------|------|
| **VPC** | Background | 入站（Ingress）威胁检测 + 噪声注入 + 内容伪装 | bpf/jitter.c + pkg/strategy/vpc_content.go | ✅ 已实现 |
| **Jitter-Lite** | Time | 出站（Egress）IAT 随机化 + NPM Padding | bpf/jitter.c (jitter_egress) | ✅ 已实现 |
| **G-Tunnel** | Space | FEC 纠错 + 多路径切换 + 转生协议 | pkg/gtunnel/ | ✅ 已实现 |
| **NPM** | Morph | 作为 Jitter 的子功能，通过 eBPF 修改包长度 | 整合入 Jitter-Lite | ✅ 已整合 |
| **B-DNA** | Fingerprint | B-DNA 模板 + 拟态延迟采样 | pkg/strategy/dna_*.go + bpf/jitter.c | ✅ 已实现 |
| **Chameleon** | Protocol | TLS/QUIC 指纹重写 + JA4 伪装 | pkg/strategy/chameleon.go + bpf/chameleon.c | ✅ V2 新增 |
| **G-Switch** | Survival | 域名转生（5 秒平滑切换） | Mirage-OS 控制 | ✅ 已实现 |

**核心原则**：
- VPC = 入站防御（威胁检测 + 噪声注入）
- Jitter-Lite = 出站伪装（IAT 随机化 + Padding）
- G-Tunnel = 空间逃逸（FEC + 多路径）
- NPM 和 B-DNA 作为子功能整合，避免代码重复

### 2. 生死裁决逻辑的"硬对齐"

#### Gateway 逻辑（内核态优先）

```c
// bpf/jitter.c - 内核态配额检查（第一优先级）
SEC("tc")
int jitter_egress(struct __sk_buff *skb) {
    // 1. 配额检查（最高优先级）
    __u32 key = 0;
    __u64 *quota = bpf_map_lookup_elem(&quota_map, &key);
    if (quota && *quota == 0) {
        return TC_ACT_STOLEN;  // 立即阻断，无视所有 Jitter 逻辑
    }
    
    // 2. 威胁等级检查
    __u32 *level = bpf_map_lookup_elem(&defense_level_map, &key);
    if (!level) return TC_ACT_OK;
    
    // 3. 根据威胁等级应用 Jitter
    __u32 jitter_ns = calculate_jitter(*level);
    skb->tstamp += jitter_ns;
    
    // 4. 流量统计
    update_traffic_stats(skb->len);
    
    return TC_ACT_OK;
}
```

#### OS 逻辑（事务原子性）

```go
// services/api-gateway/server.go - 生死裁决
func (s *Server) SyncHeartbeat(ctx context.Context, req *pb.HeartbeatRequest) (*pb.HeartbeatResponse, error) {
    // 事务必须包含三步（原子性）
    tx := s.db.Begin()
    defer tx.Rollback()
    
    // 1. Update User Balance
    var user models.User
    tx.First(&user, "id = ?", req.UserId)
    
    // 2. Deduct Quota（流量结算）
    cost := calculateCost(req.TrafficBytes, req.DefenseBytes)
    user.RemainingQuota -= cost
    tx.Save(&user)
    
    // 3. Insert Billing Log
    log := models.BillingLog{
        UserID:       req.UserId,
        TrafficBytes: req.TrafficBytes,
        Cost:         cost,
        Timestamp:    time.Now(),
    }
    tx.Create(&log)
    
    tx.Commit()
    
    // 4. 熔断决策
    newQuota := user.RemainingQuota
    if newQuota <= 0 {
        newQuota = 0  // 触发内核态 TC_ACT_STOLEN
    }
    
    return &pb.HeartbeatResponse{
        RemainingQuota: newQuota,
        DefenseLevel:   calculateDefenseLevel(user),
    }, nil
}
```

**硬对齐原则**：
- 内核态 quota_map 是第一优先级
- OS 下发 NewRemainingQuota == 0 时，内核必须无视所有 Jitter 逻辑
- 事务必须包含：Update User Balance → Deduct Quota → Insert Billing Log

### 3. 全球视野的"坐标对齐"

#### 实现缺陷

目前 Gateway 上报的 IP 缺乏经纬度，前端 3D 地球无法定位。

#### 校准方案

Mirage-OS 必须集成 MaxMind GeoIP 模块：

```go
// services/ws-gateway/geoip.go - IP 定位模块
package wsgateway

import (
    "net"
    "github.com/oschwald/geoip2-golang"
)

type GeoIPService struct {
    db *geoip2.Reader
}

func NewGeoIPService(dbPath string) (*GeoIPService, error) {
    db, err := geoip2.Open(dbPath)
    if err != nil {
        return nil, err
    }
    return &GeoIPService{db: db}, nil
}

func (g *GeoIPService) Locate(ipStr string) (lat, lng float64, country, city string, err error) {
    ip := net.ParseIP(ipStr)
    record, err := g.db.City(ip)
    if err != nil {
        return 0, 0, "", "", err
    }
    
    return record.Location.Latitude,
           record.Location.Longitude,
           record.Country.Names["en"],
           record.City.Names["en"],
           nil
}
```

#### 集成到 SyncHeartbeat

```go
// services/api-gateway/server.go - 自动换算经纬度
func (s *Server) SyncHeartbeat(ctx context.Context, req *pb.HeartbeatRequest) (*pb.HeartbeatResponse, error) {
    // ... 生死裁决逻辑 ...
    
    // 自动换算经纬度
    lat, lng, country, city, err := s.geoip.Locate(req.SrcIp)
    if err == nil {
        // 推送到 Redis Pub/Sub
        event := map[string]interface{}{
            "type":      "gateway",
            "lat":       lat,
            "lng":       lng,
            "country":   country,
            "city":      city,
            "status":    "online",
            "timestamp": time.Now().Unix(),
        }
        s.redis.Publish(ctx, "mirage:events:all", event)
    }
    
    return &pb.HeartbeatResponse{...}, nil
}
```

**坐标对齐原则**：
- Gateway 只上报 IP，不负责定位
- OS 后端自动根据 IP 换算经纬度
- 通过 Redis Pub/Sub 推送到 WebSocket
- 前端 3D 地球实时定位

## 热密钥缓存机制（性能优化）

### 问题

Shamir 3-of-5 恢复需要：
1. 从 3 个节点收集份额（网络延迟）
2. 执行拉格朗日插值（计算延迟）
3. 验证密钥有效性

总延迟可能超过 5 秒，影响 G-Switch 转生速度。

### 解决方案：热密钥常驻内存

**核心原则**：
- 正常运行时：主密钥常驻内存（热密钥模式）
- 节点重启时：执行 Shamir 恢复（冷启动模式）
- 节点迁移时：执行 Shamir 恢复（冷启动模式）

**实现位置**：
- `mirage-os/pkg/crypto/backup.go` - 热密钥管理
- `mirage-os/pkg/raft/hot_key_manager.go` - 热密钥管理器

**工作模式**：

```
┌─────────────────────────────────────────────┐
│          热密钥生命周期                      │
├─────────────────────────────────────────────┤
│                                              │
│  1. 节点启动                                 │
│     └─ 检查热密钥状态                        │
│        ├─ 已激活 → 零延迟模式 ✅             │
│        └─ 未激活 → 冷启动 🥶                 │
│                                              │
│  2. 冷启动（仅节点重启/迁移时）              │
│     ├─ 从 Raft 集群收集 3 个份额             │
│     ├─ 执行 Shamir 恢复                      │
│     ├─ 激活热密钥                            │
│     └─ 耗时: ~2-3 秒                         │
│                                              │
│  3. 正常运行（热密钥模式）                   │
│     ├─ 主密钥常驻内存                        │
│     ├─ 加密/解密零延迟                       │
│     ├─ G-Switch < 5 秒 ✅                    │
│     └─ 域名转生无需恢复密钥                  │
│                                              │
│  4. 定期刷新（可选）                         │
│     ├─ 每 24 小时轮换密钥                    │
│     ├─ 停用旧密钥 → 生成新密钥               │
│     └─ 激活新密钥                            │
│                                              │
│  5. 节点关闭                                 │
│     ├─ 停用热密钥                            │
│     ├─ 清空内存                              │
│     └─ 份额保持分布式存储                    │
│                                              │
└─────────────────────────────────────────────┘
```

**性能对比**：

| 场景 | 冷启动模式 | 热密钥模式 |
|------|-----------|-----------|
| 密钥恢复 | 2-3 秒 | 0 ms |
| 加密操作 | 需要先恢复 | 即时 |
| 解密操作 | 需要先恢复 | 即时 |
| G-Switch | 可能超时 | < 5 秒 ✅ |
| 内存占用 | 0 MB | 32 字节 |

**安全性保障**：

1. **内存保护**：
   - 使用 mlock 锁定内存页
   - 防止 swap 到磁盘
   - 进程终止时自动清零

2. **访问控制**：
   - 只有 Leader 节点持有热密钥
   - Follower 节点只存储份额
   - 节点切换时自动转移

3. **故障恢复**：
   - 节点崩溃：热密钥丢失，从份额恢复
   - 网络分区：Leader 降级，清空热密钥
   - 物理查封：自毁程序清空内存

**代码示例**：

