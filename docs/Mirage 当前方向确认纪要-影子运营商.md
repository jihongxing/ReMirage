# Mirage 当前方向确认纪要：影子运营商路线

## 一、文档性质

本文档不是普通需求文档，也不是某个单模块的能力清单。

它用于保存本轮关于 Mirage 项目定位、商业边界、技术收敛方向和后续设计原则的核心讨论结果，作为后续产品设计、架构取舍、能力裁剪和实施排期的**方向依据**。

这份纪要之所以单独存放在 `docs` 根目录，是因为它不是某一个子系统的说明，而是当前整个项目的路线确认文件。

---

## 二、最终确认的路线

Mirage 当前不再朝以下两个方向扩张：

### 2.1 不做低价消费级 VPN 市场

原因不是“做不出”，而是这条路与 Mirage 的初衷和长期形态不一致。

低价 VPN 市场的典型特征是：

- 依赖大规模用户
- 依赖高频分发和低价订阅
- 合规压力和经营风险更高
- 获客、封锁、恢复之间存在持续军备竞赛
- 业务价值更容易被压缩成“便宜、能用、节点多”

这与 Mirage 想要建立的高价值、小规模、强控制、低暴露模式不一致。

### 2.2 不做 B2B2B 底层能力平台

Mirage 也不再把自己定义为：

- 其他连接服务商的底座
- VPN 品牌的白标基础设施
- 托管网络服务商的 OEM 平台
- 面向伙伴生态的开放能力平台

原因不是这条路不成立，而是：

- 它会显著扩大系统暴露面
- 会迫使项目长期进入平台化和生态化投入
- 会要求持续适配不同合作方、不同交付形态、不同商业模式
- 会把 Mirage 拉向一个更重的、长期军备竞赛式的方向
- 会稀释 Mirage 当前最重要的优势：**小规模、高控制、低可见度**

因此，这条路被认定为“理论上可行，但当前不采纳”的备选方向，而不是现行路线。

---

## 三、Mirage 的当前定位

Mirage 当前应当明确定位为：

**一个服务极少量高价值客户的影子运营商。**

更具体地说：

- 不追求大众市场
- 不追求公开扩张
- 不追求做成平台生态
- 不追求把能力暴露给外部伙伴
- 不追求占领“VPN 市场”

Mirage 的目标是：

**只服务少量、明确、可控、高价值的客户群体，维持极高的连通质量、恢复能力和运营隐蔽性。**

这个客户规模的目标，不是几万，不是几千，而是：

**大约 200 个左右的高价值客户。**

这 200 个客户不是“买月费的普通用户”，而是：

- 对稳定性高度敏感的人
- 对中断成本敏感的人
- 对链路恢复能力有真实需求的人
- 愿意为可用性、隔离性、韧性和持续服务买单的人

---

## 四、这条路线背后的核心判断

本轮讨论最终踩中的，不是一个技术点，而是一整套方法论。

### 4.1 Mirage 的真正优势，不在“更大”，而在“更隐”

Mirage 的价值，不应该通过：

- 做更公开的平台
- 承接更多客户类型
- 输出更多外部接口
- 做更大的市场叙事

来体现。

Mirage 更适合的方式是：

- 保持技术先进性
- 控制技术暴露范围
- 控制客户规模
- 控制运营半径
- 控制系统暴露面

就像流量本身一样：

**隐藏在平常的生意里，维持低存在感，但保有极强的真实能力。**

### 4.2 项目成功的关键，不是扩张，而是收敛

Mirage 当前最重要的不是“再找一个更大的商业故事”，而是把现有技术资产收敛成一套：

- 能交付
- 能收费
- 能运营
- 能恢复
- 能隐身
- 能长期活下去

的小规模高价值服务。

### 4.3 技术先进性必须为运营隐蔽性服务

Mirage 不是为了展示技术，而是为了在现实环境中长期生存。

因此，所有技术建设都应当接受一个新约束：

**是否有助于 Mirage 在更小的暴露面下，持续稳定地服务那 200 个客户。**

如果答案是否定的，那么即使这项能力很“强”，也不应该优先建设。

---

## 五、Mirage 不再是什么

为了防止路线回摆，必须明确写下 Mirage 当前**不是什么**。

Mirage 不是：

- 一个面向大众的 VPN 产品
- 一个追求海量订阅用户的消费级工具
- 一个面向伙伴开放能力的白标/OEM 平台
- 一个为其他服务商提供底层基础设施的生态型底座
- 一个试图覆盖完整 SASE / 企业安全全栈的大而全平台
- 一个以“更多功能、更多界面、更多可见能力”为目标的公开系统

这些方向都可能成立，但它们不是 Mirage 当前要走的路。

---

## 六、Mirage 现在应该成为什么

Mirage 应当成为：

### 6.1 一个高价值客户的小规模连接服务

它的成功标准不是用户数量，而是：

- 客户质量
- 客单价
- 存活率
- 恢复能力
- 运营控制力

### 6.2 一个低暴露面的运营系统

它的成功标准不是“看起来很强”，而是：

- 外部暴露少
- 控制面收口
- 入口少且可控
- 不轻易被识别为一个大规模公开系统

### 6.3 一个小团队能持续维护的业务

它的成功标准不是组织扩张，而是：

- 一个小团队也能运行
- 问题出现时能快速定位和处置
- 不需要持续堆大规模平台化投入
- 不需要进入公开生态竞争

### 6.4 一个隐身在普通生意里的技术系统

Mirage 的存在方式，不应该是高调占领市场，而应该是：

**保持技术领先，但把业务存在感压到尽可能低。**

---

## 七、后续设计必须遵守的五条原则

### 7.1 收敛优先于扩张

任何新设计都必须先问：

- 这会不会扩大 Mirage 的暴露面？
- 这会不会让项目走向平台化？
- 这会不会把客户类型拉得太散？
- 这会不会增加长期不可控投入？

如果会，就应该谨慎甚至直接裁掉。

### 7.2 控制力优先于规模

Mirage 的价值来自控制力，而不是规模。

因此设计时优先保证：

- 谁能接入
- 谁在使用
- 谁消耗了多少资源
- 谁出了问题
- 哪个节点承载了谁
- 什么时候该切换、该降级、该熔断

### 7.3 运营隐蔽性优先于技术展示

所有“更酷、更炫、更公开”的东西，都应该让位于：

- 更低暴露
- 更少入口
- 更少可见面
- 更少不必要交互

### 7.4 小团队可持续优先于大平台想象力

Mirage 的设计必须适配现实：它不是一个靠大团队堆出来的平台项目。

因此所有建设都应该支持：

- 极简操作
- 极简部署
- 极简故障恢复
- 极简后台
- 极简流程

### 7.5 用户无感优先于复杂可玩性

对 Mirage 的客户来说，最重要的不是“有多少模式可选”，而是：

**它是否在后台稳定工作，让客户几乎不用操心。**

---

## 八、围绕这条路线，Mirage 必须具备哪些能力

接下来项目需要建设的，不是平台能力，而是**影子运营商所必需的运营能力**。

建议把后续能力建设收敛到以下八条主线。

### 8.1 严格准入与身份绑定能力

Mirage 不是开放注册产品，因此必须具备：

- 邀请制开通
- 用户身份绑定
- 设备或公钥绑定
- 用户、设备、会话三层关联
- 风险账户冻结与摘除能力

重点不是“如何让更多人进来”，而是“如何确保只有该进来的人能进来”。

### 8.2 精确的计费、配额与服务分层能力

Mirage 的服务卖点不是便宜流量，而是：

- 稳定性
- 恢复速度
- 资源优先级
- 隔离能力
- 不同等级的服务体验

因此系统必须支持：

- 按用户/会话精确归属流量
- 精确配额熔断
- 不同用户的服务等级差异
- 到期、欠费、封禁时的受控降级

### 8.3 稳定的 OS 指挥中心能力

Mirage-OS 不应该扩张成开放平台，而应该成为内部作战后台。

它必须能做到：

- 真实掌握 Gateway 拓扑
- 真实掌握 User / Client / Gateway / Session 归属
- 按 Gateway、按 Cell、按用户集合下推策略
- 统一计费、封禁、切换、熔断
- 支持极简但高效的人工干预

### 8.4 高可用但低可见的 Gateway 承载能力

Gateway 仍然是 Mirage 的前线单元，但建设目标要更明确：

- 默认拒绝非正常流量
- 明确拒绝未授权控制面
- 支持分层策略执行
- 支持多用户承载但不混淆归属
- 支持低暴露、高恢复、受控切换

重点不是继续做更多概念型模块，而是让现有能力真正形成稳定闭环。

### 8.5 用户无感的 Client 常驻与自愈能力

Client 必须最终变成：

- 后台常驻
- 自动建链
- 自动学习拓扑
- 自动切换 Gateway
- 自动处理订阅和配额状态
- 路由与连接始终一致

Mirage 的客户不应该长期盯着终端、手动贴 token、手动重连。

### 8.6 运营级观测与人工干预能力

Mirage 不需要一个炫技式大屏，但必须有一个极简、高效、足够“作战”的运营后台。

这个后台至少要能看到：

- 当前在线用户
- 当前 Gateway 负载
- 当前告警
- 当前高风险来源
- 当前服务等级与配额状态
- 当前切换和恢复事件

同时要支持最关键的人工动作：

- 封禁 / 解封
- 暂停用户
- 强制切换
- 切保守模式
- 摘除故障 Gateway

### 8.7 低暴露的基础设施与资产运营能力

Mirage 必须具备一种“尽量不显山露水”的运营方式。

因此系统层面还必须支持：

- 控制面收口
- 内部接口最小暴露
- 入口与资产统一管理
- 节点与域名轮换策略
- 风险出现时进入更保守运行态
- 不把系统做成一个明显的大规模公开基础设施

### 8.8 小团队可持续交付能力

如果 Mirage 真的只做 200 个客户，那么系统设计必须适配小团队长期维护。

这意味着必须有：

- 标准化开通流程
- 标准化节点上线流程
- 标准化回滚和恢复流程
- 最小测试与回归体系
- 极简内部工具链
- 尽量少的手工操作

## 九、现阶段最该优先补的能力

结合当前代码审计和本轮路线确认，Mirage 接下来最优先的，不是扩大范围，而是把下面六件事做实。

### 9.1 做实 User / Client / Gateway / Session 真相链

这是整个运营模型的基础。

如果不知道“谁正在通过哪个 Gateway 使用哪个会话”，那么：

- 计费不准
- 配额不准
- 风险隔离不准
- 运维判断不准

### 9.2 做实 OS 到多 Gateway 的控制闭环

Mirage-OS 必须真正能：

- 识别 Gateway
- 管理 Gateway
- 下推 Gateway
- 熔断 Gateway
- 摘除 Gateway

而不是只收心跳和日志。

### 9.3 做实 Client 后台化与无感自愈

Client 还停留在工具态，就无法支撑“高价值客户的低打扰体验”。

### 9.4 做实服务等级和配额的运行时生效

Mirage 的商业模式不是靠概念，而是靠系统行为真正体现差异化。

### 9.5 做实默认拒绝的安全模型

所有旁路入口、未授权入口、明文回退、可伪造控制链路，都必须尽快收口。

### 9.6 做实极简但高效的内部运营后台

Mirage 不需要大平台控制台，但必须有足够强的内部作战界面。

---

## 十、明确不应该再做什么

为了避免后续又回到扩张路径，必须明确写下当前阶段不建议做的事情。

当前不建议继续投入：

- 面向大众市场的低价 VPN 方案
- 面向伙伴生态的开放 API 平台
- 白标 / OEM 能力
- 复杂套餐体系
- 大而全的 B2B2B 平台能力
- 过度可视化和炫技式后台
- 超出 200 客户模型之外的规模化想象

这些方向并不是永远错误，但它们不是 Mirage 当前阶段应该承接的事情。

---

## 十一、这份纪要对后续工作的约束

从这份纪要开始，Mirage 后续的产品设计、架构设计和文档输出，都应当默认服从以下约束：

### 11.1 默认客户模型

默认按：

**“一个小团队运营一套低暴露、高控制、高韧性的系统，服务约 200 个高价值客户”**

这个模型来设计。

### 11.2 默认产品目标

默认优先建设：

- 运营能力
- 控制能力
- 隐蔽能力
- 恢复能力
- 小团队可持续能力

而不是平台扩张能力。

### 11.3 默认文档输出方向

后续再写任何：

- 路线图
- 能力清单
- 版本规划
- 商业设计
- 架构整改

都应当以这份纪要为上位约束，而不是回到大众市场或平台市场叙事。

---

## 十二、总结

这轮讨论最终确认的，不是一个普通市场定位，而是一种更贴近 Mirage 本质的生存方式：

**不做大，不做广，不做公开平台，不做大众市场。**

而是：

**保持技术先进，保持暴露面低，保持客户少而精，保持业务隐藏在普通生意之中。**

Mirage 的目标不是成为“最大的连接服务”，而是成为：

**最不显眼、最可控、最能活下去的那一类高价值连接服务。**

这就是 Mirage 当前被正式确认的路线。
