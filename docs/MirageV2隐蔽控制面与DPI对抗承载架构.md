\# Mirage V2 隐蔽控制面与 DPI 对抗承载架构 (Stealth Control Plane RFC)

\*\*版本:\*\* v2.0.1-OpSec

\*\*领域:\*\* 流量混淆 (Traffic Obfuscation)、侧信道隐写 (Side-Channel Steganography)、抗特征分析 (Anti-Fingerprinting)



\## 0. 设计共识：放弃“造新轮子”

在暗网对抗的最高维度，\*\*任何“专有”的协议、任何“自定义”的报文头部，都是极其致命的指纹。\*\* 真正的隐蔽不是“创造一种无法破解的加密”，而是“伪装成互联网上最平庸、最庞大的背景噪声”。控制语义的承载必须严格寄生于现有的标准加密协议内部，实现物理特征上的“大象无形”。



\---



\## 1. 核心理论：密码学混沌与 AEAD 视界黑洞

要让外围侦测看起来“只是一团混沌”，必须绝对依赖现代传输层安全协议（TLS 1.3 / QUIC）的 AEAD（认证加密与附加数据）特性。



\### 1.1 AEAD 的物理现实

在 QUIC 协议完成初始握手（1-RTT 之后）进入数据传输阶段后，其所有应用层数据帧（包括 Stream ID、Offset、Length 以及实际 Payload）\*\*全部被完全加密\*\*。

\* \*\*DPI 的视角：\*\* 外部审查设备只能看到 UDP 报文的源目 IP、端口和外层 QUIC Header（极少量的明文，仅用于路由和连接 ID），报文载荷表现为完美的\*\*高信息熵（Entropy）真随机乱码\*\*。

\* \*\*防御结论：\*\* 我们绝不在外层 TCP/UDP 报文或 QUIC 甚至 TLS 握手阶段附加任何自定义字节。所有的控制语义，必须发生在 AEAD 的“视界内部”。

\## 2. 承载方案 A：QUIC 隐蔽流多路复用 (Shadow Stream Multiplexing)

这是最高效、最正统的控制语义承载方式，利用 QUIC 原生的多路复用（Multiplexing）机制。



\### 2.1 架构设计

\* \*\*通道隔离：\*\* 在底层的单条 QUIC Connection 中，约定 `Stream ID 0`（或特定的单向控制 Stream）为“特权控制神经”。

\* \*\*业务承载：\*\* 用户的实际代理数据、网页浏览流量运行在 `Stream 1 \~ N`。

\* \*\*控制注入：\*\* 当发生 `Persona Flip`、`Budget Sync` 或 `Survival Mode Change` 时，将序列化后的 Protobuf 控制指令写入 `Stream 0`。



\### 2.2 隐匿性分析 (Anti-DPI)

\* \*\*无新增指纹：\*\* 对 GFW 而言，它根本不知道这个 UDP 包里包含的是 `Stream 0` 的控制指令，还是 `Stream 4` 的一段加密视频数据。它看起来仅仅是该连接内部数据吞吐量发生了微小的波动。

\* \*\*抗协议清洗：\*\* 只要 GFW 允许 QUIC / HTTP/3 流量通行，控制面就绝对畅通。它不需要像传统代理那样去费尽心机伪装 HTTP Header。

\## 3. 承载方案 B：废包隐写术 (Steganographic Dummy Payloads)

这是一种极其极致的侧信道（Side-Channel）隐写方案，适用于对“流量时序和包长”进行极限微调的场景，真正做到“控制流不增加任何额外带宽特征”。



\### 3.1 架构设计

Mirage V1 已有 NPM（网络填充模块），会基于马尔可夫链发送真随机数的废包（Dummy Packets）以伪装流量形态。我们将这些“废包”变为控制载体。



\* \*\*劫持填充流：\*\* 当控制层有紧急指令（如：入口即将被封锁，立即切换备用 IP）发出时，拦截 NPM 即将发出的下一个废包。

\* \*\*负载替换：\*\* 将废包内部的真随机数替换为 `\[HMAC(Magic\_Header) + Ciphertext\_Control\_Command + Random\_Padding]`。整个长度必须严格对齐 NPM 原本计划发送的废包长度。

\* \*\*隐写提取：\*\* 接收端在丢弃废包前，先对前 N 个字节进行 HMAC 校验。若命中，则提取控制指令；若未命中，则作为正常废包丢弃。



\### 3.2 隐匿性分析 (Anti-ML Profiling)

\* \*\*零特征畸变：\*\* 控制指令完美融入了 NPM 生成的包长分布模型（Length Distribution Model）中。基于机器学习（ML）的流量分类器统计出的包长均值、方差毫无变化。

\* \*\*时序隐形：\*\* 因为借用的是本来就要发送的废包，控制指令的下发不会改变包间间隔（IAT, Inter-Arrival Time），时序指纹（Timing Fingerprint）被完美保护。

\## 4. 极端对抗：防御 GFW 的流量降维分析

即使我们把控制流藏进了 QUIC Stream 或废包里，高级别的审计者依然可能通过\*\*“降维分析”\*\*来推测系统行为。必须采取以下对抗措施：



\### 4.1 防御状态切换的“流量潮汐”

\* \*\*致命破绽：\*\* 当 Go 控制面下发“切换 Persona”指令时，虽然指令本身是加密的，但切换后，连接的包长分布模型瞬间发生剧烈改变（例如从类似 WebRTC 的高频小包，瞬间变成类似 Netflix 的低频大包）。这种\*\*“状态突变”\*\*极易被机器学习模型捕捉，标记为“疑似代理协议切换”。

\* \*\*承载设计（平滑过渡机制）：\*\* 控制语义中必须包含 `Transition\_Duration`（过渡时间）。接收端在解析到切换指令后，\*\*禁止原子性生硬切换\*\*包长/时序模型，而是必须在设定的秒数内，通过数学插值的方式，让马尔可夫链的概率矩阵平滑过渡到新模型。



\### 4.2 恒定时间锁与响应混淆

\* \*\*致命破绽：\*\* 处理业务数据和处理控制指令，CPU 的计算耗时不同。如果 DPI 在骨干网进行微秒级抓包，计算请求到响应的 RTT，会发现处理控制包的 RTT 比普通包长 100 微秒，从而暴露“这是一个特殊报文”。

\* \*\*承载设计：\*\* 处理所有隐蔽流（Stream 0）或隐写废包的逻辑，必须引入硬件时钟对齐。强行用 Spin-loop 将处理耗时对齐到固定值，消除基于 RTT 的侧信道信息泄漏。



\## 5. 总结：黑暗森林中的生存准则
