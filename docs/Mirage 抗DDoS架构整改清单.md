# Mirage 抗DDoS 架构整改清单

## 一、文档目标

本文档用于固化本轮关于 Mirage 抗 DDoS 架构的共识，并将讨论结果整理为正式整改清单。

这份清单不再把问题理解为“如何把单个节点武装到牙齿”，而是回到 Mirage 已经明确的系统定位：

- `OS` 是不应暴露在公网的大脑
- `Gateway` 是可耗材、可摘除、可替换的公网触手
- `Client` 是具备持续学习与被动自愈能力的前线终端

因此，Mirage 的抗 DDoS 目标不是“每个真实 IP 永远打不死”，而是：

- 小到中等规模攻击时，节点尽量扛住
- 遭遇体积型攻击时，节点可以被放弃，但整体服务不能被拖死
- 控制面、路由面、客户端自愈链路必须保证系统能够重组

---

## 二、先说结论

本轮评审确认了三个关键判断。

### 2.1 不能神化单节点防护

如果攻击流量已经超过节点上游物理带宽，单机软件防御无解。

这类情况下：

- `eBPF`
- `XDP`
- `TC`
- 用户态代理
- 更复杂的策略参数

都无法改变“链路先被塞满，包还没到应用就已经死掉”的物理事实。

### 2.2 Gemini 的修正方向是正确的

Gemini 本轮补充的几个核心点是成立的：

- Mirage 不能按“单 QUIC 服务”思路设计入口防护
- `Gateway` 抗 DDoS 不能只靠单机硬扛，必须依赖 `OS` 编排和 `Client` 自愈
- 需要严格区分：
  - 资源耗尽型攻击
  - 体积型攻击
- 体积型攻击下，“优雅排空”不能作为默认假设

### 2.3 Mirage 的正确目标是系统级生存

Mirage 的真正优势不应体现在“某一台公网节点绝对抗打”，而应体现在：

- `OS` 永不直接暴露
- `Gateway` 可以快速摘除和替换
- `Client` 可以自我寻路和恢复
- 路由、策略、拓扑的恢复链路独立于单一受击节点

---

## 三、当前代码状态判断

结合现有代码，当前 Mirage 已经具备部分抗 DDoS 基础，但距离“系统级闭环”还有明显差距。

### 3.1 当前已经具备的基础

- `Gateway` 已具备 `XDP / TC / Sockmap` 数据面加载能力
- `Gateway` 已具备黑名单管理器，并可将黑名单同步到 `eBPF LPM Trie`
- `Gateway` 的 `TC ingress` 已能查询 `blacklist_lpm` 并静默丢弃
- `OS` 已具备 `Gateway` 注册、下线标记、黑名单下推、拓扑注册等控制面能力
- `OS` 已存在基础 `DDoS` 检测函数和 `Gateway Offline` 管理能力
- `G-Switch` 已具备入口战死、热备池、冷却池等“可耗材化”基础思路

### 3.2 当前不足

- `XDP` 程序目前主要用于流量形态处理，不是专门的入口防火墙
- 黑名单命中目前主要发生在 `TC ingress`，还没有前移到 `XDP`
- 尚未看到完整的多协议入口准入守卫
- 尚未形成统一的“软防 / 硬断”响应状态机
- `OS` 侧 `DetectDDoS` 还存在占位式逻辑，无法作为可靠裁决依据
- 还未看到完整的 `Gateway UnderAttack -> 停止分配 -> Client 学习新拓扑 -> 节点替换` 闭环

---

## 四、Mirage 抗 DDoS 的正确分层

Mirage 的抗 DDoS 架构必须拆成四层，而不能把所有压力都丢给公网节点。

### 4.1 Layer 1：Upstream / 物理层

负责处理：

- 超过单节点物理带宽的体积型攻击
- 上游链路拥塞
- 云厂商或运营商级流量清洗

这一层的结论只有一个：

**超过链路承载上限的攻击，不应寄希望于单节点软件防御。**

### 4.2 Layer 2：Gateway / 入口准入层

负责处理：

- 非法协议画像
- 握手洪泛
- 单位时间新建连接爆发
- 小到中等规模资源耗尽型攻击

目标是把无效流量尽早丢弃在：

- `XDP`
- `TC`
- 无状态准入验证

而不是把压力带进完整协议栈和会话管理器。

### 4.3 Layer 3：OS / 编排裁决层

负责处理：

- 节点是否还应继续接客
- 节点是否应进入 `UnderAttack`
- 节点是否应被移出分配池
- 节点是否已死亡并需要替补

`OS` 的任务不是“替节点抗攻击”，而是“在节点失去战斗力时，快速重组系统”。

### 4.4 Layer 4：Client / 生存恢复层

负责处理：

- 受击 `Gateway` 失联后的被动恢复
- 拓扑学习
- 备用节点发现
- 会话重建与无感回切

如果 `Client` 不具备持续学习与自愈能力，那么任何“可耗材化网关”都只是纸面概念。

---

## 五、必须严格区分的两类攻击

这是本轮整改的核心分界线。

### 5.1 软防场景：资源耗尽型攻击

特征：

- 物理带宽尚未被打满
- 主要压力体现在：
  - 握手洪泛
  - 连接状态耗尽
  - CPU 飙升
  - 内存和连接表压力

这类攻击的正确处理是：

- `Gateway` 继续存活
- `OS` 停止向该节点分配新用户
- 老会话尽量保留
- 本地守卫和准入机制吸收攻击

### 5.2 硬断场景：体积型攻击

特征：

- 上游链路被塞满
- 节点心跳上报能力丧失
- 节点无法可靠向 `OS` 或 `Client` 发出迁移指令

这类攻击的正确处理是：

- 不再假设可以“优雅排空”
- `OS` 应按死亡节点处理
- 启动备用节点
- 客户端通过独立恢复链路重新发现有效入口

## 六、架构整改原则

### 6.1 不做单协议假设

Mirage 不是单 `QUIC` 系统，入口防护不能按“非 QUIC 全丢”实现。

正确做法是：

- 每个入口绑定允许的协议画像
- 每种入口建立最低成本的前置校验
- 不符合入口画像的流量尽早拒绝

### 6.2 不做单 IP 粗暴限流

Mirage 面向的真实用户环境可能包含：

- `CGNAT`
- 企业共享出口
- 移动网络共享出口

因此不能把“源 IP 超阈值”当成唯一准入依据。

### 6.3 尽可能前移到 XDP

能在 `XDP` 做的，不应拖到 `TC`；
能在 `TC` 做的，不应拖到用户态；
能在准入前拒绝的，不应拖到完整握手后。

### 6.4 体积型攻击下优先保全系统，而不是保全节点

对于被打满的 `Gateway`，Mirage 的目标应是：

- 快速识别
- 快速摘除
- 快速替补

而不是强行等待“平滑排空”。

## 七、立刻修

这一层处理收益最大、争议最小、最直接影响生存能力的问题。

### 7.1 Gateway-Immediate-1：新增真正的 `XDP Ingress Guard`

**问题**

当前 `XDP` 程序主要承担流量形态处理，不是专门的入口防护守卫。

**整改动作**

- 新增独立的 `XDP ingress guard`
- 仅做极轻量的入口准入判断
- 第一阶段先覆盖：
  - 黑名单命中
  - 非法 L3/L4 画像
  - 明显异常的握手前特征
  - 超过阈值的入口突发

**验收结果**

- 大量无效入口包在 `XDP` 即被丢弃
- 节点在小到中等规模洪泛下 CPU 和 `sk_buff` 压力明显下降

### 7.2 Gateway-Immediate-2：将 `blacklist_lpm` 前移到 `XDP`

**问题**

当前黑名单命中主要发生在 `TC ingress`，时机偏晚。

**整改动作**

- 在 `XDP guard` 中直接查询 `blacklist_lpm`
- 命中后立即 `XDP_DROP`
- 保留 `TC` 层作为降级和一致性兜底，而不是主命中层

**验收结果**

- 黑名单命中流量不再进入后续协议栈处理
- 数据面黑名单真正成为第一道拒绝门

### 7.3 Gateway-Immediate-3：按入口建立最小准入校验

**问题**

当前入口防护过于依赖后续模块，缺乏分入口的准入语义。

**整改动作**

- 为 `QUIC / WSS / WebRTC / 其他保底入口` 分别定义允许画像
- 入口守卫只做最小成本检查，不做重型解析
- 明显不匹配该入口语义的流量直接拒绝

**验收结果**

- 入口不再是“只要打到端口就能进入”
- 不同协议入口具备独立的最小防护边界

### 7.4 Gateway-Immediate-4：补齐无状态准入能力

**问题**

当前代码中尚未形成明确可验证的握手前无状态准入闭环。

**整改动作**

- 为主入口补齐无状态挑战或地址验证
- 在未完成最小验证前，不分配重型连接状态
- 把握手洪泛的主要压力前移到无状态阶段

**验收结果**

- 伪造源和批量初始握手无法轻易拖垮连接状态
- 握手洪泛更接近被当作“过滤流量”而不是“真实连接请求”

### 7.5 OS-Immediate-1：修正 `DetectDDoS` 的占位逻辑

**问题**

当前 `OS` 侧 DDoS 检测仍有占位式写法，不能作为可靠状态裁决。

**整改动作**

- 接通真实流量、真实连接速率和真实心跳数据
- 区分：
  - `ResourcePressure`
  - `LinkSaturation`
  - `HeartbeatLoss`
- 不再用占位变量做结论

**验收结果**

- `OS` 能基于真实指标判定节点处于：
  - 正常
  - 受压
  - 疑似失联
  - 已死亡

### 7.6 OS-Immediate-2：明确 `UnderAttack` 与 `Dead` 两种节点状态

**问题**

当前控制面虽然有 `ONLINE / OFFLINE / DEGRADED`，但缺少明确的抗 DDoS 语义状态。

**整改动作**

- 新增：
  - `UNDER_ATTACK`
  - `DRAINING`
  - `DEAD`
- `UNDER_ATTACK` 表示还能服务但停止接纳新用户
- `DEAD` 表示不再假设该节点还能完成排空和迁移协作

**验收结果**

- `OS` 能明确区分“节点还活着但受压”和“节点已经在物理意义上不可用”

### 7.7 Client-Immediate-1：把节点失联恢复链路和业务流量解耦

**问题**

如果客户端只能依赖当前受击节点告知迁移目标，那么体积型攻击下恢复链路会一并失效。

**整改动作**

- 明确客户端恢复链路不能依赖单一受击 `Gateway`
- 客户端必须保留独立的备用拓扑获取路径
- 主业务链路断开后，客户端应能主动进入重新发现流程

**验收结果**

- 受击节点即使瞬间失联，客户端仍能启动独立恢复逻辑

## 八、两周内修

这一层重点解决“单点机制已经正确，但还没有编成系统”的问题。

### 8.1 Gateway-TwoWeeks-1：实现多协议感知的入口守卫

**整改动作**

- 将 `XDP guard` 从单纯黑名单过滤升级为多协议感知入口
- 保持极轻量，不做深层业务逻辑
- 为不同入口维护独立的准入模板和指标

**验收结果**

- Mirage 的入口守卫不再带有“单协议服务”假设

### 8.2 Gateway-TwoWeeks-2：把粗粒度 IP 限流升级为多维准入控制

**整改动作**

- 限流依据不再只绑定物理源 IP
- 加入：
  - 单位时间新建请求量
  - 有效验证通过率
  - 会话令牌有效性
  - 入口画像匹配率
- 对异常行为做多维度熔断

**验收结果**

- 减少 `CGNAT`、企业出口等场景下的误伤
- 准入控制更接近“行为治理”，而不是“IP 连坐”

### 8.3 OS-TwoWeeks-1：建立 `软防` 响应链路

**整改动作**

- `Gateway` 本地识别资源耗尽压力
- 向 `OS` 上报进入 `UNDER_ATTACK`
- `OS` 停止将该节点分配给新用户
- 老会话继续保留
- 攻击结束后节点自动退出 `UNDER_ATTACK`

**验收结果**

- Mirage 对资源耗尽型攻击具备“边服务边防守”的能力

### 8.4 OS-TwoWeeks-2：建立 `硬断` 响应链路

**整改动作**

- `OS` 基于心跳超时、链路饱和、观测数据综合判定节点 `DEAD`
- 自动将节点从可分配池移除
- 自动标记为不可用拓扑
- 触发备用节点接替流程

**验收结果**

- 体积型攻击下不再等待节点“自己恢复”
- 控制面可以直接按死亡节点处理

### 8.5 Client-TwoWeeks-1：补齐运行时拓扑学习

**整改动作**

- 客户端不再只依赖启动时 bootstrap 池
- 必须持续学习最新 `Gateway` 池和优先级
- 失联后能快速从新的有效节点集中选择替补

**验收结果**

- `Gateway` 替换能力在客户端侧真正可用

### 8.6 Client-TwoWeeks-2：建立失联后的原子恢复状态机

**整改动作**

- 区分：
  - 主链路抖动
  - 节点受压
  - 节点死亡
- 明确恢复顺序：
  - 短时重试
  - 拓扑刷新
  - 备用节点竞速连接
  - 会话恢复

**验收结果**

- 客户端遇到公网节点硬断时，不会停在无意义的长时间重试中

---

## 九、版本级改造

这一层是 Mirage 真正进入 V2 编排能力后的系统升级。

### 9.1 V2-Architecture-1：把抗 DDoS 纳入 `Survival Orchestrator`

**目标**

让抗 DDoS 不再是外围脚本或孤立规则，而成为 `Survival Mode` 的一部分。

**整改动作**

- 将 `UnderAttack / Dead / Recovering` 纳入统一状态机
- 与：
  - `Link State`
  - `Session State`
  - `Control State`
 统一编排

**验收结果**

- 抗 DDoS 从“规则拼装”升级为“编排内核的一部分”

### 9.2 V2-Architecture-2：将节点替补纳入 `State Commit Engine`

**目标**

节点替换不能只是“发现挂了就换”，而应进入统一提交流程。

**整改动作**

- 对以下动作引入事务化控制：
  - 节点摘除
  - 热备接替
  - 拓扑发布
  - 客户端可见路由更新
- 确保替换失败时可回滚，可重试，可审计

**验收结果**

- `Gateway` 被打死后的系统重组过程可解释、可追踪、可恢复

### 9.3 V2-Architecture-3：将抗 DDoS 纳入预算模型

**目标**

抗 DDoS 动作本身也要受预算约束。

**整改动作**

- 为以下行为定义成本：
  - 高强度入口守卫
  - 高频切换
  - 热备节点常驻
  - 上游清洗资源使用
  - 高等级恢复路径
- 让不同服务等级绑定不同生存预算

**验收结果**

- Mirage 的抗 DDoS 能力与服务等级、资源成本和运营策略统一起来

### 9.4 V2-Architecture-4：建立独立的恢复与替补发布平面

**目标**

受击节点即使完全假死，客户端仍能获得新的有效入口。

**整改动作**

- 建立独立于单 `Gateway` 业务面的恢复发布平面
- 让 `OS` 可以在节点死亡后仍然发布新的替补拓扑
- 客户端将该平面作为最后兜底恢复渠道

**验收结果**

- “Gateway 可耗材化”不再是概念，而是拥有真正独立的恢复链路

### 9.5 V2-Architecture-5：引入 `Gateway` 预热池和容量分层

**目标**

节点被打死后不依赖临时起机器。

**整改动作**

- 建立：
  - 活跃节点池
  - 温备用池
  - 冷备用池
- 不同层级具备明确的拉起、接替和冷却策略

**验收结果**

- 攻击发生后可以切到预热节点，而不是等待云资源慢启动

## 十、最终落地口径

Mirage 抗 DDoS 的最终设计不应表述为：

- 每个节点都能硬扛所有攻击
- 暴露真实 IP 后仍可无限对抗
- 所有场景都能优雅迁移

Mirage 更准确、更成熟的系统口径应是：

1. 对资源耗尽型攻击，`Gateway` 通过前置准入与数据面过滤尽量存活  
2. 对体积型攻击，`OS` 不迷信单节点排空，而是快速裁决、快速摘除、快速替补  
3. `Client` 必须具备独立恢复能力，不能依赖受击节点告知新路由  
4. `OS / Gateway / Client` 三层协同，才是 Mirage 真正的抗 DDoS 能力  

---

## 十一、总结

本轮整改的核心结论可以压缩成一句话：

**Mirage 的抗 DDoS 目标不是让每个公网节点成为永远打不死的堡垒，而是让受击节点可以被快速识别、快速放弃、快速替换，并让整套系统在 `OS` 编排与 `Client` 自愈能力支撑下持续存活。**
