# Mirage Client 鲁棒性整改清单

## 一、文档目标

本文档用于承接本次针对 `phantom-client` 的审计结论，并把 Client 端需要补齐的能力整理成一份可执行的整改清单。

这份清单只围绕一个目标展开：

**把 Client 从“可运行的连接工具”升级为“用户几乎无感、可长期常驻、可自愈、可托管”的正式产品形态。**

结合当前项目定位，理想中的 Client 应该具备以下产品特征：

- 用户完成一次开通或兑换后，后续尽量不需要人工干预
- 日常以后台常驻方式运行，而不是依赖用户长期盯着终端
- 网络波动、Gateway 切换、节点失联、短时控制面异常时，Client 能优先自愈
- 配额、有效期、服务等级等业务能力由系统托管，而不是让用户自己猜测当前状态
- 路由切换必须稳定、原子、安全，不能出现“逻辑上切换了，系统路由还停在旧节点”的情况

---

## 二、当前结论

当前 Client 已经具备一套基础的启动、建链、转发、Kill Switch 与内存安全能力，说明核心方向是通的。

但从“正式产品”角度看，现阶段更接近：

- 一个可用的前台 CLI 原型
- 一个具备部分自恢复能力的网络工具
- 一个对多 Gateway、长期托管、后台运行支持还不完整的客户端骨架

当前最突出的问题不在“有没有功能”，而在“这些功能是否已经形成闭环”。

具体来说，Client 端现在主要缺 5 条闭环能力：

1. **连接状态机没有收拢成单一真相源**
2. **拓扑学习没有真正接通**
3. **订阅、配额、服务等级没有形成运行时托管**
4. **后台服务化尚未完成，仍偏工具型使用方式**
5. **路由切换与连接切换还没有做到真正原子一致**

因此，本轮整改不是“多加几个参数”，而是把 Client 端做成一套真正能让用户放心放在后台长期运行的系统组件。

---

## 三、整改原则

### 3.1 无感优先

能自动恢复的，不要让用户介入；能自动续拉状态的，不要让用户手动刷新；能后台完成的，不要要求用户盯着终端。

### 3.2 单一真相源优先

同一时刻只能有一个真实连接状态、一个真实当前 Gateway、一个真实路由状态。不能允许“显示连的是 A，底层实际已经切到 B”。

### 3.3 先保证正确，再追求更复杂的调度

在连接、路由、订阅没有形成稳定闭环前，不应继续堆叠更复杂的多路径或策略调度逻辑。

### 3.4 运行时托管优先于启动时校验

正式产品不能只在启动时检查一次 token 或有效期。只要 Client 是常驻产品，就必须具备运行时刷新、续拉、降级与恢复能力。

### 3.5 所有切换动作都必须可验收

每个关键动作都应该可以验证，例如：

- Gateway 切换后，连接对象、状态展示、Kill Switch 路由是否一致
- 控制面短时不可达时，Client 是否还能维持保底连接
- 订阅到期、配额用尽、服务等级变更时，Client 是否进入正确策略

## 四、连接状态机整改项

这一层解决的是 Client 最核心的问题：**谁负责连接、谁负责切换、谁说了算。**

### FSM-1：建立统一的连接状态机

**优先级：P0**

**问题**

当前连接建立、探测成功、接收异常、发送异常、重连、切网关这些动作分散在多个路径里，缺少统一状态机收口。

**实施项**

- 定义正式的连接状态，例如：`Init / Bootstrapping / Connected / Degraded / Reconnecting / Exhausted / Stopped`
- 所有状态切换都经过同一个状态机入口
- 不再允许多个 goroutine 各自直接推动重连或覆盖底层连接对象
- 为每次状态切换生成统一事件，供日志、状态展示和上层服务消费

**验收标准**

- 任意时刻都能明确知道 Client 当前处于什么状态
- 不再出现多个协程同时触发重连、同时修改连接对象的情况

### FSM-2：把重连收敛成单飞执行

**优先级：P0**

**问题**

当前收发两条主循环都可能在错误时直接调用 `Reconnect()`，容易出现并发重连和状态覆盖。

**实施项**

- 引入单飞重连控制器，同一时刻只允许一个重连流程执行
- 其他异常只负责上报“需要重连”，不直接操作连接
- 为重连过程增加幂等保护，避免重复清理、重复建链、重复切路由

**验收标准**

- 网络抖动时不会触发多条并发重连链路
- 重连失败不会把内部状态打乱

### FSM-3：把“探测成功”与“正式接管连接”拆开

**优先级：P0**

**问题**

当前并发探测 bootstrap 节点时，后返回成功的探测仍可能覆盖 `c.quic`，导致 `currentGW` 与真实连接错位。

**实施项**

- 探测阶段只返回候选结果，不直接覆盖正式连接
- 由状态机选出唯一胜者后，再执行正式接管
- 接管动作必须原子更新：`quic engine + current gateway + connected state`

**验收标准**

- `currentGW` 与实际 QUIC 连接对象始终一致
- 状态展示、路由切换、故障恢复都基于同一个真实当前 Gateway

### FSM-4：补齐分级退化策略的显式行为

**优先级：P1**

**问题**

当前虽然有 Route Table、Bootstrap Pool、Resonance 三层回退思路，但没有形成明确、可观测的退化状态。

**实施项**

- 把一级、二级、三级回退定义成正式状态或正式阶段
- 为每级退化增加进入条件、退出条件、超时与回退原因
- 上报每次退化发生的原因、尝试次数、恢复耗时

**验收标准**

- 可以准确知道 Client 当前是在正常连接、轻度退化还是绝境恢复
- 故障复盘时能还原完整重连路径

### FSM-5：建立连接健康评分与主动切换门槛

**优先级：P2**

**问题**

当前 Client 更像“断了再重连”，缺少对弱劣化链路的提前识别。

**实施项**

- 增加基础健康指标，例如握手时延、Datagram 丢失率、连续发送失败、连续接收超时
- 当健康度持续低于门槛时，主动进入预切换或预热探测
- 让切换从“被动断线恢复”升级为“有限度主动优化”

**验收标准**

- Client 不必等到完全断链才开始动作
- 弱连接场景下的恢复时间明显缩短

## 五、拓扑学习整改项

这一层解决的是：**Client 不能只靠启动时拿到的几个地址活着。**

### TOPO-1：实现真正可用的 Route Table 拉取能力

**优先级：P0**

**问题**

当前 `PullRouteTable()` 仍是空实现，启动流程虽然调用了它，但实际上没有学到任何运行时拓扑。

**实施项**

- 把 Route Table 拉取做成正式协议能力
- 至少支持返回当前可用 Gateway 列表、优先级、区域/Cell 信息、版本号
- 返回结果写入内存路由表，并记录最近更新时间

**验收标准**

- Client 启动后能真正拿到运行时维护的 Gateway 列表
- 路由表不再只是空壳结构

### TOPO-2：把拓扑学习改成持续刷新，而不是只拉一次

**优先级：P0**

**问题**

即使补齐一次性拉取，如果后续没有周期刷新，Client 依然无法适应节点上下线、调度变化和控制面重分配。

**实施项**

- 增加周期性拉取机制
- 对刷新失败做指数退避，但不能因为一次失败就清空现有拓扑
- 为路由表增加版本比较，避免旧数据覆盖新数据

**验收标准**

- Gateway 动态变化后，Client 能在运行中学习到新的节点池
- 控制面短时波动不会导致本地拓扑失真

### TOPO-3：让绝境发现链路真正接入主流程

**优先级：P0**

**问题**

当前 `SetResonanceResolver()` 存在，但主启动链路没有完整接通，意味着最末级自愈能力还停留在预留能力阶段。

**实施项**

- 在正式启动流程中注入 Resonance Resolver
- 明确其启用条件、超时策略与结果回写逻辑
- 发现到新节点后，同时更新内存路由表，而不是只做一次临时连接

**验收标准**

- 所有已知节点不可用时，Client 会自动进入绝境发现流程
- 发现成功后，新节点会进入后续可复用的拓扑池

### TOPO-4：区分“启动种子池”和“运行时拓扑池”

**优先级：P1**

**问题**

当前 Bootstrap Pool 既承担首次建链，又承担长期兜底，职责混在一起。

**实施项**

- 明确 `Bootstrap Pool` 只作为启动种子和保底兜底
- `Route Table` 作为运行时主拓扑来源
- 对两类地址分别维护优先级、寿命和使用场景

**验收标准**

- Client 的主切换路径优先依赖运行时拓扑
- Bootstrap 不再承担过多常态职责

### TOPO-5：为拓扑增加最小可信约束

**优先级：P1**

**问题**

如果未来拓扑学习能力补齐，但缺少签名、版本或来源校验，Client 仍可能学到错误拓扑。

**实施项**

- 为路由表结果增加签名或可信来源校验
- 至少校验版本、发布时间、签发来源
- 对异常拓扑更新执行保守拒绝策略

**验收标准**

- 非可信或过期拓扑不能覆盖现有有效拓扑
- Client 在拓扑异常时优先保守运行，而不是盲切错误节点

## 六、订阅托管整改项

这一层解决的是：**用户付费之后，不应该自己操心剩余配额、到期时间和服务等级切换。**

### SUB-1：修复 URI 兑换链路对关键配置的丢失

**优先级：P0**

**问题**

当前 `redeemFromURI()` 重新组装 `BootstrapConfig` 时，只保留了 `BootstrapPool` 和一个伪造的 `ExpiresAt`，会丢失 `PreSharedKey`、`CertFingerprint`、`UserID` 等关键运行时信息。

**实施项**

- 让 URI 兑换结果完整映射到正式配置结构
- 不能再伪造到期时间
- 保留用户标识、认证材料、证书校验材料、真实配额与真实有效期

**验收标准**

- “一次兑换即可使用”的链路能生成完整、可长期托管的运行配置
- URI 模式与直接 token 模式在核心能力上保持一致

### SUB-2：把订阅校验从“启动时一次性检查”升级为“运行时托管”

**优先级：P0**

**问题**

当前有效期只在启动时检查一次，Client 常驻后无法感知订阅续费、降级、到期或封禁。

**实施项**

- 增加周期性 entitlement 拉取
- 同步订阅有效期、剩余配额、服务等级、封禁状态
- 对状态变化执行正式策略，而不是只记录日志

**验收标准**

- 用户续费后，Client 能自动续得新状态
- 用户到期或被停用后，Client 会执行正确的受控降级或断开策略

### SUB-3：把服务等级策略做成 Client 运行时能力

**优先级：P0**

**问题**

如果未来 OS 侧提供不同付费等级，但 Client 端没有运行时感知与执行能力，服务分层就落不到产品行为上。

**实施项**

- 在 Client 本地增加 `service class` 概念
- 允许 OS 下发不同等级对应的连接策略，例如：
  - 可用 Gateway 池范围
  - 切换积极度
  - 保底发现能力是否开启
  - 后台自愈频率与心跳频率
- 让 Client 把服务等级变化即时作用到行为上

**验收标准**

- 不同服务等级的用户，Client 行为存在稳定、可验证的差异
- 服务等级切换不需要用户手工重装或重新配置

### SUB-4：增加离线宽限与受控退化策略

**优先级：P1**

**问题**

正式产品不应该因为控制面短时不可达，就立即把已在线用户全部打成不可用。

**实施项**

- 增加最小离线宽限窗口
- 区分“控制面失联”“配额耗尽”“订阅到期”“账号停用”四类场景
- 对不同场景执行不同的受控行为，例如保留当前连接、禁止新建连接、降低服务等级、限制切换能力

**验收标准**

- 控制面短时故障不会把正常在线用户瞬间踢下线
- 真正到期或停用时，又能进入明确受控状态

### SUB-5：补齐本地状态缓存与恢复逻辑

**优先级：P2**

**问题**

如果 Client 完全依赖每次冷启动重新兑换或重新拉取状态，长期使用体验会很脆弱。

**实施项**

- 为非敏感托管状态建立最小本地缓存
- 对敏感材料坚持内存优先与最小落盘原则
- 让 Client 重启后能快速恢复最近一次已知的订阅与拓扑状态

**验收标准**

- 短时间重启不会把用户重新带回“手工补配置”的体验
- 本地恢复与安全边界之间保持平衡

## 七、后台服务化整改项

这一层解决的是：**Client 要从终端工具变成长期常驻产品。**

### SVC-1：把前台 CLI 启动形态改成正式后台服务

**优先级：P0**

**问题**

当前 Client 仍然依赖命令行参数、stdin、前台日志输出和信号退出，更像开发工具而不是终端产品。

**实施项**

- 将 Client 主进程改造成正式 daemon/service
- Windows 与 Linux 分别接入系统服务管理
- 标准运行方式改为后台常驻，CLI 仅保留为调试入口

**验收标准**

- 用户安装并开通后，Client 默认以后台服务运行
- 终端关闭不影响服务持续工作

### SVC-2：把“首次开通”与“长期运行”拆成两个清晰阶段

**优先级：P0**

**问题**

当前一次启动流程同时承担兑换配置、创建 TUN、建链、显示状态、等待退出，生命周期过于混杂。

**实施项**

- 定义“首次开通/绑定”流程
- 定义“后台运行”流程
- 定义“用户查看状态/手动诊断”流程

**验收标准**

- 用户完成首次开通后，后续使用不再依赖重复输入 token 或 URI
- 运行态与配置态职责清晰分离

### SVC-3：补齐后台健康守护与自恢复能力

**优先级：P1**

**问题**

正式后台产品需要具备进程级与功能级的持续健康维护，而不仅是“连接断了就重连”。

**实施项**

- 增加服务级健康检查
- 增加异常退出后的自动拉起机制
- 为关键组件建立启动顺序与失败回滚逻辑，例如 TUN、路由、连接、状态同步

**验收标准**

- 进程崩溃、网络瞬断、短时依赖异常后，服务能自动回到稳定态
- 异常恢复过程中不会遗留半残路由或僵尸状态

### SVC-4：把日志、状态与用户提示分层

**优先级：P1**

**问题**

当前很多状态通过 stderr 或终端文本输出，既不利于后台运行，也不利于用户无感。

**实施项**

- 将日志分为运维日志、诊断日志、用户可见状态
- 普通用户只接收必要提示，例如需要重新登录、服务被停用、系统权限缺失
- 大多数恢复过程应静默完成

**验收标准**

- 正常运行时用户不需要盯日志
- 真正需要人工介入时，提示信息清晰且准确

### SVC-5：为升级、配置轮换、证书更新预留服务化能力

**优先级：P2**

**问题**

如果后续版本需要热更新配置、轮换证书、调整服务等级，没有后台服务框架会很难平滑交付。

**实施项**

- 预留在线配置刷新入口
- 预留平滑重载而非强制退出重启的能力
- 让证书、拓扑、服务等级刷新可以在后台完成

**验收标准**

- 常见配置变更不需要用户重新执行完整初始化流程
- 后续产品化迭代成本可控

## 八、路由安全整改项

这一层解决的是：**只要 Client 在运行，系统路由就必须始终与真实连接状态一致。**

### ROUTE-1：让所有切换路径都触发 Kill Switch 路由更新

**优先级：P0**

**问题**

当前 `Reconnect()` 只有 Route Table 一级切换显式调用了 `switchFn`，二级 Bootstrap 回退和三级 Resonance 恢复成功后未必同步更新路由。

**实施项**

- 把路由更新从某个分支中的附带动作，提升为正式切换步骤
- 只要当前 Gateway 发生变化，就必须统一触发路由更新
- 路由更新失败时，连接接管不能静默成功

**验收标准**

- 不论通过哪一级恢复成功，路由都会跟随切到当前 Gateway
- 不再出现“逻辑已切换、路由仍停在旧网关”的错位

### ROUTE-2：把“连接切换”和“路由切换”做成原子事务

**优先级：P0**

**问题**

当前连接对象切换与路由变更分散执行，中间如果失败，可能留下半切换状态。

**实施项**

- 定义正式切换顺序，例如：候选探测成功 -> 预加新路由 -> 接管新连接 -> 删除旧路由 -> 提交状态
- 为失败场景建立回滚逻辑
- 把切换结果统一写入状态机

**验收标准**

- 切换过程中即使失败，也不会把用户设备留在错误路由状态
- 连接与系统路由始终同向变化

### ROUTE-3：去掉对 `8.8.8.8:53` 的硬编码物理网卡探测依赖

**优先级：P0**

**问题**

当前 QUIC 建链通过 `net.Dial("udp4", "8.8.8.8:53")` 识别物理出口，这在受限网络、高对抗网络或定制 DNS 环境中都不可靠。

**实施项**

- 改成基于系统路由表、本地默认网关或显式出口接口选择
- 避免把公共 DNS 可达性当作物理出口探测前提
- 为受限网络准备保守回退策略

**验收标准**

- 在禁止访问 `8.8.8.8` 的环境中，Client 仍可完成建链
- 出口接口选择不依赖外部公共地址

### ROUTE-4：补齐启动、异常退出、切换失败时的路由回滚保护

**优先级：P1**

**问题**

当前 Kill Switch 已具备基础激活和恢复逻辑，但正式产品还需要覆盖更多半失败场景。

**实施项**

- 对激活中途失败、切换中途失败、服务异常退出分别增加回滚保护
- 在后台服务恢复时检查并修复残留路由
- 把“人工重启电脑恢复网络”从常规路径降到最后兜底

**验收标准**

- 大多数异常退出场景都能自动恢复路由
- 用户设备不会因为 Client 异常而长期停留在错误路由状态

### ROUTE-5：增加路由与连接一致性自检

**优先级：P2**

**问题**

仅靠切换时触发一次更新还不够，正式产品需要周期性确认“当前路由是不是仍然指向当前网关”。

**实施项**

- 定期自检当前 Gateway、Host Route、默认路由、TUN 状态是否一致
- 发现不一致时优先自动修复
- 将严重不一致上报为需要人工诊断的故障

**验收标准**

- 路由漂移、系统外部改路由等情况能被识别
- Client 能持续维持 fail-closed 与当前连接的一致性

## 九、建议实施顺序

为了尽可能小改动获得最大产品收益，建议按下面顺序推进：

### 第一阶段：先把“会不会乱”修掉

- `FSM-1`
- `FSM-2`
- `FSM-3`
- `ROUTE-1`
- `ROUTE-2`
- `SUB-1`

这一阶段的目标是先消灭连接错位、路由错位、配置丢失这类高风险问题。

### 第二阶段：把“能不能自己活下去”补齐

- `TOPO-1`
- `TOPO-2`
- `TOPO-3`
- `SUB-2`
- `SUB-3`
- `SVC-1`
- `SVC-2`

这一阶段完成后，Client 才真正具备“长期常驻、持续学习拓扑、运行时接受托管”的基础能力。

### 第三阶段：把“体验像产品”做完整

- `FSM-4`
- `FSM-5`
- `TOPO-4`
- `TOPO-5`
- `SUB-4`
- `SUB-5`
- `SVC-3`
- `SVC-4`
- `SVC-5`
- `ROUTE-3`
- `ROUTE-4`
- `ROUTE-5`

这一阶段的目标是把 Client 从“能自愈”进一步升级为“稳定、安静、可长期托管的正式产品”。

---

## 十、最终目标

如果这份整改清单落完，Mirage Client 的产品形态应当升级为：

- 用户完成开通后即可长期后台运行
- 多 Gateway 环境下可以持续学习、持续切换、持续自愈
- 服务等级、配额、有效期由系统持续托管
- 路由与连接始终保持一致，不因切换留下隐患
- 用户只在极少数真正需要人工介入的情况下才会被打扰

这才符合 Mirage 对 Client 的最终要求：

**不是一个“需要用户操作的代理程序”，而是一个“用户几乎感觉不到、但在关键时刻永远在线的高鲁棒性连接组件”。**
