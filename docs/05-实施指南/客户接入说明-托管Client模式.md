# 客户接入说明：托管 Client 模式

## 一、文档目标

本文档用于定义 Mirage 在“平台全托管、客户只使用 Client”的交付模式下，面向终端客户的标准接入流程。

本文档只描述客户可见的接入体验，不暴露平台内部架构，不要求客户理解后端运行细节。

适用范围：

- 新客户首次开通
- 存量客户换设备重装
- 客服或运营向客户发送接入说明
- 付费后自动发放激活凭证的接入流程

---

## 二、产品边界

在托管 Client 模式下，客户只接触以下三类对象：

1. Mirage Client
2. 一次性激活凭证
3. 续费入口

客户不需要感知，也不需要管理以下内容：

- 后端接入资源
- 节点信息
- 调度逻辑
- 状态维护
- 平台内部控制系统

Mirage 采用全托管模式提供服务，后端资源和运行维护由平台统一负责。

---

## 三、客户标准接入路径

### 3.1 首次开通

首次开通时，客户应获得以下材料：

1. Client 下载地址
2. 一次性激活凭证
3. 简版安装说明
4. 续费入口或续费方式说明

平台交付给客户的，不应是节点地址或复杂配置，而应是“安装 + 激活”这一条极简路径。

### 3.2 激活完成后的状态

客户完成激活后，Client 应进入标准可用状态。

对客户而言，这意味着：

- 无需再手动添加配置
- 无需理解后端接入结构
- 无需维护节点清单
- 只需正常使用服务

### 3.3 后续使用

后续使用中，客户的高频动作应被限制在以下范围：

- 打开或关闭 Client
- 查看账号状态
- 查看到期时间
- 完成续费
- 联系支持

---

## 四、客户接入流程

### 步骤 1：下载安装 Client

客户下载安装适配自己系统版本的 Mirage Client。

建议支持的平台包括：

- Windows
- macOS
- Linux

平台对外应始终只提供一个明确的下载入口，不要求客户自行判断后端资源类型。

### 步骤 2：获取一次性激活凭证

客户购买或开通服务后，平台向客户发放一次性激活凭证。

激活凭证可以是以下两种形式之一：

1. 一次性激活码
2. 一次性激活链接

推荐优先使用“一次性激活链接”，因为它更适合降低客户操作成本。

### 步骤 3：在 Client 中完成激活

客户在 Client 中输入激活码，或直接打开一次性激活链接。

Client 应自动完成以下动作：

- 拉取一次性交付材料
- 完成首次初始化
- 保存运行所需配置
- 进入可用状态

整个过程不应要求客户手动填写节点地址、证书参数或高级网络选项。

### 步骤 4：开始使用服务

激活完成后，客户即可开始使用服务。

对客户来说，产品体验应尽量接近“安装完成即可使用”，而不是“安装后还需要二次配置”。

### 步骤 5：到期续费

客户到期前，平台应向客户提供明确的续费入口。

续费完成后，平台应自动延长客户服务状态，Client 继续正常运行，不要求客户重新进行复杂初始化。

---

## 五、客户可见信息范围

在托管 Client 模式下，建议只对客户显示以下信息：

- 当前服务状态
- 当前账号状态
- 套餐等级
- 到期时间
- 续费入口
- Client 版本
- 基础诊断状态

不建议对客户显示以下内容：

- 后端节点列表
- 节点编号
- 节点地域池细节
- 平台内部组件名称
- 手工切换逻辑
- 内部控制路径

产品设计的重点不是“给客户更多可玩性”，而是“让客户尽量无感使用”。

## 七、平台侧配套要求

要实现真正的“客户只装一个 Client”，平台侧必须具备以下配套能力：

### 7.1 激活凭证发放能力

平台必须可以稳定发放一次性激活凭证，并确保：

- 凭证可单次使用
- 凭证有明确有效期
- 凭证与客户账号绑定
- 凭证使用后自动失效

### 7.2 首次初始化能力

平台必须保证 Client 在激活时能够顺利完成首次初始化，包括：

- 接收交付材料
- 写入本地运行配置
- 进入可用状态

### 7.3 服务状态同步能力

平台必须能够向 Client 传递最小必要的状态信息，例如：

- 服务是否有效
- 套餐等级
- 到期时间
- 是否需要续费

### 7.4 续费后的状态延续能力

客户续费后，平台应尽量做到：

- 无需客户重装
- 无需客户重新输入复杂配置
- 无需客户重新理解接入方式

平台应尽量把“续费”设计成服务状态延续，而不是一次新的技术部署。

## 八、对客户的承诺边界

Mirage 在托管 Client 模式下，对客户最适合承诺的是：

1. 接入方式简单
2. 后端资源由平台托管
3. 系统会尽量自动处理状态变化和恢复
4. 客户只需要维护自己的服务状态

不适合直接承诺的是：

1. 绝对零中断
2. 客户永远无需任何支持
3. 客户可以控制复杂后端结构
4. 客户可以完全绕过平台运营逻辑

产品应该强调“托管式使用体验”，而不是“把复杂后端暴露给客户”。

## 九、标准 FAQ

### Q1：我需要自己维护节点吗？

不需要。  
Mirage 采用全托管模式提供服务，后端资源由平台统一维护。

### Q2：我需要自己填写复杂配置吗？

不需要。  
标准流程下，客户只需要安装 Client 并完成激活。

### Q3：我会看到后端接入资源吗？

通常不会。  
Mirage 的设计目标是让客户尽量只看到一个可用的 Client，而不是复杂后端结构。

### Q4：如果环境变化了怎么办？

系统会尽量自动处理恢复和状态调整。  
客户通常不需要自行处理后端切换细节。

### Q5：到期后怎么办？

到期前完成续费即可。  
平台会尽量将续费后的服务状态平滑延续到当前 Client。

## 十、推荐的客户接入话术

### 正式版

Mirage 提供的是全托管连接服务。  
您只需安装 Client，并使用一次性激活凭证完成初始化；后端接入、状态维护和服务运营均由平台统一处理。

激活完成后即可直接使用，后续只需关注服务状态并在到期前完成续费即可。

### 极简版

安装 Client，输入一次激活凭证，即可开始使用。  
后端由平台托管，您不需要管理节点或复杂配置。

### 偏服务版

Mirage 不是需要您自己维护后端资源的工具，而是一项托管服务。  
您只需要一个 Client 和一个一次性激活凭证，后续服务状态和运行维护均由平台负责。

---

## 十一、产品实现要求

如果 Mirage 要真正以托管 Client 模式交付，那么产品实现必须持续朝以下方向收敛：

1. 客户只需安装一个 Client
2. 激活必须比手动配置更容易
3. 后端资源必须对客户透明
4. 续费必须比重新部署更轻
5. 客户界面只保留少量必要信息

如果某项设计会迫使客户理解内部后端结构，那么它就不符合托管 Client 模式的目标。
