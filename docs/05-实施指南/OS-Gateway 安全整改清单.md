# Mirage OS / Gateway 安全整改清单

## 一、文档目标

本文档用于承接本次针对 Mirage `OS` 与 `Gateway` 的安全审计结论，并将问题整理为可执行的整改清单。

这份清单重点解决两类核心问题：

1. **如何尽早拒绝非正常用户与异常流量**
2. **如何把“高对抗”能力从概念能力升级为真正可落地、可验证的系统能力**

本文不追求一次性覆盖所有长期演进方向，而是按三个层级推进：

- **立刻修**：高风险、可被直接利用、应该优先封堵的问题
- **两周内修**：需要少量联调或补链路，但应尽快收口的问题
- **版本级改造**：需要跨模块、跨协议或跨数据面的系统升级

---

## 二、先说结论

当前 Mirage 已经具备不少安全与对抗能力的基础模块，例如：

- Gateway 侧的 eBPF 数据面
- 威胁感知与黑名单管理
- 蜜罐与影子欺骗能力
- RAM Shield、反调试、看门狗、焦土协议
- OS 侧的 JWT、挑战认证、mTLS 控制台

但从安全闭环的角度看，当前系统仍有一个明显问题：

**“能感知、能记录、能做部分反制”，但还没有形成“默认拒绝、不可绕过、可审计、可验证”的完整防护链。**

更直白地说：

- 有些入口已经上锁，但旁路入口还没上锁
- 有些威胁已经能识别，但识别后没有第一时间阻断
- 有些高对抗能力已经存在，但更多体现在组件层，而不是体现在系统整体行为上

因此，本轮整改的目标不是“再加几个安全模块”，而是把现有能力真正收敛成一套统一的安全运行模型。

---

## 三、整改原则

### 3.1 控制面必须默认拒绝

所有能改变系统状态、影响 Gateway 行为、触发战术切换、执行 kill / 自毁 / 下发配置的入口，必须默认拒绝，不能依赖“知道端口的人应该都是自己人”。

### 3.2 入口越早拒绝，成本越低

能在入口层拒绝的，不要放到业务层；能在数据面拒绝的，不要拖到控制面；能在握手前后立刻淘汰的，不要等到异常扩散后再处理。

### 3.3 高对抗不是“多几个模块”，而是“不可预测、不可枚举、不可轻易绕过”

Mirage 的高对抗能力应该体现在：

- 控制面难以伪造
- 非授权流量难以进入
- 异常行为难以持续试探
- 被发现异常后系统能自动进入更保守、更隐匿、更低暴露面的状态

### 3.4 先封高危旁路，再做高级能力

如果控制面本身可被未授权调用，那么更复杂的高对抗功能价值会被严重稀释。因此本轮必须先把旁路入口、明文入口、越权入口收口。

### 3.5 所有整改都要可验收

每个改造项都应该有清晰结果，例如：

- 未授权访问是否直接被拒绝
- 非法 Gateway 是否无法接入
- 异常 IP 是否能在入口被淘汰
- 战术指令是否只能由授权控制面下发

---

## 四、立刻修

这一层只处理高风险、低争议、收益最大的项目。目标是先把最容易被直接利用的洞补上。

---

## 五、OS 立刻修

### OS-Immediate-1：关闭普通用户到管理面的权限升级路径

**问题**

当前普通用户登录后拿到的 token，与管理面 token 的边界不够清晰，存在将普通 token 续签为管理 token 的风险。

**整改动作**

- 立即取消普通 token 到管理员身份的续签逻辑
- 普通登录只发普通用户 token
- 管理员 token 只能来自独立的管理员认证链路
- 所有续签操作必须保持原角色，不允许自动提权

**验收结果**

- 普通用户 token 无法访问管理接口
- 普通用户 token 无法通过续签变成管理员 token

### OS-Immediate-2：为所有管理接口补上角色校验

**问题**

当前很多 API 虽然挂了 JWT，但仍然缺少真正的角色边界与资源归属边界。

**整改动作**

- 为 `users / gateways / cells / threats / internal ops` 统一补上角色校验
- 管理类接口只允许 `admin` 或 `operator`
- 普通用户接口只允许访问自己的资源
- 用户对其他用户的 `bindPubkey / deactivate / query` 等操作一律拒绝

**验收结果**

- 普通用户无法查看其他用户信息
- 普通用户无法修改其他用户状态
- 普通用户无法访问蜂窝、网关、威胁等管理接口

### OS-Immediate-3：立即把 WebSocket 指挥面改成“只读”

**问题**

当前 WebSocket 面板不仅能看，还能写，并且缺少明确的身份校验与来源约束。

**整改动作**

- 立即禁止前端通过 WebSocket 发战术类写操作
- WebSocket 第一阶段只保留事件订阅与状态展示
- 所有写操作先回收至受保护的 REST / Console / 内部控制面

**验收结果**

- 任意 WebSocket 客户端无法下发战术切换、配置变更、自毁指令
- WebSocket 只承担观测能力，不再承担控制能力

### OS-Immediate-4：收口所有“内部接口”

**问题**

当前 `Provisioner HTTP`、`gateway-bridge REST`、部分 gRPC 入口默认暴露面过大，且存在未鉴权风险。

**整改动作**

- 所有内部 HTTP 服务先绑定到 `127.0.0.1` 或专用内网地址
- 关闭不必要的公网监听
- 清理调试反射能力，非必要不启用 reflection
- 给内部 REST 立即加一层最小可用的鉴权，例如共享密钥头或 mTLS

**验收结果**

- 这些入口无法被外部未授权直接调用
- 内部接口误暴露时也会被鉴权拦住

### OS-Immediate-5：给登录、挑战、认证接口补速率限制

**问题**

当前登录、挑战、验证接口是安全入口，但没有看到足够明确的频控措施。

**整改动作**

- 对 `register / login / challenge / breach / validate` 增加 IP 级与账号级限速
- 对连续失败的来源添加短时冻结
- 记录高频尝试来源到安全审计日志

**验收结果**

- 爆破与撞库无法高频连续尝试
- 异常登录行为可被快速识别与追踪

---

## 六、Gateway 立刻修

### Gateway-Immediate-1：生产模式强制 mTLS，不允许明文回退

**问题**

当前 Gateway 在拿不到 TLS 配置时仍可能回退到明文 insecure 连接，这和高对抗目标冲突。

**整改动作**

- 生产模式下强制要求 mTLS
- 只要 TLS 不完整、证书失效、CA 不匹配，就拒绝连接
- 禁止自动回退到 insecure gRPC
- 启动失败要明显报错，而不是静默降级成不安全模式

**验收结果**

- 非可信 OS 无法接入 Gateway 控制链路
- 配置缺失时 Gateway 无法以不安全模式继续运行

### Gateway-Immediate-2：让证书钉扎真正生效

**问题**

当前证书钉扎虽然初始化了，但没有真正进入连接校验链。

**整改动作**

- 在 Gateway 连接 OS 的握手完成后执行证书指纹校验
- 钉扎失败立即断开连接并记录安全事件
- 钉扎校验结果纳入健康状态与告警

**验收结果**

- 中间人证书与错误上游证书无法通过握手后的业务校验
- 证书异常会触发显式安全告警

### Gateway-Immediate-3：把“入口感知”改成“入口可拒绝”

**问题**

当前 ingress 检测能识别部分异常扫描，但大多只是上报后继续放行。

**整改动作**

- 对明显的扫描流量先定义第一批强规则
- 对高确定性异常流量直接丢弃或静默丢弃
- 对中等风险来源至少执行限速、延迟、隔离或蜜罐引流
- 明确入口策略分级：`放行 / 观察 / 限制 / 引流 / 阻断`

**验收结果**

- 已知明显扫描流量不再一律放行
- 高风险探测在入口即可被压制

### Gateway-Immediate-4：先修复黑名单下发到数据面的断点

**问题**

当前黑名单管理器已经存在，但数据面生效链路并不完整，容易形成“逻辑上已封、实际上未封”。

**整改动作**

- 补齐黑名单 eBPF map 定义与加载
- 确认黑名单命中后能在入口路径真正生效
- 为黑名单同步增加成功/失败统计

**验收结果**

- 黑名单写入后能在数据面立即命中
- 命中黑名单的流量会被稳定拒绝

### Gateway-Immediate-5：把对抗模式切换与 kill / 焦土指令收敛到唯一可信来源

**问题**

当前战术切换、下行控制、自毁类动作的触发路径较多，来源约束不够强。

**整改动作**

- 规定 kill / 焦土 / 模式切换只能来自唯一可信控制面
- 所有高危命令必须带明确身份与签名
- 高危命令必须写安全审计日志

**验收结果**

- Gateway 不再接受来源不明的高危控制动作
- 所有高危动作都能追溯“是谁、何时、从哪里发出的”

## 七、两周内修

这一层用于把“已封高危洞”升级成“比较完整、比较稳”的安全闭环。

---

## 八、OS 两周内修

### OS-2W-1：建立统一的 RBAC 与资源归属模型

**整改动作**

- 明确系统角色，例如 `user / admin / operator / auditor`
- 将权限校验从零散控制器提升为统一中间件或守卫
- 为关键资源加 owner check 与 role check 双校验

**验收结果**

- 同一类资源的权限策略一致
- 不再出现“挂了 JWT 但边界不清”的情况

### OS-2W-2：为内部服务建立统一鉴权标准

**整改动作**

- 内部 HTTP 统一要求 shared secret 或 mTLS
- 内部 gRPC 统一要求双向身份校验
- 所有内部服务增加来源白名单与访问审计

**验收结果**

- OS 内部不再存在“裸内部服务”
- 任一内部请求都能识别调用方身份

### OS-2W-3：建立控制面审计日志

**整改动作**

- 记录所有敏感动作，例如：
  - 用户封禁/解封
  - 网关 kill
  - 策略下发
  - 等级切换
  - 高危指令触发
- 审计日志至少包含操作者、时间、来源、目标、动作结果

**验收结果**

- 每个敏感动作都能追溯
- 安全部门或运维能在事后回放关键操作链路

### OS-2W-4：建立“异常来源到 Gateway 封禁”的统一回路

**整改动作**

- OS 侧威胁情报与 Gateway 侧黑名单统一建模
- 建立从威胁事件到封禁指令的标准回路
- 支持封禁 TTL、自动解封、人工解除与同步状态回写

**验收结果**

- OS 能把识别到的高风险来源稳定下发到 Gateway
- 封禁不是一次性动作，而是可管理状态

### OS-2W-5：增加基本安全基线

**整改动作**

- API 服务启用输入校验
- 补齐基础请求限流
- 增加安全 Header、CORS 收敛、错误响应脱敏
- 将开发默认密钥、默认口令、默认数据库配置全部移出生产路径

**验收结果**

- 生产环境不存在默认密钥与默认弱配置
- API 的输入与输出边界更清晰、更保守

---

## 九、Gateway 两周内修

### Gateway-2W-1：建立标准的入口处置策略

**整改动作**

- 为探测流量、异常指纹、黑名单命中、蜜罐命中分别定义标准处置方式
- 把入口策略固定为几种可配置动作：
  - 直接拒绝
  - 静默丢弃
  - 人工可见告警
  - 隔离观察
  - 引流蜜罐

**验收结果**

- 面对异常来源时系统行为一致，不再依赖临时处理

### Gateway-2W-2：补齐蜜罐、指纹、黑名单三者联动

**整改动作**

- 让蜜罐命中事件能够反哺威胁分析器
- 让高危指纹可自动推动黑名单更新
- 让引流与封禁之间形成可配置策略，而不是独立模块各自工作

**验收结果**

- 蜜罐不再只是展示能力，而是进入处置链路
- 同一异常来源会在多个模块之间形成统一风险视图

### Gateway-2W-3：建立 Gateway 本地安全状态机

**整改动作**

- 为 Gateway 定义标准运行态，例如：
  - 正常态
  - 警戒态
  - 高压态
  - 隔离态
  - 静默态
- 不同状态对应不同入口策略、对外暴露面、对抗强度

**验收结果**

- Gateway 在压力上升时能自动进入更保守的运行状态
- 高对抗能力开始体现为整体行为，而不是散落配置

### Gateway-2W-4：为关键安全动作补观测指标

**整改动作**

- 增加以下核心指标：
  - 入口拒绝数
  - 蜜罐引流数
  - 黑名单命中数
  - 威胁升级次数
  - 控制面鉴权失败数
  - mTLS 异常次数

**验收结果**

- 安全能力从“感觉存在”变成“可量化”
- 运营和研发都能明确看到安全改造收益

### Gateway-2W-5：补充安全回归测试

**整改动作**

- 增加未授权控制面接入测试
- 增加异常流量入口拒绝测试
- 增加黑名单生效测试
- 增加 mTLS 缺失与证书错误测试

**验收结果**

- 关键安全能力可通过自动化验证
- 后续改动不会轻易把安全链路改坏

## 十、版本级改造

这一层是为了把 Mirage 的“高对抗”真正做成系统能力，而不是局部功能。

---

## 十一、OS 版本级改造

### OS-Version-1：建立统一的安全控制平面

**目标**

把 Mirage OS 从“多个服务并存”升级成“有统一策略、统一身份、统一审计、统一下发”的安全控制中心。

**改造方向**

- 统一内部身份体系
- 统一控制面协议
- 统一高危命令审批与审计
- 统一 Gateway 安全状态编排

### OS-Version-2：把安全策略产品化

**目标**

让不同风险等级、不同付费等级、不同部署环境可以使用不同安全策略，而不是硬编码在服务中。

**改造方向**

- 支持按用户等级、Cell 等级、区域、场景分配安全策略
- 支持不同 Gateway 使用不同防护模板
- 支持策略灰度、回滚、版本化管理

### OS-Version-3：建立跨节点威胁情报中心

**目标**

将单个 Gateway 看到的异常，升级为整个 Mirage 网络共享的防御资产。

**改造方向**

- 汇总多 Gateway 的异常来源
- 统一风险评分
- 统一封禁同步
- 统一误判回收

---

## 十二、Gateway 版本级改造

### Gateway-Version-1：实现真正的分层入口防御

**目标**

把 Gateway 防御从“发现异常再处理”升级成“入口即分流、入口即分级、入口即拒绝”。

**改造方向**

- 在 eBPF / 数据面中实现更稳定的入口分类
- 按风险执行不同动作
- 将高确定性异常直接在最前层处理掉

### Gateway-Version-2：把高对抗做成状态驱动系统

**目标**

让 Mirage 的高对抗能力体现在系统自动切换，而不是只体现在零散配置项。

**改造方向**

- 根据威胁等级自动调整暴露面
- 根据压力动态切换对抗参数
- 根据控制面可达性与探测强度进入不同保守模式

### Gateway-Version-3：建立可验证的高对抗指标体系

**目标**

未来对外讲“高对抗”，不能只讲概念，要有数据证明。

**建议指标**

- 未授权控制面拒绝率
- 黑名单命中后的入口阻断延迟
- 蜜罐命中到正式封禁耗时
- 异常来源持续探测存活时间
- 控制面失败时的安全降级成功率
- 高压状态下的可用性与误伤率

### Gateway-Version-4：建立误判修正机制

**目标**

高对抗系统不能只有“封”，还要有“验证与释放”。

**改造方向**

- 建立观察区与隔离区
- 支持短期封禁与自动释放
- 支持白名单、可信指纹与人工复核

---

## 十三、建议排期

### 第一阶段：立刻修

建议本周内完成，目标是先封洞：

- 关闭权限升级路径
- 给管理接口补角色校验
- WebSocket 改只读
- 内部 HTTP / gRPC 收口
- Gateway 禁止明文回退
- 修复黑名单到数据面的断点

### 第二阶段：两周内修

建议作为下一轮稳定性与安全版本：

- 统一 RBAC
- 统一内部服务鉴权
- 建立审计日志
- 建立入口处置策略
- 打通蜜罐、指纹、黑名单联动
- 补齐安全回归测试

### 第三阶段：版本级改造

建议作为安全专项版本推进：

- 统一安全控制平面
- 高对抗状态机
- 跨节点威胁情报中心
- 可验证的高对抗指标体系

---

## 十四、最终目标

完成以上整改后，Mirage 的安全能力应该从“模块很多”升级为“系统可信”。

对内应达到：

- 未授权入口进不来
- 异常流量走不深
- 高危命令发不乱
- 所有关键动作查得到
- 安全状态能量化、能回归、能演练

对外才有资格真正说：

**Mirage 不只是具备高对抗模块，而是具备高对抗运行能力。**
