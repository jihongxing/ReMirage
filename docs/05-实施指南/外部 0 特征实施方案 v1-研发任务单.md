# 外部 0 特征实施方案 v1 — 研发任务单

## 1. 文档说明

本文档将“外部 0 特征”目标收敛为一份可直接执行的研发任务单。

本轮目标不是追求数学意义上的“绝对不可识别”，而是优先完成**外部强特征清零**：

- 默认主路径表现为标准 `QUIC/H3` 背景流量
- 包长分布不出现固定尖峰
- IAT 不表现为明显程序化发送模式
- 主动探测得到合法 `h3` 行为或静默丢弃
- 路径切换不出现明显“流量潮汐”

本文档同时包含：

- `P0 / P1 / P2` 分阶段实施方案
- 按模块拆分的研发执行表
- 按文件拆解的具体研发任务单

---

## 2. 适用范围

本方案严格限定于**外部网络观察者**视角可见的特征：

- 被动抓包观察
- 常规主动探测
- 短期统计聚类

不包含以下范围：

- 主机侧日志泄露
- 本地内存取证
- eBPF Map 内部值暴露
- 节点失陷后的取证问题

这些内容应继续归入主机安全和运维安全专项。

---

## 3. 现状判断

按当前代码事实，项目已经具备“外部 0 特征 v1”所需的部分基础能力，但尚未形成完整闭环：

- Client 侧 `QUIC` 主链已统一到 `h3` 外观
- Client 侧发送链已存在 `SendPathShim`，支持 `Padding + IAT`
- Gateway 已具备公网 `QUIC/H3 bearer listener`
- Gateway 已具备 `UDPPreFilter + QUICPostAcceptValidator`
- eBPF 侧已具备 `B-DNA / NPM / Jitter / Persona` 的配置与写图通路

当前核心差距不再是“完全没有能力”，而是以下三类问题：

1. 默认主路径虽然可用，但发送整形和拟态仍未默认化
2. Gateway bearer、Orchestrator、StealthControlPlane、Persona 仍未完全闭环
3. 切换事务尚未真正平滑化，仍可能暴露状态突变

---

## 4. 优先级定义

- `P0`：上线前必须完成，否则“外部 0 特征 v1”不能成立
- `P1`：主路径可运行但切换与控制面仍会暴露特征，需在联调阶段完成
- `P2`：深度拟态增强项，不阻断 v1 成立，但影响长期对抗能力

---

## 5. 分阶段实施方案

### P0

| 模块 | 研发动作 | 验收项 | 主要风险 |
|---|---|---|---|
| Client 默认发送链 | 把 `SendPathShim` 从“可选能力”升为默认主链能力；为不同网络画像设置 `PaddingMean/PaddingStddev/IATMode/IATMeanUs` 默认值；将 `phantom-client/pkg/gtclient/client.go` 的 passthrough 默认改成受控默认，核心实现落在 `send_path_shim.go` | Client 出口抓包不再出现明显固定包长尖峰；IAT 分布不呈现程序化直线；发送链配置可热更新 | 参数选得过激会抬高 RTT、降低吞吐、影响弱网稳定性 |
| Client QUIC 外观基线 | 固化 Client QUIC 外观基线：`ALPN=h3`、`InitialPacketSize=1200`、窗口参数、CID 长度；以 `quic_engine.go` 为唯一真相 | pcap 中默认主链统一表现为标准 `h3`；与当前审计文档一致 | `quic-go` 仍有残余指纹差异，无法一次性做到 Chrome 级完全拟态 |
| Gateway 公网主入口 | 把 `QUIC/H3 bearer listener` 定成默认公网主入口；补启动失败日志、证书缺失告警、配置说明 | 在标准部署配置下，443/UDP 为默认主路径；启动日志能明确显示 listener 状态 | 无证书、证书错误、环境端口冲突会让主入口退化回 WSS |
| Gateway 数据面闭环 | 固化 `QUIC/H3 listener -> Orchestrator -> DataPlane Injector` 主链，补失败路径观测点 | 入站 QUIC Datagram 能稳定进入数据面；数据面未接好时告警清晰 | TUN/NFQUEUE 不可用时容易出现“建链成功、数据不通” |
| 抓包验收基线 | 建立三类固定抓包：Client 出口、Gateway 公网入口、切换瞬间 | 每次改动都能产出可比较的 pcap/统计结果 | 没有自动化抓包回归时，后续改动容易回退特征 |

### P1

| 模块 | 研发动作 | 验收项 | 主要风险 |
|---|---|---|---|
| 原子切换事务 | 把 `dual-send + epoch barrier + MTU/FEC 调整 + activePath 切换` 接进 `orchestrator.go` 的 promote/demote 主流程 | 切换期间无明显流量潮汐、无大面积丢包、无突兀包长分布跳变 | 事务接得不完整会出现“逻辑切换成功、网络外观暴露” |
| Persona 驱动链 | 将 persona 切换真正落到 `fingerprint_map / jitter_config_map / npm_config_map` | persona 切换后，抓包能看到受控变化；eBPF map 与运行时行为一致 | 只有 map 更新但 bearer path 不生效，会造成“控制面变了，数据面没变” |
| Stealth Control Plane 激活 | 让 `StealthControlPlane` 从 `ChannelQueued` 变成真正挂到公网 bearer 的工作态 | 控制命令不再长期排队；bearer 恢复后可自动 drain | 若绑定方式不对，会在恢复时出现回放乱序或旧命令污染 |
| 主动探测防护 | 统一 `UDPPreFilter + QUICPostAcceptValidator + h3 合法响应` 行为 | 非法探测静默丢弃或快速关闭；合法 h3 返回一致的 `403/404` | 不同行为分支若不一致，会成为新的主动探测特征 |
| WSS 降级外观 | 不把 WSS 当主路径，但补齐它的 TLS/行为一致性，避免降级时瞬间暴露强特征 | QUIC 不可用时降级到 WSS 仍能通过基本探测 | 继续使用 Go 原生 TLS 指纹会让降级链明显暴露 |

### P2

| 模块 | 研发动作 | 验收项 | 主要风险 |
|---|---|---|---|
| QUIC 深度拟态 | 评估 fork `quic-go` 或引入更强拟态能力，处理 `GREASE`、TP 顺序、`max_udp_payload_size` 等残余差异 | JA4Q/精细指纹差异继续缩小 | 成本高、维护重，可能引入兼容性债务 |
| 模板池扩容 | 将 B-DNA 模板从少量预设扩到版本化模板池，按浏览器/平台分桶轮换 | 长期观察下不再只聚类到少数几个固定画像 | 模板采集、更新、验收成本高 |
| IAT/包长模型升级 | 从“单分布拟态”升级到分场景模型，区分空闲、轻载、突发、恢复态 | 长时间统计下分布更接近真实浏览器会话 | 模型过拟合会牺牲性能或稳定性 |
| Client 特权增强项 | 仅在企业 Linux 场景增加 client-side eBPF 作为增强层，不作为默认依赖 | 有权限环境下进一步压低出口特征 | 终端权限不可控，不能反过来绑架默认方案 |

## 8. 统一验收模板

每个任务完成后，统一按以下四类标准验收：

### 8.1 代码验收

- 单测通过
- 集成测试通过
- 不破坏默认主路径连通性

### 8.2 行为验收

- 有对应抓包结果
- 能说明“改动前 vs 改动后”的网络外观差异

### 8.3 文档验收

- `docs/外部零特征消除审计与整改清单.md` 状态同步更新
- 过时结论及时关闭或改写

### 8.4 回归验收

- 不破坏默认建链
- 不显著恶化吞吐与 RTT
- 不因降级链激活引入新的强特征

## 9. 阶段性门槛

### P0 通过门槛

- 默认主路径只走 `QUIC/H3`
- Client 与 Gateway 双侧抓包没有明显强特征尖峰
- 主动探测行为稳定，不暴露异常分支

### P1 通过门槛

- 切换、降级、恢复不再出现肉眼可识别的流量突变
- 控制面与 persona 切换进入真实运行闭环

### P2 通过门槛

- 剩余差异只落在高成本精细指纹层
- 不再是低成本筛查或短期聚类目标

---

## 10. 一句话执行原则

先把“**默认主路径看起来像正常 H3**”做扎实，再做“**切换时看不出系统在切换**”，最后才是“**逼近浏览器级深度拟态**”。
