# 运营前安全审计 — 可执行任务清单

## 1. 文档说明

本文档基于 `docs/Mirage 运营前高频攻击面检查清单.md` 中定义的 18 个任务（T01~T18），对当前仓库代码进行了逐文件审计，将审计发现转化为可直接用于 Spec 生成和开发执行的任务清单。

每个任务包含：审计发现（代码级证据）、可执行开发子任务、涉及文件、验收标准。

### T01 统一 OS 内部入口的鉴权与网络边界

#### 审计发现

1. `xmr-webhook.controller.ts` 的 `POST /webhook/xmr/confirmed` 完全裸露，无任何鉴权装饰器（无 `@UseGuards`），任何能访问该端口的请求方可直接调用并给用户加余额
2. `delivery.controller.ts` 的 `GET /delivery/:token` 同样无鉴权，内部调用 `PROVISIONER_URL` 使用明文 HTTP（默认 `http://localhost:18443`），无签名、无重放保护
3. `bridge-client.service.ts` 仅靠静态 `X-Internal-Secret` header 鉴权，且 secret 为空时仅打 warn 日志不阻断请求
4. `main.ts` 中所有路由统一挂在 `/api` 前缀下，webhook 和 delivery 与用户 API 共享同一公网入口

#### 可执行子任务

- [ ] T01-1: 新增 `InternalHMACGuard` 中间件，校验 `HMAC-SHA256(timestamp + nonce + body_hash)`，5 分钟防重放窗口
- [ ] T01-2: 给 `XMRWebhookController` 和 `DeliveryController` 挂载 `InternalHMACGuard`
- [ ] T01-3: `DeliveryController` 内部 `fetch(PROVISIONER_URL)` 调用增加 HMAC 签名 header
- [ ] T01-4: `BridgeClientService` 的 `BRIDGE_INTERNAL_SECRET` 为空时启动直接抛异常，不允许降级
- [ ] T01-5: 将 `/webhook/*` 和 `/delivery/*` 从公网 `/api` 前缀分离，绑定到独立的内部监听端口或 Unix socket

#### 涉及文件

- `mirage-os/api-server/src/modules/billing/xmr-webhook.controller.ts`
- `mirage-os/api-server/src/modules/billing/delivery.controller.ts`
- `mirage-os/api-server/src/modules/gateways/bridge-client.service.ts`
- `mirage-os/api-server/src/main.ts`
- 新增 `mirage-os/api-server/src/common/internal-hmac.guard.ts`

#### 验收标准

- 未携带内部签名或签名无效的请求一律返回 `401/403`
- 重放相同请求在窗口期内只允许通过一次
- `/webhook/*` 和 `/delivery/*` 不再直接暴露给公网入口

### T02 把 XMR 充值结算收口成单一真相源

#### 审计发现

1. `xmr-webhook.controller.ts` 第 37-43 行：直接 `updateMany` 把 deposit 标记 CONFIRMED 并 `increment` 用户余额，无状态前置条件检查（未限制 `PENDING -> CONFIRMED`），可被重复调用导致余额重复增加
2. `monero_manager.go` 的 `confirmDeposit()` 有事务但 `Updates` 中无 `WHERE status = 'pending'` 条件
3. `schema.prisma` 的 `Deposit.txHash` 是 `@unique`，但缺少 `transfer_index` 字段，同一笔 tx 的多个 output 无法区分
4. `GenerateDepositAddress()` 是用户级地址（一个用户一个固定地址），不是订单级子地址

#### 可执行子任务

- [ ] T02-1: `xmr-webhook.controller.ts` 废弃余额落账逻辑，只保留通知功能（推送 WebSocket 事件），不再直接改余额
- [ ] T02-2: `monero_manager.go` 的 `confirmDeposit()` 增加 `WHERE status = 'pending'` 条件，确保只有 `PENDING -> CONFIRMED` 可落账
- [ ] T02-3: `schema.prisma` 的 `Deposit` 增加 `transferIndex Int? @map("transfer_index")`，唯一键改为 `@@unique([txHash, transferIndex])`
- [ ] T02-4: `GenerateDepositAddress()` 改为订单级子地址模型，每次充值请求生成新子地址并关联到 Deposit 记录

#### 涉及文件

- `mirage-os/api-server/src/modules/billing/xmr-webhook.controller.ts`
- `mirage-os/services/billing/monero_manager.go`
- `mirage-os/api-server/src/prisma/schema.prisma`

#### 验收标准

- 同一笔链上转账无论轮询多少次，余额只能增加一次
- Webhook 即使被重复打或伪造，也不能改余额
- 数据库能明确追溯"哪一笔链上转账对应哪一笔订单入账"

### T04 把信令共振补成世代化防重放

#### 审计发现

1. `signal_crypto.go` 的 `SignalPayload` 只有 `Timestamp` + `TTL`，无 `Epoch`、`ManifestID`、`ExpireAt` 字段
2. 反重放仅依赖内存变量 `lastAcceptedTimestamp`，Client 重启后归零，旧但签名合法的信令可再次被接受
3. `resolver.go` 无 Epoch 校验逻辑，无本地持久化 `CurrentEpoch`
4. `doh.go`、`gist.go`、`mastodon.go` 各通道实现中无世代校验

#### 可执行子任务

- [ ] T04-1: `signal_crypto.go` 的 `SignalPayload` 增加 `Epoch uint64`、`ManifestID [16]byte`、`ExpireAt int64`
- [ ] T04-2: `SerializePayload` / `DeserializePayload` 增加新字段的序列化/反序列化
- [ ] T04-3: `OpenSignal()` 增加 Epoch 校验：只接受 `>= CurrentEpoch`，同 Epoch 要求更高 Timestamp
- [ ] T04-4: `OpenSignal()` 增加 `ExpireAt` 校验：过期信令即使验签通过也丢弃
- [ ] T04-5: Client 侧持久化 `CurrentEpoch` 到本地文件（`~/.phantom-client/epoch`），重启后加载
- [ ] T04-6: `resolver.go` 增加 `validateEpoch()` 方法，在 `openFn` 返回后校验世代

#### 涉及文件

- `mirage-gateway/pkg/gswitch/signal_crypto.go`
- `phantom-client/pkg/resonance/resolver.go`
- `phantom-client/pkg/resonance/doh.go`
- `phantom-client/pkg/resonance/gist.go`
- `phantom-client/pkg/resonance/mastodon.go`
- `phantom-client/pkg/persist/`（持久化 Epoch）

#### 验收标准

- 回放三天前的合法信令，Client 必须拒绝
- Client 重启后仍能拒绝旧信令
- 新信令发布后，旧 Epoch 不能再次覆盖路由表

3. 仓库内无独立的 eBPF 程序级健康状态跟踪，一个组件异常可能拖垮整套数据面

#### 可执行子任务

- [ ] T07-1: `loader.go` 在 `LoadAndAttach()` 前增加 `canaryAttach()` 方法：创建 `dummy0` 虚拟接口，先挂载所有程序跑自检，通过后再挂生产网卡

- [ ] T07-3: 达阈值时自动 `detach` 对应程序（`netlink.FilterDel` / `link.Close`），切换到 iptables fallback
- [ ] T07-4: 每个 BPF 程序增加独立健康状态标记（`map[string]HealthStatus`），异常时只摘单个程序
- [ ] T07-5: 启动日志明确输出 canary 校验结果，生产挂载前必须看到 `canary passed`

### T13 收口控制面授权链，移除"任意公钥用户即管理员"的路径

#### 审计发现

1. `breach.service.ts` 约第 55 行：`findMany({ where: { ed25519Pubkey: { not: null }, isActive: true } })` 遍历所有有公钥的活跃用户，验签通过直接签发 `role: 'admin'` — **任何绑定 Ed25519 公钥的普通用户都能拿到 admin token**
2. `auth.controller.ts`：`GET /auth/challenge` 和 `POST /auth/breach` 公开暴露，无角色限制
3. `gateway-bridge/pkg/grpc/server.go`：TLS 使用 `credentials.NewServerTLSFromFile()`，不是严格 mTLS（虽然有 CN 白名单拦截器，但 TLS 层本身不强制客户端证书）
4. `mirage-os/configs/mirage-os.yaml`：`grpc.tls_enabled: false` 作为默认值
5. `gateway/cmd/gateway/main.go` 约第 350 行：`if cfg.Security.CommandSecret != ""` 才启用签名校验 — secret 为空时高危命令无校验
6. `command_auth.go`：HMAC 只覆盖 `commandType + timestamp`，不覆盖 payload 和 nonce，无重放缓存

#### 可执行子任务

- [ ] T13-1: `breach.service.ts` 增加角色过滤：只允许专门的 operator/admin 角色用户通过 breach 认证，不再复用普通用户公钥表。方案：新增 `OperatorKey` 模型或在 `User` 上增加 `isOperator Boolean @default(false)` 字段
- [ ] T13-2: `gateway-bridge/pkg/grpc/server.go` TLS 改为使用 `tls.RequireAndVerifyClientCert`（`tls_manager.go` 的 `GetServerTLSConfig()` 已支持，但 `server.go` 未使用该方法）
- [ ] T13-3: `config.go` 的 `validate()` 增加生产模式校验：`TLSEnabled` 为 false 时直接返回错误
- [ ] T13-4: `mirage-os/configs/mirage-os.yaml` 移除 `tls_enabled: false` 默认值，改为必填项
- [ ] T13-5: `gateway/cmd/gateway/main.go`：`CommandSecret` 为空时 Gateway 启动失败（移除 `if != ""` 条件判断）
- [ ] T13-6: `command_auth.go` HMAC 覆盖范围扩展为 `commandType + timestamp + nonce + SHA256(payload)`，增加 nonce 重放缓存（内存 LRU，TTL 120s）

#### 涉及文件

- `mirage-os/api-server/src/modules/auth/breach.service.ts`
- `mirage-os/api-server/src/modules/auth/auth.controller.ts`

#### 涉及文件
