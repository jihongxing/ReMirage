# Mirage Project 定价策略与计费架构

## 核心理念

**防御不是免费的，安全需要成本对冲。**

Mirage Project 采用按量计费模式，将技术优势（高性能转发）直接转化为财务优势，确保系统长久生存能力。

---

## 一、定价策略

### 1.1 成本构成

#### 业务流量（Base Traffic）
- 用户真正发送/接收的数据
- 定价：**$0.10/GB**

#### 防御冗余流量（Defense Overhead）
- NPM 填充：10%-35% 额外流量
- G-Tunnel FEC：10%-20% 冗余包
- 定价：**$0.05/GB**（半价，因为是防御成本）

### 1.2 分级定价（蜂窝级别）

| 蜂窝级别 | 节点质量 | 防御能力 | 业务流量单价 | 防御流量单价 |
|---------|---------|---------|------------|------------|
| **标准蜂窝** | 普通节点 | 基础扰动 | $0.10/GB | $0.05/GB |
| **白金蜂窝** | 专线链路 | AVX-512 极限纠错 | $0.15/GB | $0.08/GB |
| **钻石蜂窝** | 全球转生频率最高 | 物理主权隔离 | $0.20/GB | $0.10/GB |

### 1.3 流量包优惠

| 流量包 | 总流量 | 原价 | 优惠价 | 折扣 |
|-------|-------|------|-------|------|
| 入门包 | 100GB | $10 | $8 | 20% |
| 标准包 | 500GB | $50 | $35 | 30% |
| 企业包 | 2TB | $200 | $120 | 40% |
| 定制包 | 10TB+ | 协商 | 协商 | 50%+ |

## 二、技术实现

### 2.1 内核态计费探针（eBPF）

#### 计费 Map 定义

```c
// common.h 新增
struct {
    __uint(type, BPF_MAP_TYPE_PERCPU_ARRAY);
    __uint(max_entries, 2);
    __type(key, __u32);
    __type(value, __u64);
} traffic_stats SEC(".maps");

// 0: 业务流量（原始数据）
// 1: 防御流量（NPM 填充 + FEC 冗余）
```

#### 计费逻辑

```c
// jitter.c 出口处
SEC("tc")
int jitter_lite_egress(struct __sk_buff *skb) {
    // ... 现有逻辑 ...
    
    // 记录业务流量
    __u32 key_base = 0;
    __u64 *base_bytes = bpf_map_lookup_elem(&traffic_stats, &key_base);
    if (base_bytes) {
        __sync_fetch_and_add(base_bytes, skb->len);
    }
    
    // 如果应用了 NPM 填充，记录防御流量
    if (padding_applied) {
        __u32 key_defense = 1;
        __u64 *defense_bytes = bpf_map_lookup_elem(&traffic_stats, &key_defense);
        if (defense_bytes) {
            __sync_fetch_and_add(defense_bytes, padding_size);
        }
    }
    
    return TC_ACT_OK;
}
```

### 2.3 实时成本监控

```go
// 每分钟从内核拉取流量统计
func (cc *CostCalculator) PullKernelStats() {
    baseKey := uint32(0)
    defenseKey := uint32(1)
    
    var baseBytes, defenseBytes uint64
    cc.loader.GetMap("traffic_stats").Lookup(&baseKey, &baseBytes)
    cc.loader.GetMap("traffic_stats").Lookup(&defenseKey, &defenseBytes)
    
    cc.baseTraffic = float64(baseBytes) / (1024 * 1024 * 1024)
    cc.defenseTraffic = float64(defenseBytes) / (1024 * 1024 * 1024)
}
```

### 3.2 成本控制策略

#### 模式 1：成本优先

- 防御等级上限：20%
- 超预算自动降级
- 月成本可控

#### 模式 2：安全优先

- 防御等级无上限
- 威胁自动提升到 100%
- 成本不设限

#### 模式 3：平衡模式（默认）

- 防御等级：20%-30%
- 月成本告警：超过 $200 提示
- 超过 $500 自动降级

### 3.3 欠费熔断机制

#### 内核态阻断（eBPF）

```c
// 查询配额状态
struct quota_status {
    __u32 enabled;      // 是否启用
    __u64 remaining_gb; // 剩余流量 (GB)
};

SEC("tc")
int quota_check(struct __sk_buff *skb) {
    struct quota_status *quota = bpf_map_lookup_elem(&quota_map, &key);
    
    if (quota && quota->enabled && quota->remaining_gb == 0) {
        // 欠费：返回 HTTP 302 重定向到充值页面
        return TC_ACT_REDIRECT;
    }
    
    return TC_ACT_OK;
}
```

#### 用户态通知（Go）

```go
// 流量包用尽时

    // 3. 返回引导页面
```

## 四、Mirage-OS 计费模块

### 4.1 核心服务

#### Billing-Service

- 接收 Gateway 上传的流量统计
- 实时计算费用
- 生成账单

#### Quota-Manager

- 管理用户流量包
- 实时扣减配额

| 对比项 | AWS/GCP | Mirage Project |

## 八、总结

**按量计费 = 技术优势 × 财务优势**

- 成本对冲：防御强度与资源消耗正相关
- 资源公平：避免算力霸占和带宽投机
- 用户友好：符合"无感接入"的心理预期
- 长久生存：确保系统可持续运营
