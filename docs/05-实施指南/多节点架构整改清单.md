# Mirage 多节点架构整改清单

## 一、文档目标

本文档用于承接本次针对目标拓扑的代码审计结论，并将问题整理为可执行的架构整改清单。

目标拓扑如下：

```text
1 个 Mirage OS
N 个 Mirage Gateway
N * M 个 Phantom Client
```

这意味着 Mirage 必须从“小规模可运行”升级为“多网关、多客户端、共享与隔离并存”的真正基础设施架构。

当前 Linux 上的小规模 e2e 已通过，但现状只能证明：

- 1 个 OS 可运行
- 少量 Gateway 可接入
- 单个 Client 可完成基础连接

还不能证明：

- 1 个 OS 能稳定管理 N 个 Gateway
- 1 个 Gateway 能正确承载 M 个 Client
- 多 Client 的配额、计费、状态和路由在共享 Gateway 场景下仍然准确

因此，本次整改的核心目标不是“补几个测试”，而是让系统的数据模型、协议模型、控制模型和运行模型真正匹配目标拓扑。

---

## 二、当前问题总览

本次审计得到的关键结论可以归纳为 5 类：

1. 配额作用域错误
   当前配额主要以 Gateway 为粒度生效，而不是以用户、客户端或会话为粒度隔离。

2. 计费归属不准确
   当前流量上报缺少精确用户身份，OS 在共享 Gateway 场景下无法正确知道“这笔流量属于谁”。

3. 共享 Gateway 的数据模型不完整
   Gateway 可以承载多个连接，但数据库模型仍然更接近“一个 Gateway 绑定一个用户”的思路。

4. OS 到 Gateway 的控制面链路不完整
   目前控制中心更擅长接收上报，尚未形成稳定的“按 Gateway / 按 Cell / 按拓扑下推”的可靠控制平面。

5. Client 对多 Gateway 拓扑的学习能力不足
   当前客户端主要依赖启动时 bootstrap 池，缺少稳定的运行时拓扑学习与更新机制。

---

## 三、整改原则

在进入分层整改前，先明确本轮架构整改的 5 条原则：

### 3.1 用户隔离优先于节点共享

只要一个 Gateway 可以承载多个 Client，就必须先解决用户级别的配额、计费、归属和状态隔离问题。不能继续使用“节点级共享状态”代替“用户级真实状态”。

### 3.2 Gateway 是承载节点，不是用户主键

Gateway 应该被看作网络承载单元，而不是用户归属单元。一个 Gateway 上可以承载多个用户、多个客户端、多个会话，因此所有业务归属都不能再只靠 `gateway_id` 反推。

### 3.3 OS 必须成为真正的控制平面

Mirage OS 不能只收心跳、看状态、记录日志。它必须具备：

- 注册和识别 Gateway
- 维护拓扑关系
- 按维度下推状态
- 对多节点进行一致管理

否则就不是真正意义上的 `1 OS + N Gateway`。

### 3.4 Client 必须学习运行时拓扑

Client 不能只知道“启动时那几个地址”。在目标架构下，Client 必须能在运行过程中学习、更新、切换和丢弃 Gateway 信息。

### 3.5 协议、数据库、控制面必须同时升级

这不是单点修复问题。仅修改某一层都不够：

- 只改 Proto，不改 DB，归属仍然落不住
- 只改 DB，不改 Gateway，流量仍然无法区分
- 只改 Client，不改 OS，拓扑仍然学不到

本次整改必须分层推进，但最终要形成闭环。

下面按 `OS / Gateway / Client / Proto / DB` 五层列出整改项。

## 五、OS 层整改项

### OS-1：建立真实的 Gateway 注册与拓扑索引

**优先级：P0**

**问题**

当前 OS 虽然能接收 Gateway 心跳，但“Gateway 属于哪个 Cell、地址是什么、当前是否可下推、当前是否可调度”的注册关系并没有形成统一、稳定、可查询的拓扑索引。

**整改目标**

让 Mirage OS 持有完整的 Gateway 拓扑视图，包括：

- `gateway_id`
- `cell_id`
- `downlink_addr`
- 当前在线状态
- 当前角色状态
- 当前承载能力

**改造项**

- 增加统一的 Gateway 注册流程，而不是仅靠心跳顺带更新部分状态
- 为 OS 内部建立标准的 Gateway 索引缓存
- Redis 中的 Gateway 地址、Cell 归属、在线状态必须由统一入口写入
- 下游所有 fan-out、路由、策略下发逻辑都必须基于这套索引，而不是各自猜测

**验收标准**

- 新 Gateway 接入后，OS 可在一个统一接口中看到其完整注册信息
- OS 可准确列出某个 Cell 下的全部在线 Gateway
- OS 能精确定位某个 Gateway 的下行可达地址

### OS-2：把“按节点下推”与“按 Cell 下推”做成正式能力

**优先级：P0**

**问题**

当前 OS 到 Gateway 的控制面下推链路不完整，部分接口还是占位实现，难以支撑 N 个 Gateway 的统一控制。

**整改目标**

Mirage OS 必须支持至少 3 种正式下推模式：

- 按单个 Gateway 下推
- 按 Cell 下推
- 按用户绑定的 Gateway 集合下推

**改造项**

- 明确控制平面的 fan-out 入口
- 将“策略下发”“配额下发”“黑名单下发”“事件类下发”拆分成正式能力
- 去掉“返回成功但实际未执行”的占位接口
- 建立 OS 侧的下推状态记录和失败重试机制

**验收标准**

- OS 可对指定 Cell 下所有在线 Gateway 完成一次真实下推
- 下推失败可被记录、重试、追踪
- Web Console/CLI 能看到下推结果而不是只看到请求成功

### OS-3：建立 Client / User / Gateway 的归属映射服务

**优先级：P0**

**问题**

共享 Gateway 架构下，OS 当前缺少“某个 Client 当前接在哪个 Gateway、属于哪个 User、使用哪个会话”的精确映射。

**整改目标**

OS 必须成为全局归属真相源，至少能回答：

- 某个客户端当前属于哪个用户
- 某个客户端当前接入哪个 Gateway
- 某个 Gateway 当前承载哪些客户端/会话
- 某个用户当前有多少活跃客户端

**改造项**

- 新增 Client 会话登记能力
- 新增 User-Gateway-Client 三元映射关系
- 将当前依赖 `gateway_id -> user_id` 反推的逻辑迁移为显式映射查询

**验收标准**

- OS 可以查询某个 Gateway 的当前客户端列表
- OS 可以查询某个 User 的当前会话分布
- 所有计费与配额操作不再依赖猜测用户

### OS-4：建立多网关下的控制面一致性验收

**优先级：P1**

**整改目标**

让 OS 在多 Gateway 架构下具备可验证的一致行为。

**改造项**

- 为下推链路增加审计日志
- 为拓扑变化增加状态事件
- 为 Gateway 注册/下线/漂移增加统一观测

**验收标准**

- 新增面向多节点控制平面的集成测试
- 新增“按 Cell 下推”“按 Gateway 下推”“Gateway 漂移重注册”的验收脚本

## 六、Gateway 层整改项

### Gateway-1：将配额从“Gateway 全局桶”升级为“用户/会话隔离桶”

**优先级：P0**

**问题**

当前 Gateway 内核与下行配额逻辑更接近“整台网关一个总桶”，这与共享 Gateway 架构完全不匹配。

**整改目标**

配额至少要支持按以下任一粒度隔离：

- `user_id`
- `client_id`
- `session_id`

推荐优先方案：

- 控制面按 `session_id` 下发
- Gateway 在数据面按 `session_id -> user_id` 做映射
- OS 最终按 `user_id` 汇总计费与熔断

**改造项**

- 重构 `quota_map` 的 key 设计，不再使用固定单 key
- 在 Gateway 内建立连接到用户/会话的映射关系
- 熔断判断必须只影响对应用户/会话，而不是影响整机

**验收标准**

- 同一台 Gateway 上 2 个客户端并发跑流量，一个耗尽配额不会影响另一个
- Gateway 内部可查询每个用户/会话的剩余配额

### Gateway-2：流量上报必须携带精确归属信息

**优先级：P0**

**问题**

Gateway 当前只上报 `gateway_id` 级别流量，无法支撑共享节点上的精确计费。

**整改目标**

Gateway 必须把流量统计拆分到用户或会话维度，再上报给 OS。

**改造项**

- 将统计对象从“整机总量”拆分为“按用户/按会话累计”
- 上报时附带精确归属信息
- 支持分批、增量、幂等上报

**验收标准**

- 同一 Gateway 上多个客户端并发时，OS 能分别看到各自流量
- 重新上报不会造成重复结算

### Gateway-3：建立正式的下行状态对齐模型

**优先级：P1**

**问题**

目标架构下，N 个 Gateway 不可能依赖零散命令式调用长期维持正确状态。

**整改目标**

Gateway 应以“期望状态对齐”为主要控制模式，而不是依赖临时命令。

**改造项**

- Gateway 启动时主动拉取自身期望状态
- 心跳中带状态摘要
- 发现状态不一致时自动对齐
- 事件类下发与状态类下发分离

**验收标准**

- Gateway 重启后无需人工干预即可恢复到 OS 期望状态
- 多 Gateway 同时重连时不会出现状态漂移

### Gateway-4：将用户会话从节点状态中解耦

**优先级：P1**

**问题**

当前很多状态仍然容易被理解为“某个 Gateway 只对应一个用户”，这会污染运维与观测。

**整改目标**

Gateway 侧状态分成两类：

- 节点状态：网关自身健康、资源、威胁等级、在线情况
- 会话状态：某个用户/客户端在此网关上的连接情况

**验收标准**

- 任何单个客户端断开都不会误伤整台 Gateway 的用户归属信息
- 监控面可分别查看节点级与会话级状态

## 七、Client 层整改项

### Client-1：修复多 Gateway bootstrap 的竞态连接问题

**优先级：P0**

**问题**

当前并发 bootstrap 探测存在“状态记录的获胜者”和“实际连接的获胜者”不一致的风险。

**整改目标**

保证客户端在 bootstrap 阶段：

- 只有一个最终胜出的连接
- `currentGateway`
- 实际数据通道
- kill switch 路由目标

这三者永远一致。

**改造项**

- 为 bootstrap 增加单一胜出者控制
- 失败或晚到的探测结果必须被丢弃
- 切换时必须保证状态原子更新

**验收标准**

- 3 个以上 Gateway 并发探测时，最终状态和实际连接一致
- 重复运行不出现随机错位

### Client-2：实现运行时拓扑学习能力

**优先级：P1**

**问题**

Client 目前基本只依赖 bootstrap 池，无法真正感知 N 个 Gateway 的运行时拓扑。

**整改目标**

Client 必须在正常运行中学会：

- 拉取最新 Gateway 列表
- 更新本地路由池
- 标记不可用节点
- 根据策略切换到更合适的节点

**改造项**

- 正式实现 `PullRouteTable`
- 定义客户端侧的运行时节点池
- 区分“启动入口池”和“运行时拓扑池”

**验收标准**

- Client 启动后可从 OS/Gateway 学到更多节点
- 某个 bootstrap 地址失效后，仍可依赖运行时路由池恢复

### Client-3：为客户端引入稳定身份标识

**优先级：P1**

**整改目标**

每个 Client 或每个会话都需要一个稳定身份，用于：

- 配额隔离
- 路由追踪
- 会话统计
- 精确恢复

**改造项**

- 引入 `client_id` 或 `session_id`
- 在连接建立、切换、重连时保持身份连续性

**验收标准**

- OS 能追踪单个 Client 的迁移过程
- Client 切换 Gateway 时仍被视为同一会话

## 八、Proto 层整改项

### Proto-1：TrafficRequest 必须支持精确归属

**优先级：P0**

**问题**

当前流量上报只带 `gateway_id`，无法支撑共享 Gateway 架构。

**整改目标**

`TrafficRequest` 至少增加以下字段中的一组：

- `user_id`
- `client_id`
- `session_id`

推荐最小升级方案：

- `gateway_id`
- `user_id`
- `session_id`
- 分段流量统计

**验收标准**

- OS 侧不再需要通过 Gateway 猜用户
- 共享 Gateway 上的并发客户端都能独立计费

### Proto-2：Heartbeat 与状态同步消息要补齐拓扑语义

**优先级：P1**

**整改目标**

让心跳不再只是“我活着”，还要能表达：

- 我的下行可达地址是什么
- 我属于哪个 Cell
- 我当前承载多少客户端/会话
- 我当前状态版本是什么

**验收标准**

- OS 不再依赖外部猜测或 Redis 偶然缓存来重建 Gateway 拓扑

### Proto-3：新增路由表/拓扑同步协议

**优先级：P1**

**整改目标**

正式定义 Client 学习多 Gateway 拓扑所需的同步协议。

建议至少包含：

- Gateway 列表
- 优先级/权重
- 可用状态
- 过期时间
- 版本号

**验收标准**

- Client 能依赖正式协议更新拓扑，而不是靠临时实现

---

## 九、DB 层整改项

### DB-1：从“Gateway 绑定 User”升级为“Gateway 承载多 Session”

**优先级：P0**

**问题**

当前 `gateways` 表上的单个 `user_id` 字段无法表达共享 Gateway。

**整改目标**

把“节点信息”和“会话归属”拆开建模。

**建议新增**

- `gateway_sessions`
- `client_sessions`
- `user_gateway_bindings`（如有长期绑定需求）

**表职责建议**

- `gateways`：只存节点级信息
- `client_sessions`：存客户端/会话级生命周期
- `gateway_sessions`：存某会话当前挂在哪个 Gateway

**验收标准**

- 同一 Gateway 可同时挂多个会话
- 会话断开只影响自身，不污染节点元数据

### DB-2：计费日志必须可回溯到用户与会话

**优先级：P0**

**整改目标**

每条计费流水必须至少可回溯到：

- `user_id`
- `gateway_id`
- `session_id`

必要时还应包含：

- `client_id`
- 时间窗口
- 幂等键

**验收标准**

- 同一 Gateway 上多个用户的账单可独立核对
- 重放上报不会重复扣费

### DB-3：拓扑状态与控制状态分离存储

**优先级：P1**

**整改目标**

数据库需要同时支持两类事实：

- 当前运行事实：节点、会话、在线状态、承载关系
- 控制面期望事实：目标状态、策略版本、拓扑版本

**验收标准**

- 系统重启后能恢复期望状态
- 拓扑漂移与状态漂移都能被追踪

## 十、分阶段推进顺序

为了降低改造风险，建议按下面顺序推进。

### 第一阶段：先修 P0，建立“可正确计费、可正确熔断”的基础

范围：

- Proto：为流量上报补齐用户/会话身份
- DB：拆出会话模型，去掉 Gateway 单用户假设
- Gateway：把配额桶从全局改成隔离桶
- OS：停止基于 Gateway 猜用户

完成标志：

- 同一 Gateway 上 2 个以上客户端并发时，计费和熔断准确

### 第二阶段：补齐 OS 控制平面

范围：

- Gateway 正式注册
- Cell 级 fan-out
- 节点级下推
- 控制状态追踪

完成标志：

- 1 个 OS 能稳定管理 N 个 Gateway

### 第三阶段：补齐 Client 运行时拓扑学习

范围：

- 修复 bootstrap 竞态
- 实现路由表同步
- 实现运行时节点池
- 实现稳定身份会话

完成标志：

- Client 不再只依赖初始 bootstrap 池

### 第四阶段：补齐多节点 e2e 与压测

范围：

- 多 Gateway
- 多 Client
- 混合用户等级
- 共享 Gateway 并发
- Gateway 漂移与切换

完成标志：

- `1 OS + N Gateway + N*M Client` 可以通过持续化验收测试

## 十一、建议新增的验收测试

以下测试建议作为本轮整改的正式验收清单：

### E2E-1：单 OS、多 Gateway、单用户多 Client

验证点：

- 同一用户多个客户端接入不同 Gateway
- 配额累计正确
- 切换后身份连续

### E2E-2：单 OS、单 Gateway、多用户多 Client

验证点：

- 同一 Gateway 承载多个用户
- 配额互不影响
- 账单互不串扰

### E2E-3：单 OS、多 Gateway、多用户混合共享

验证点：

- OS 可对多个 Gateway 做统一控制
- 某个 Gateway 故障不影响其他 Gateway 归属正确性

### E2E-4：Bootstrap 竞态一致性测试

验证点：

- 多 Gateway 并发探测时，客户端最终状态与实际连接一致

### E2E-5：运行时拓扑学习测试

验证点：

- Client 启动时只有少量 bootstrap 地址
- 运行中可学习更多 Gateway
- 原有入口失效后仍能恢复

### E2E-6：按 Cell 下推测试

验证点：

- OS 可向某个 Cell 下全部 Gateway 下推策略
- 指定 Cell 外的 Gateway 不受影响

---

## 十二、最终验收标准

只有当下面 5 条同时成立，才能认为 Mirage 已经真正支持目标拓扑：

1. 一个 Gateway 可以承载多个客户端，而不会发生配额串扰
2. 一个 Gateway 可以承载多个用户，而不会发生计费归属错误
3. 一个 OS 可以准确识别并控制 N 个 Gateway
4. Client 可以学习并使用运行时拓扑，而不只依赖启动池
5. 多节点 e2e 能稳定覆盖 `1 OS + N Gateway + N*M Client`

## 十三、结论

当前 Mirage 的代码基础已经足以支撑“小规模可运行”的验证，但距离目标中的 `1 个 OS、N 个 Gateway、N*M 个 Client` 还有一段明显的架构距离。

问题的本质不是某个函数写错了，而是系统中仍然残留着一些“小规模、单归属、单节点思维”的假设。这些假设在小型测试里不一定暴露，但在共享 Gateway、多客户端并发、OS 统一调度的真实目标架构下会直接失效。

因此，本次整改的重点应当是：

- 先解决归属问题
- 再解决配额和计费问题
- 再补齐 OS 控制面
- 最后补齐 Client 的运行时拓扑能力

只有这样，Mirage 才能从“当前可演示系统”真正进入“目标架构系统”。
