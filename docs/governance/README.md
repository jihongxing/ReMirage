# ReMirage Governance Baseline

本目录是 ReMirage 的新治理基线，用来解决两个持续放大的问题：

1. 项目边界不断扩张，但没有统一收口位置。
2. 同一类事实被多个文档、代码、纪要和清单重复定义，导致"真相源"裂变。

这套框架与现有 `docs/00-05`、零散纪要、整改清单、发布清单并行存在，不直接替换旧文档。旧资料在迁移完成前默认视为输入材料，不自动视为真相源。

## 目标

- 先定义边界，再决定每个边界内的唯一真相源。
- 让"文档真相"和"运行时真相"各归其位，不再互相抢角色。
- 为后续文档迁移提供明确落点，而不是继续平铺新增文档。

## 基本原则

1. 先边界，后设计，最后实现。
2. 一个问题域只能有一个权威入口。
3. 运行时事实可以以代码、协议、配置结构体的形式存在，不强制都写成文档。
4. 会议纪要、整改清单、草案、发布说明默认不是长期真相源，除非其结论已回写到权威位置。
5. 任何跨边界改动，都要先更新边界文档或真相源地图。

## 目录结构

- `boundaries/`
  ReMirage 的边界定义，回答"系统做到哪里、停在哪里、谁负责什么"。

- `source-of-truth-map.md`
  真相源地图，回答"某类事实到底该去哪里看、哪里改、哪里不能当权威"。

- `document-lifecycle.md`
  文档状态规则，回答"草案、输入材料、正式真相源、归档材料如何区分"。

- `migration-backlog.md`
  初版迁移登记，回答"旧文档未来分别要迁往哪里、当前是什么状态"。

- `templates/`
  新建边界卡和真相源卡时使用的模板。

## 使用顺序

1. 新增功能或能力前，先看 `boundaries/`，确认是否越界。
2. 设计接口、数据模型、配置结构前，先查 `source-of-truth-map.md`，确认权威位置。
3. 若当前问题域没有权威位置，先补边界卡或真相源卡，再进入实现。
4. 若旧文档与新框架冲突，以本目录定义的边界和真相源规则为准，再决定迁移方式。

## 当前采纳范围

### 治理骨架（已建立）

- 产品边界、系统边界、组件职责边界、协议域边界、运行时真相边界
- 真相源地图、文档生命周期规则、旧文档迁移登记起点

### 协议域迁移（已完成七轮）

- `docs/protocols/` 协议域新入口，六大协议正式语义入口全部建立
- NPM map 命名统一、B-DNA 模板/画像边界收口、默认画像库外推为版本化 registry
- 旧 `docs/03-自研协议` 整体降级为迁移输入材料

### 外部零特征审计（进行中）

- `docs/外部零特征消除审计与整改清单.md` 作为主审计 spec
- 已识别架构级结构阻断（两套切换体系、原子切换未闭合）
- 已识别数据面主链未闭合状态（client 单一路径 QUIC + gateway 无公网 bearer）

后续迁移原则是：先迁移结论，再迁移描述；先迁移权威入口，再迁移辅助材料。
