# Component Responsibilities Boundary

本文件定义仓库内各核心组件的职责边界，是“谁拥有哪类问题”的权威入口。

## 组件职责总表

| 组件 | 必须负责 | 不应负责 | 主输出 |
|------|----------|----------|--------|
| `mirage-os` | 控制语义、状态汇聚、策略下发、管理接口、编排协同 | 直接承载业务流量、替代客户端本地恢复、替代网关执行承载细节 | 控制面 API、状态模型、策略与管理能力 |
| `mirage-gateway` | 链路接入、承载执行、网关侧策略落地、事件上报 | 定义全局产品规则、长期持有控制面真相、承接无关管理后台职责 | 网关运行时、承载策略、链路事件 |
| `phantom-client` | 终端接入、保活、切换、恢复、最小用户侧运行逻辑 | 控制全局拓扑、定义中心化计费或管理模型 | 客户端接入与恢复能力 |
| `mirage-cli` | 运维命令、诊断、辅助操作、查询入口 | 成为状态真相源、绕过控制面写业务状态 | 命令式操作入口 |
| `sdk` | 对外集成接口、开发者接入面 | 承担运行时编排、复制控制面内部状态 | 集成封装与对外接口 |
| `deploy` | 部署脚本、环境资产、发布辅助 | 定义业务语义、替代正式契约 | 可执行部署资产 |
| `mirage-proto` | 跨组件共享的协议定义 | 保存业务实现逻辑、承担运行时状态 | 协议与消息结构 |
| `docs` | 记录边界、契约、决策、运维知识 | 替代运行时代码真相 | 权威文档与辅助材料 |

## 一类问题只能有一个主责任组件

| 问题域 | 主责任组件 |
|--------|------------|
| 节点与策略管理 | `mirage-os` |
| 链路接入与承载执行 | `mirage-gateway` |
| 终端恢复与接入体验 | `phantom-client` |
| 运维入口与排查动作 | `mirage-cli` |
| 外部系统集成封装 | `sdk` |
| 协议与跨语言消息结构 | `mirage-proto` |
| 部署资产与环境编排 | `deploy` |

## 组件协作原则

1. 组件可以消费别人的契约，但不能私自改写别人的主权事实。
2. 若一个概念需要跨多个组件出现，必须先指定唯一拥有者，再定义同步方式。
3. 同一概念若在多个组件都有本地缓存或投影，缓存不是主真相源。
4. 工具组件只能“操作”或“呈现”真相，不能“拥有”真相。

## 典型越界信号

- `mirage-cli` 开始定义核心业务规则。
- `mirage-gateway` 长期保存控制面应拥有的全局配置语义。
- `phantom-client` 独自演化与控制面不一致的会话状态机。
- 多份部署脚本各自约定不同的配置字段含义。
- 文档、proto、代码对同一字段语义各写一套说法。

## 变更规则

- 新组件加入仓库前，必须先在本文件中定义职责与排除项。
- 组件职责发生变化时，先修改本文件，再修改实现。
- 若多个组件都声称自己拥有同一能力，默认视为边界冲突，暂停继续扩张。
