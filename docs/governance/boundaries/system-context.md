# System Context Boundary

本文件定义 ReMirage 的系统边界，是“系统由哪些平面组成、仓库负责到哪里”的权威入口。

## 系统分层

ReMirage 当前按四个核心平面理解：

| 平面 | 作用 | 主要承载组件 |
|------|------|--------------|
| 控制平面 | 管理节点、聚合状态、下发策略、提供管理接口 | `mirage-os` |
| 网关平面 | 承接连接、执行承载与策略、维持链路工作 | `mirage-gateway` |
| 接入平面 | 让终端或边缘节点接入系统并完成恢复 | `phantom-client` |
| 工具与集成平面 | 提供 CLI、SDK、部署与辅助资产 | `mirage-cli`、`sdk`、`deploy`、`mirage-proto` |

## 仓库边界

本仓库负责：

- ReMirage 核心组件的实现与协同。
- 运行时协议、接口契约、部署资产和运维规则。
- 对外暴露的管理与接入能力。

本仓库不负责：

- 第三方基础设施本身的生命周期管理。
- 未被本仓库协议或接口定义约束的外部系统内部实现。
- 任意业务方在 Mirage 之上的业务逻辑。

## 核心系统关系

### 控制平面

`mirage-os` 负责汇聚管理语义。它可以知道“整体该如何协调”，但不直接承担数据转发职责。

### 网关平面

`mirage-gateway` 负责链路接入、转发、承载策略和网关侧执行。它可以执行控制策略，但不定义全局管理语义。

### 接入平面

`phantom-client` 负责最终接入、保活和恢复。它面向接入体验，不替代控制平面做全局编排。

### 工具与集成平面

`mirage-cli`、`sdk`、`deploy` 和 `mirage-proto` 为系统提供可操作性、可集成性和可部署性，但不拥有主业务状态。

## 允许的跨平面交互

- 控制平面向网关平面下发策略和控制语义。
- 网关平面向控制平面回传状态、事件和观测结果。
- 接入平面通过受控契约接入网关或控制面能力。
- 工具平面只通过明确契约访问其他平面，不绕过契约直接定义状态。

## 不允许的跨平面模式

- CLI 或脚本直接成为业务状态真相源。
- 文档先定义接口、代码另行实现且长期不对齐。
- 网关与客户端绕开控制面私下演化全局状态模型。
- 多个平面各自维护同一概念但没有主权归属。

## 边界检查清单

涉及新需求时，必须先判断：

1. 这个变更落在哪个平面？
2. 它是否引入新的平面间耦合？
3. 它是否让某个平面承担原本不属于自己的状态主权？
4. 它是否需要新增或修改跨平面契约？
