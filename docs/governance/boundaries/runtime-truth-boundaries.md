# Runtime Truth Boundaries

本文件定义“运行时真相”应该落在哪些地方，是“代码、协议、配置谁说了算”的权威入口。

## 核心原则

1. 运行时真相优先落在可执行资产中。
2. 文档负责声明归属、语义和约束，不替代运行时实现本身。
3. 每一类运行时事实都要有唯一主源，其他地方只能引用、适配或投影。

## 运行时真相划分

| 事实类型 | 主真相源 | 说明 | 非真相源示例 |
|----------|----------|------|--------------|
| Mirage OS 运行时数据库模型 | `mirage-os/pkg/models/*.go` | 以模型定义为运行时数据库主源；治理决策由 `docs/adr/001-database-truth.md` 说明 | 会议纪要、Prisma 注释、零散 SQL 片段 |
| 跨组件 protobuf/gRPC 协议 | `mirage-proto/*.proto` | `.proto` 文件定义消息结构和字段语义 | 生成代码、Wiki 摘要 |
| HTTP 契约 | `docs/api/*.md` 中按接口面拆分的合同文档 | 文档定义外部接口语义与约束，代码实现对齐文档 | README 摘要、测试描述 |
| Gateway 配置结构 | `mirage-gateway/cmd/gateway/main.go` 中的配置结构体 | 结构体定义字段主权，样例 YAML 为派生 | 多份部署样例、口头约定 |
| 部署可执行资产 | `deploy/` 与各组件 `deployments/` 下的实际资产 | 以要执行的脚本、compose、manifest 为准 | 介绍性文档、历史清单 |
| 发布检查结论 | 发布对应的 traceability index | 发布结论只对对应发布周期有效，不自动成为长期真相 | 历史整改清单、旧勾选状态 |

## 文档与运行时的关系

- 文档可以声明“某类真相源位于代码路径”，但不能复制一份平行定义长期存在。
- 若文档必须描述运行时结构，应明确标注其为解释性材料，而不是主源。
- 任何运行时结构变更，都应先改主真相源，再回写解释性文档。

## 当前约束

### 数据库

Mirage OS 的数据库真相已通过 `ADR-001` 固化，后续涉及数据库字段、主键、表语义的变更，必须以 `mirage-os/pkg/models/*.go` 为起点。

### 协议

跨组件共享消息结构，必须从 `mirage-proto/*.proto` 发起变更。生成代码和调用方实现都属于派生物。

### 配置

Gateway 配置字段的结构主权归 `GatewayConfig`。`configs/gateway.yaml` 等样例文件用于演示和默认值，不拥有字段定义权。

### 部署

部署脚本、compose 和 manifest 以实际要执行的文件为准。说明文档只能解释这些资产，不应形成第二套部署规则。

## 何时需要新增真相源

只有当以下条件同时满足时，才应增加新的主真相源类别：

1. 出现了新的事实类型，无法归入现有数据库、协议、接口、配置、部署或发布结论。
2. 新类别能被明确限定边界，不会与现有主真相源重叠。
3. 新类别能提供稳定入口，而不是临时项目文档。
