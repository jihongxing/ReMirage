# Product Scope Boundary

本文件定义 ReMirage 的产品边界，是“项目做什么、不做什么”的权威入口。

## 产品定义

ReMirage 是一套面向复杂网络环境的托管式连接基础设施。它关注的是连接的长期稳定性、可恢复性、统一管理能力，以及在受限环境下的持续可用性。

它不是单一代理程序，也不是只解决“加密转发”的轻量工具，而是由控制面、网关面、客户端接入面和配套工具链共同构成的系统。

在正式产品定义上，ReMirage 的目标形态是：

- `Hyper-Resilient Overlay Network`：强调多承载编排、恢复、自愈与业务连续性
- `Zero-Trust SD-WAN 隐蔽数据链路`：强调受控接入、策略治理与低暴露传输

“影子网络”可以作为品牌化叙事标签使用，但不能替代上述正式定义词，也不能单独作为验收名词。

## 目标用户

- 需要跨区域、跨网络环境维持稳定连接的团队。
- 需要统一管理多个节点、多个客户端和多种接入策略的运维或平台团队。
- 对隐私、恢复能力、最小运行痕迹较敏感的部署场景。
- 对业务连续性、链路生存性、低暴露接入有强约束的高价值 B2B 场景。

## 明确纳入边界的能力

| 能力域 | 纳入范围 |
|--------|----------|
| 连接基础设施 | 连接建立、接入维护、多路径调度、失败恢复、状态协调 |
| 控制平面 | 节点注册、策略下发、状态聚合、配置编排、控制接口 |
| 终端接入 | 客户端接入、保活、切换、恢复、拨测与基本自愈 |
| 运维工具链 | CLI、部署资产、运行检查、发布验证、必要的 Sidecar / SDK 接入面 |
| 协议与承载 | 自研协议、承载层策略、控制与数据协同所需契约 |
| 受控交付形态 | 托管式 Gateway、Sidecar 接入、受控 SDK/Lib 集成 |

## 明确排除在边界之外的事项

| 非目标域 | 说明 |
|----------|------|
| 通用业务平台 | ReMirage 不承载任意业务域流程、订单系统、CRM 或通用 SaaS 后台 |
| 无边界的安全产品化 | 安全能力仅服务于连接基础设施本身，不单独扩展为泛安全平台 |
| 任意终端运营能力 | 不负责与连接基础设施无关的终端管理、MDM、桌面运维 |
| 模糊的“万能力网络底座” | 任何新能力若不能明确归入控制、网关、接入、协议、运维之一，则默认不纳入 |
| 开放式 B2B2B 底层能力平台 | 不做任何第三方均可任意嵌入、转售、二次封装的开放网络底座 |
| 无治理约束的白标/OEM 输出 | 不提供脱离 Mirage 运行、审计和策略治理体系的纯贴牌交付 |

## 产品判断标准

一个需求只有同时满足以下条件，才应进入 ReMirage 主干：

1. 它直接提升连接稳定性、恢复性、可管理性或最小暴露能力。
2. 它能被归入已有系统平面或组件职责，而不是新开一块模糊能力域。
3. 它的运行时真相能够落到现有真相源体系中，而不是新增一份平行 spec。

## 需求进入前必须回答的问题

1. 这个需求属于哪一个能力域？
2. 它改变的是产品边界、系统边界，还是已有边界内的实现？
3. 它的权威定义将落在哪个真相源？
4. 它会不会让某个组件承担双重身份？

## 变更规则

- 改变“ReMirage 是什么”的需求，必须先修改本文件。
- 新增一级能力域，必须先在本文件中定义纳入理由和排除项。
- 若需求无法通过本文件判断归属，默认暂停实现，先补边界定义。

## 商业模型与服务分层

### 运营定位

Mirage 定位为服务极少量高价值客户的影子运营商与受控网络基础设施提供方。目标客户规模约 200 个高价值客户，不追求大众市场、公开扩张或开放平台生态。

它的目标不是消费级“翻墙工具”，而是在受限网络环境下，为高价值业务提供可恢复、可编排、低暴露的数据链路能力。

### 明确纳入的商业方向

| 方向 | 纳入理由 |
|------|----------|
| 高价值 B2B 业务连续性链路 | 与连接稳定性、恢复性、最小暴露能力直接一致 |
| 托管式覆盖网交付 | 保持 Mirage 对策略、审计、发布和恢复链路的控制力 |
| 受控集成能力输出 | 允许以 Sidecar 或受控 SDK/Lib 形式进入客户系统，但仍受 Mirage 契约和治理约束 |

### 明确否决的商业方向

| 方向 | 否决理由 |
|------|----------|
| 低价消费级 VPN 市场 | 与高价值/小规模/强控制/低暴露模式不一致 |
| 开放式 B2B2B 底层能力平台 | 显著扩大暴露面，迫使长期平台化投入，并削弱治理能力 |
| 无治理约束的白标/OEM 能力输出 | 稀释核心优势（小规模、高控制、低可见度） |

### 核心场景锚点

以下场景可以作为产品演进方向，但不自动等于当前已交付能力：

1. 跨境高价值指令与资产调度链路
2. 高价值节点与关键基础设施的抗失联/抗阻断链路
3. 特殊网络环境下的企业级业务连续性链路

这些场景的商业叙事、行业映射与交付包装，应作为派生材料维护；是否已经具备对应能力，仍以能力真相源为准。

### 服务分层模型

采用三层收费结构：

1. **接入等级费**：用户购买服务等级，决定资源池和服务能力
2. **流量配额费**：用户购买流量包，决定资源消耗规模
3. **增值服务费**：高价值客户附加收费项（独享 Gateway、固定区域、高级支持）

### 三档服务等级

| 等级 | 定位 | 资源池 |
|------|------|--------|
| Standard | 专业级入门客户 | 标准资源池 |
| Platinum | 主力盈利档位，对稳定性/恢复速度敏感的商业客户 | 高优先级资源池 |
| Diamond | 高价值/高隔离/高保障客户 | 最高等级资源池 |

### 定价锚点

Mirage 的价格锚点不是消费级 VPN，而是高价值网络基础设施服务（轻量化跨境专线、带运维能力的高可用网络服务）。

### 运行时映射

- `users.cell_level` = 服务等级主字段（1=standard, 2=platinum, 3=diamond）
- `cells.cell_level` = 资源池等级
- `remaining_quota` = 流量使用额度
- `QuotaPurchase` = 购买记录承载点

运行时计费结构以 `mirage-os/pkg/models/*.go` 为主源。

## 运营原则

1. 收敛优先于扩张 — 新设计先问是否扩大暴露面
2. 控制力优先于规模 — 价值来自控制力而非用户数
3. 运营隐蔽性优先于技术展示 — 更低暴露、更少入口
4. 小团队可持续优先于大平台想象力 — 极简操作/部署/恢复
5. 用户无感优先于复杂可玩性 — 后台稳定工作，客户不用操心
