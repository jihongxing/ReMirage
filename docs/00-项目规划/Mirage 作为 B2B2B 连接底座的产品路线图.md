# Mirage 作为 B2B2B 连接底座的产品路线图

## 一、文档目标

本文档用于回答一个核心问题：

**Mirage 是否可以从“直接面向终端用户的连接产品”，升级为“为其他连接服务商、VPN 品牌、托管网络服务商提供底层能力的 B2B2B 连接底座”？**

结论先说在前面：

**可以，而且这条路比消费级 VPN 更符合 Mirage 当前的技术结构与长期商业逻辑。**

但这条路也有一个前提：

Mirage 必须从“项目能力集合”升级为“合作伙伴可接入、可运营、可计费、可交付”的标准化平台产品。

---

## 二、Mirage 在这条路线里的新定位

Mirage 不再把自己定义为：

- 一个直接卖给终端用户的 VPN 品牌
- 一个追求大规模低价订阅的消费级产品
- 一个单纯强调协议隐匿能力的技术项目

而是定义为：

**高韧性连接底座（Resilient Connectivity Substrate）**

它服务的对象不是普通用户，而是：

- 需要自己对外提供连接服务的公司
- 需要更稳定底层传输能力的服务商
- 需要白标或 OEM 网络底座的合作方
- 需要在复杂网络环境下提供差异化连接服务的运营团队

换句话说：

上游合作方卖的是“他们的品牌、他们的服务、他们的客户关系”，  
Mirage 卖的是“让他们不用重做底层网络”的能力。

---

## 三、为什么这条路对 Mirage 成立

### 3.1 Mirage 的技术形态更像“底座”，而不是“单一成品”

从当前项目结构看，Mirage 已经天然具备平台底座的一些雏形：

- `Mirage-OS` 负责控制、调度、配额、策略和节点管理
- `Mirage-Gateway` 负责边缘承载、策略执行、威胁感知和流量处理
- `phantom-client` 负责接入、切换、自愈和用户侧保持连接

这套结构本质上不是“一个简单客户端 + 一个出口节点”，而是：

**控制平面 + 承载平面 + 接入侧终端** 的完整骨架。

这正是做 B2B2B 平台的基础条件。

### 3.2 Mirage 的价值点，更适合卖给“服务商”而不是“终端用户”

Mirage 真正强的地方，不是“让普通人便宜上网”，而是：

- 在复杂环境下保持连通
- 在多 Gateway 环境下做切换和恢复
- 具备更强的资源分层和控制能力
- 适合做高价值连接服务的底层承载

这些价值点，终端用户往往不会完整理解，也未必愿意为之付高价。

但服务商、托管运营商、白标品牌方，会更容易理解这些能力的商业价值。

### 3.3 行业里已经存在相邻成功模式

Mirage 不是第一个尝试“上游能力平台化”的项目。

公开资料显示，行业里已经存在三类成熟相邻模式：

1. **面向服务商的 SD-WAN / SASE 底座**
   Cisco 明确提供 `Managed SD-WAN for Service Providers`，并强调 white-label managed services、Day 1/Day 2 orchestration、usage-based billing 等能力。

2. **服务商伙伴生态型平台**
   VMware 官方公开材料显示，其 SD-WAN / SASE 方案不仅有全球 PoP 和 Gateway 体系，还由 `285+` 服务商提供托管或转售服务。

3. **白标 VPN / Reseller 底座**
   这类产品把底层节点、计费、自动开通、品牌替换统一打包，让合作方只负责销售和品牌。

这说明：

**“底层连接能力平台化，合作方在上层做品牌和客户运营”这件事，本身是已经被市场验证过的。**

## 四、Mirage 的目标客户是谁

如果 Mirage 走 B2B2B 路线，潜在客户不应该再泛化，而应该精准聚焦。

建议优先锁定以下四类客户。

### 4.1 Boutique VPN / 专业连接品牌方

这类客户通常具备：

- 有品牌、有渠道、有用户
- 但不想自己长期维护复杂底层网络
- 希望更快推出“更稳、更高端、更可恢复”的产品线

对他们来说，Mirage 的价值是：

- 提供更好的底层连通性
- 提供更快的节点切换和恢复
- 提供更高质量的高端档位产品底座

### 4.2 MSP / MSSP / 托管网络服务商

这类客户通常不是卖“VPN”，而是卖：

- 托管网络服务
- 远程接入服务
- 安全接入服务
- 轻量化专线或专业连接服务

对他们来说，Mirage 的价值是：

- 可以更快上线差异化连接服务
- 可以减少自研承载层和控制面的成本
- 可以用更小团队做出更高层级的网络产品

### 4.3 行业解决方案商

这类客户往往服务垂直行业，例如：

- 跨境协同团队
- 分布式研究与媒体团队
- 对稳定连接要求高的小型专业组织
- 需要特殊网络环境适配能力的服务商

对他们来说，Mirage 的价值是：

- 不需要自己理解所有底层网络细节
- 但可以把稳定和恢复能力打包成自己的方案能力

### 4.4 区域型或主权型私有接入运营方

这类客户不是纯粹白标转售，而是更看重：

- 数据和节点控制权
- 私有部署
- 自有资源池
- 更强的隔离和定制化

这类客户客单价更高，但交付和合规要求也更高，建议放在第二阶段之后再重点承接。

---

## 五、Mirage 不应该先打的客户

为了避免路线偏移，以下客户群不建议作为第一阶段重点。

### 5.1 低价白牌分销商

这类客户的核心诉求通常是：

- 更便宜
- 更快起量
- 更低门槛转售

他们会把 Mirage 拉回低价 VPN 逻辑，不符合当前项目方向。

### 5.2 期待“一站式完整 SASE 套件”的大型企业

Mirage 当前更强的是连接底座与韧性承载，而不是完整的 SSE / CASB / DLP / SWG 全家桶。

如果太早去承接这类客户，会把产品边界拉得过宽。

### 5.3 超高合规行业的第一批头部客户

例如极重监管、极重认证、极重本地化审计的客户。

这些客户未来可以做，但不应该作为 Mirage B2B2B 路线的第一站。

## 六、Mirage 作为连接底座，需要什么能力

这部分不是“全部重做”，而是“把现有能力平台化、租户化、合作方化”。

建议按五层能力来建设。

### 6.1 多租户控制平面

这是 B2B2B 路线的第一基础。

Mirage 必须从“我在运营一个系统”，升级为“我在运营多个合作方的独立服务空间”。

至少需要补齐：

- Partner / Tenant 模型
- 每个租户独立的用户、Gateway、Client、配额、策略、日志视图
- 租户级资源池隔离
- 租户级服务等级与计费策略
- 租户级审计与权限边界

没有这一层，就只能算“给多个客户共用一个后台”，还不能算真正的平台。

### 6.2 Partner API 与控制台

合作方不会接受“找你人工开账号、人工加节点、人工查流量”的模式。

必须提供：

- 创建租户
- 创建下游用户或分组
- 发放接入凭据
- 配额和套餐管理
- 状态查询
- 用量查询
- 告警与事件订阅
- 基础自动化能力

除了 API，还需要一套面向合作方的简洁控制台。

### 6.3 白标与 OEM 能力

这是 B2B2B 和普通平台的关键区别。

合作方至少要能替换：

- 品牌名称
- 控制台标识
- 客户端品牌层
- 域名和接入入口
- 套餐说明与计费口径

如果 Mirage 只能提供裸能力，而不能被合作方包装成自己的服务，它的商业吸引力会明显下降。

### 6.4 服务分层与计费托管

Mirage 既然要做底座，就不能只统计流量。

它需要支持合作方经营自己的产品层级。

至少要支持：

- 接入等级
- 流量配额
- 独享 / 共享资源池
- SLA 等级
- 增值能力开关
- 租户级账单与报表

这一层与 Mirage 现有的 `cell_level`、`remaining_quota`、配额购买、节点分层思路是相通的，但需要正式产品化。

### 6.5 可观测性、SLA 与运维协作

B2B2B 客户买的不是“黑盒能力”，而是“可运营能力”。

必须给合作方看到最小可用的运行真相：

- 当前节点健康
- 当前服务等级
- 当前告警
- 当前连通情况
- 当前配额消耗
- 当前切换和恢复事件

同时，Mirage 自己也必须建立：

- NOC 视角
- Partner 视角
- 租户终端视角

否则平台无法支撑正式商业交付。

## 七、Mirage 的产品包装建议

为了避免产品说不清，建议 Mirage 不要一开始只卖“一个平台账号”，而是包装成三层产品。

### 7.1 Mirage Fabric

这是底层连接承载层。

面向合作方表达为：

- 节点池
- 资源池
- 多 Gateway 承载
- 切换与恢复能力
- 分层接入能力

### 7.2 Mirage Partner Console

这是合作方运营层。

面向合作方表达为：

- 开户
- 用户管理
- 套餐管理
- 配额管理
- 状态与告警
- 报表

### 7.3 Mirage OEM Access

这是合作方对外交付层。

面向合作方表达为：

- 白标客户端
- 白标入口
- API 发放接入材料
- 租户定制策略

这三层组合起来，Mirage 才会从“一个很强的系统”变成“一个可卖的 B2B2B 产品”。

---

## 八、建议商业模式

Mirage 如果做 B2B2B，不建议只收流量费。

更合理的结构是：

### 8.1 平台接入费

合作方为使用控制平面、租户能力、运营能力支付基础费用。

### 8.2 资源与服务等级费

按资源池等级、节点等级、SLA 等级收费。

### 8.3 用量费

按流量、并发、活跃终端数或 Gateway 资源消耗收费。

### 8.4 高价值增值费

例如：

- 独享 Gateway
- 专属区域资源池
- 私有部署
- 定制恢复策略
- 专属支持与更高 SLA

这套模型更符合 B2B2B 平台，而不是消费订阅。

## 九、与同行业方案的对比

这里需要先说明：

Mirage 当前没有一个“完全同形态”的直接对手。

更准确地说，它处在几个相邻赛道的交叉点：

- 面向服务商的 SD-WAN / SASE 平台
- 白标连接服务底座
- 高韧性传输与恢复底层

因此，对比应该按“相邻方案”来看。

### 9.1 与 Cisco Managed SD-WAN for Service Providers 对比

Cisco 的公开材料明确强调：

- white-label managed services
- 端到端服务保障
- Day 1 / Day 2 编排能力
- usage-based billing
- 可由 Cisco 代托管平台能力

Mirage 与它的关系不是“谁更大”，而是“定位是否重叠”。

**Cisco 的强项**

- 品牌和渠道能力极强
- 企业和服务商信任基础更强
- 运营体系成熟
- 更适合标准化托管网络市场

**Mirage 的机会**

- 更适合做高韧性、复杂环境、恢复导向的连接底座
- 可以更灵活地服务中小型合作方
- 可以比大型厂商更专注于“连得上、切得快、活得久”这一类差异化场景

**Mirage 的短板**

- 平台化、运维化、合作方产品化能力还不成熟
- 没有 Cisco 级别的服务商生态和品牌背书

### 9.2 与 VMware SD-WAN / SASE 服务商生态对比

VMware 官方资料显示：

- 其方案有全球 PoP / Gateway 体系
- 支持多租户、可扩展承载
- 由 `285+` 服务商提供托管或转售
- 已经形成成熟的服务商生态

**VMware 的强项**

- 生态成熟
- 全球部署能力强
- 服务商伙伴多
- 更适合大范围标准化铺开

**Mirage 的机会**

- Mirage 不需要一开始就做成“大而全的全球通用平台”
- 可以先做成“高价值连接场景的专用底座”
- 在恢复、自愈、复杂网络环境适应能力上形成更强差异

**Mirage 的短板**

- 节点规模、伙伴规模、全球覆盖都还远未达到同级别
- 缺少成熟的合作伙伴交付体系

### 9.3 与 Fortinet 面向服务商的 SASE / Sovereign SASE 对比

Fortinet 公开材料已经把方向说得很清楚：

- 支持 service providers、large organizations、governments 构建自己的私有 SASE
- 强调更快上线、可定制、可控架构
- 强调主权、数据控制、合规和可包装能力

**Fortinet 的强项**

- 完整安全栈能力更强
- 更适合强合规、强安全、强私有化交付场景
- 更容易进入大型客户正式采购体系

**Mirage 的机会**

- Mirage 不必和它正面竞争“全套 SASE”
- 更适合聚焦在“连接底座”而不是“全安全平台”
- 对于需要高恢复、高灵活、高差异化调度的合作方，Mirage 可以更轻、更快、更聚焦

**Mirage 的短板**

- 不是完整安全栈
- 缺少主权部署、合规交付、强审计产品化能力

### 9.4 与白标 VPN Reseller 平台对比

白标 VPN 平台通常强调：

- 快速上线
- 自动开通
- 自动计费
- 品牌替换
- 节点复用

这类方案的优点是快，但通常定位更偏消费级或轻量转售。

**这类平台的强项**

- 上线快
- 门槛低
- 更适合小代理和轻量分销

**Mirage 的机会**

- 不必跟它们拼“5 分钟开店”
- Mirage 可以站在更高一层，服务那些需要真正底层能力的合作方
- Mirage 可以成为“高端版、韧性版、专业版”的底座

**Mirage 的短板**

- 启动门槛更高
- 产品包装和自动化程度目前还不够

## 十、Mirage 的差异化主张

如果要用一句话概括 Mirage 在这个市场里的位置，建议是：

**Mirage 不是另一个白标 VPN 平台，而是一套为高价值连接服务商准备的高韧性连接底座。**

它真正应该强调的差异，不是“更多功能”，而是：

### 10.1 更强的恢复能力

在节点抖动、链路波动、环境变化下，Mirage 的目标不是“静态可用”，而是“动态可恢复”。

### 10.2 更强的分层资源管理能力

Mirage 适合承载：

- 共享资源池
- 高等级资源池
- 独享承载
- 分租户差异化资源分配

### 10.3 更适合复杂网络环境的连接承载

Mirage 的卖点不是“普通上网”，而是“在难环境里仍然可控、可切换、可继续服务”。

### 10.4 更适合做合作方差异化产品

合作方可以基于 Mirage 做出：

- 标准版连接服务
- 高稳定版连接服务
- 高隔离版连接服务
- 私有部署版连接服务

这比“卖同质化节点”更有价值。

## 十一、推荐路线图

建议按三个阶段推进，而不是一步到位。

### Phase 1：把 Mirage 变成“可卖给第一个合作方”的产品

目标：

- 不再只是内部系统
- 具备最小多租户能力
- 具备最小 Partner API
- 具备最小合作方控制台
- 具备最小白标能力

这一阶段必须优先完成的能力：

- Tenant 数据模型
- Partner 账户体系
- 配额、等级、账单按租户隔离
- 合作方可发放接入凭据
- 合作方可看状态和用量
- 客户端最小白标能力

这是 Mirage 从项目走向产品的第一道门槛。

### Phase 2：把 Mirage 变成“合作方能长期运营”的平台

目标：

- 合作方能真正用它经营业务
- Mirage 能支持多合作方并行运营
- 基本的 SLA、运维和事件闭环建立起来

这一阶段的重点：

- 多租户资源池隔离
- Partner 级告警与报表
- 节点、Gateway、Client 的运行态可观测
- 服务等级正式生效
- 计费和配额托管闭环
- 路由、切换、自愈行为形成可解释事件流

这一阶段完成后，Mirage 才具备“平台型收入”的基础。

### Phase 3：把 Mirage 变成“高价值合作方可定制采购”的底座

目标：

- 能承接更高客单价客户
- 能支持私有化、独享资源、主权部署等更高阶场景

这一阶段的重点：

- Dedicated Gateway / Dedicated Cell
- 更强白标与 OEM 客制化
- 私有部署能力
- 更高等级的审计、SLA、支持
- 合作方分层生态

这一阶段才适合往“高价值 B2B2B 平台”方向做大。

## 十二、从当前代码出发，最该先补的东西

结合当前仓库状态，如果 Mirage 要走这条路，最先应该补的不是“再多一个协议”，而是下面这些产品化短板：

### 12.1 多租户真实隔离

这是第一优先级。

没有 Tenant 模型，Mirage 永远只能算一个系统，不算平台。

### 12.2 OS 到 Gateway 的可靠控制面

如果控制面不能稳定地下推到多 Gateway、多租户、多资源池，那么上层合作方平台就立不住。

### 12.3 Client 的无感化与托管化

合作方不会接受一个“还像开发工具”的客户端。

Client 必须逐步走向：

- 后台服务化
- 无感运行
- 订阅托管
- 稳定切换
- 路由与连接一致

### 12.4 Partner API 和最小控制台

没有 Partner API，就没有 B2B2B。

### 12.5 计费、配额、服务等级的租户化

Mirage 现有的等级和配额思路是可复用的，但必须从“系统内部能力”升级为“合作方可运营能力”。

## 十三、最终判断

如果 Mirage 继续走“直接卖给终端用户”的路，它会越来越像一个高门槛、高技术、高风险的产品。

但如果 Mirage 改走 B2B2B 连接底座路线，它会更像：

- 一个上游基础设施平台
- 一个服务商能力中台
- 一个可以被合作方打包成自己品牌的连接引擎

这条路会更窄，但也更合理。

它的核心价值不在“用户数”，而在：

- 单客价值
- 收入质量
- 平台复用率
- 合作方长期黏性

所以更准确的判断不是：

**Mirage 能不能成为 VPN 的 VPN。**

而是：

**Mirage 完全有机会成为“专业连接服务商的底层网络引擎”。**

这比“另一个 VPN 品牌”更接近它真正应该去的位置。
