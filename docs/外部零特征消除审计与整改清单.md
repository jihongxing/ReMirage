# Mirage 外部零特征消除 — 审计与整改清单

> **总纲**：当前阻断点已经从"运行时主链完全未接入"收敛为三个更具体的问题：Client 默认 QUIC 主路径仍非浏览器级外观、Gateway 仍缺少真实公网 QUIC/H3 bearer listener、Orchestrator 原子切换事务仍未闭合。在此之前，零外部特征目标不能判定为进入可上线整改收尾阶段。
>
> **现状定性**：代码侧主链已经**部分收敛**。Client 已引入 `ClientOrchestrator`，Gateway 主入口已接入 `ChameleonListener + Orchestrator.StartPassive + DataPlane Injector + StealthControlPlane.ReceiveLoop`。但"浏览器级默认主路径 + 真实公网 H3 listener + 原子切换事务"三件事仍未完成，因此当前仍处于**主链未正式收口**阶段，而不是收尾调优阶段。
>
> 审计范围：严格限定为**外部网络观察者**（被动流量分析 + 主动探测）能识别的强特征。
> 本机内核取证、日志泄露、eBPF Map 内部值等主机安全硬化项不在本文范围内，另见附录 C。
>
> **审计结论（2026-04-23 代码复核）**：**不通过**。但本文件上一版中的部分条目已经被代码关闭，不能继续按旧文字直接推进；本版开始按 `已关闭 / 部分成立 / 仍开放` 三种状态维护。

## 术语约定

- **被动观察者**：只能抓取链路上的 pcap，不能主动发包。能力：包长序列、IAT 分布、TLS/QUIC 握手明文字段、DNS 查询、流量突变。
- **主动探测者**：可以向目标 IP/端口发送探测包。能力：握手行为差异、回包特征、端口扫描响应、协议协商异常。
- **外部强特征**：被动或主动观察者通过单次或短期统计即可判定"此流量非正常浏览器流量"的特征。
- **bearer listener**：承载真实用户数据面流量的网络监听器（区别于 gRPC 控制面监听器）。
- **bearer path**：从进程 socket → 网卡 ifindex → hook 点（ingress/egress/clsact/XDP）的完整数据面路径。

## 整改主线收敛建议

在进入逐项审计之前，先明确三条整改主线的优先顺序。当前代码的运行时真相已经做过一轮收敛，但仍存在"主链已接线、外观未达标、切换未原子化"的断层，因此后续整改必须继续围绕唯一主链推进，而不是回到并行体系。

### 主线一：统一运行时真相 — 只保留一套编排主链

收敛 `phantom-client → Orchestrator → 5 bearers` 为唯一运行路径，不再并行维护 TransportManager 和 Orchestrator 两套切换体系（详见 S-01）。

### 主线二：打实默认主路径

Client 源头浏览器级 QUIC 外观 + Gateway 真实 QUIC/H3 listener + 受控抓包验收。这条没闭合之前，后面的冷备协议都只是"存在组件"。

### 主线三：冷备与原子切换

把 WSS/WebRTC/DNS/ICMP 逐个接入同一编排链，并把 dual-send/CID rotation/epoch/FEC MTU 接到同一个切换事务里。

## Layer S：架构级结构阻断

不属于单一特征点，而是影响所有后续修复的结构性问题。

### [S-01] 运行时主链已基本收敛，旧问题描述已过时 ✅ Closed / Stale

**位置**：`mirage-gateway/pkg/gtunnel/orchestrator.go` vs `mirage-gateway/pkg/gtunnel/transport.go` + `chameleon_client.go`

本条在本轮代码复核中已不再成立为 P0 阻断项：

1. Gateway 主启动链已接入 `ChameleonListener → Orchestrator.StartPassive() → DataPlane Injector → StealthControlPlane.ReceiveLoop`
2. `TransportManager` 和 `ConnectWithFallback/probeAndPromote/promoteToQUIC` 已被标注为 `Deprecated`
3. Client 侧 `ProbeAndConnect()` 已构建 `ClientOrchestrator`，`Send()/Receive()` 优先走注入的统一 `Transport`

**当前结论**：

- 运行时真相已基本收敛到 Orchestrator 体系
- 遗留代码仍在仓库中，但当前未发现其重新进入生产主链
- 后续动作不再是"先收敛主链"，而是继续补 `S-02 / A-02 / B-01b`

**后续要求**：
1. 保持 `TransportManager` 不再回流到生产入口
2. 后续有条件时，将 deprecated 路径移入归档或删除，避免未来回潮

### [S-02] Orchestrator 的"原子切换"还没有真正实现，只是状态切换 🔴 P0

**位置**：`mirage-gateway/pkg/gtunnel/orchestrator.go`

`Orchestrator.Send`（line 236）和 `Orchestrator.Recv`（line 248）都只绑定当前 `activePath`。`demote`（line 538）/ `promote`（line 578）的本质是切 `activePath` 并递增 `epoch`（line 199）/ `nextEpoch`（line 204）。

更关键的是，`DualSendDuration` 虽然在 `orchestrator.go:92` 和 `orchestrator.go:115` 定义了，但切换逻辑里并没有调用双发选收；真正的双发能力在 `multipath.go:417`（`MultiPathBuffer.EnableDualSend`）一套单独实现里，没被接进 Orchestrator。

所以现在离"原子切换"还差一条完整运行链：demote/promote 时应启动 DualSendMode → 双发选收 → epoch barrier → CID rotation → FEC MTU 通知，这些步骤当前是分散的独立组件。

**整改要求**：
1. demote/promote 流程中接入 `MultiPathBuffer.EnableDualSend`
2. 切换事务包含：双发选收 → epoch barrier → CID rotation → FEC MTU 调整
3. 切换期间 Send/Recv 走双发路径，切换完成后收敛到新 activePath

被动观察者只看链路 pcap 即可识别的特征。这是最高优先级，因为无需任何交互即可大规模筛查。

1. `ProbeAndConnect()` 已创建 `ClientOrchestrator`

### [A-02] Client 默认 QUIC 指纹仍然是强特征，外部 0 特征目标在源头就已经失守 🔴 P0

### [A-03] Client 侧不存在 eBPF 数据面子系统 — 用户态源头消除特征为主修复路径 🔴 P0

**位置**：`phantom-client/` 整个目录

phantom-client 仓库中没有任何 `bpf/`、`ebpf/` 目录，没有 `.c`、`.o` 文件，没有 cilium/ebpf 依赖。Client 只有用户态 QUICEngine 主路径 + 用户态 SessionShaper（`session_shaper.go:29`，仅做批量窗口时序去相关，不是可控 padding/IAT 控制器）。

这意味着设计文档中 NPM（XDP 层 Padding）、B-DNA（TC 层 TCP/QUIC 指纹重写）、Jitter-Lite（TC 层 skb->tstamp 控制）、VPC（TC 层噪声注入）在 Client 侧根本不存在运行时。

**关键约束**：结合"最终用户只装一个 client"的目标，终端设备通常没有 root/CAP_BPF。不能把 eBPF 当默认前提。

**被动观察者视角**：
- Client 出站流量没有 NPM Padding → 包长序列（SPL）暴露真实载荷大小
- Client 出站流量没有 Jitter-Lite → IAT 分布暴露真实发送模式
- Client 出站 TCP/QUIC 握手没有 B-DNA 重写 → JA4/JA4Q 指纹暴露 Go 原生栈

**整改要求 — 两阶段实施，用户态方案前置**：

**第一阶段（主修复路径）：Client 源头用户态消除特征**
1. **uTLS/QUIC 参数拟态**：使用 uTLS 从源头生成正确 TLS 指纹，QUIC Transport Parameters 对齐 Chrome
2. **用户态 Padding**：在 QUIC Datagram 发送前（`client.go:307` `SendDatagram` 之前）填充，实现包长分布拟态
3. **用户态 IAT 控制**：将 `session_shaper.go` 从当前的"批量窗口发送"扩展为真正的 send-path controller，支持可控 IAT 分布注入（正态/指数分布）

按现有代码，第一阶段最应该落点的是 `session_shaper.go` 和 `client.go`，把现在的"窗口批量发送"扩展成真正的 send-path controller。

**第二阶段（仅限有权限的 Linux 场景）：Client 侧 eBPF**
1. 仅在企业/自管 Linux 场景（有 root/CAP_BPF）启用 client-side eBPF
2. 构建 Client 侧 eBPF 加载器，挂载 NPM/Jitter-Lite TC 程序
3. 此阶段为增强项，不是上线阻断项

### [A-04] 协同链尚未形成可验收闭环，但子结论需按现状改写 🔴 P0

**位置**：全局

设计文档（协议协同矩阵 §3.2）定义的标准数据流：
```
用户请求 → [B-DNA] → [G-Tunnel] → [NPM] → [Jitter-Lite] → [VPC] → 多路径发送
```

当前 Client 侧：虽然已接入 `ClientOrchestrator`，但源头用户态的 `padding / IAT / 指纹拟态` 仍未进入发送链，`Send()` 仍是加密后直接 `transport.SendDatagram()`。

当前 Gateway 侧：`Orchestrator` 已启动为被动模式，`ChameleonListener` 与 `DataPlane Injector` 也已接线；但默认公网 QUIC/H3 bearer listener 仍缺失，且 NPM/B-DNA/Jitter-Lite/VPC 是否挂载在正确 bearer path 上仍未完成验收。

**整改要求**：
1. Gateway 侧：补齐真实公网 QUIC/H3 listener，并把默认主路径的 bearer path 固化到可验收的进程/socket/ifindex/hook
2. Client 侧：按 `A-03` 第一阶段实现用户态 `padding / IAT / 指纹拟态` 发送链
3. 端到端验证：抓包确认默认主路径真正经过 `B-DNA → G-Tunnel → NPM → Jitter-Lite → VPC`
4. 文档口径统一：后续不再使用"Gateway 侧 Orchestrator 未启动"作为本条子判断

### [A-05] WSS 降级路径未使用 uTLS，JA3/JA4 暴露 Go 原生栈 🟠 P1

**位置**：`mirage-gateway/pkg/gtunnel/chameleon_client.go:88`

`DialChameleon()` 使用 `websocket.Dialer`（line 109）+ Go 原生 `tls.Config`。`dialWithUTLS()`（line 295）已实现但未接入实际拨号路径。

**被动观察者视角**：WSS 握手的 TLS ClientHello 是 Go 原生栈指纹（Cipher Suites 顺序、Extensions 列表与 Chrome 完全不同），JA3/JA4 一眼可辨。

**整改要求**：`DialChameleon` 使用 `dialWithUTLS` 建立底层 TCP+TLS 连接，再在其上建立 WebSocket。

### [A-06] Gateway 侧重写无法消除 Client 一侧被动观察者看到的原始出站特征 🟠 P1

**位置**：`mirage-gateway/bpf/bdna.c:260-268`（QUIC）、`bdna.c:326-332`（TLS）

两处都只做了 `skb->mark` 标记，注释写"由用户态完成实际重写"，但用户态重写逻辑不存在。

**关键认知**：Gateway 侧重写无法消除靠近 Client 一侧被动观察者看到的原始出站特征。源头生成正确指纹是主修复路径。这与当前代码完全一致 — Client 出站在 `client.go:307` 直接 `SendDatagram`，中间无任何指纹处理。

**整改要求**：
- **主路径（推荐）**：Client 侧使用 uTLS + quic-go Transport Parameters Hook，从源头生成正确指纹
- **补充路径**：Gateway 侧实现用户态拦截重写（需要 NFQUEUE 或 iptables REDIRECT），仅覆盖 Gateway 出站方向

### [A-07] B-DNA 指纹模板仅 6 个，多样性不足 🟠 P1

**位置**：`mirage-gateway/bpf/bdna.c:57-62`（6 个 eBPF 模板）、`chameleon.go`（3 个 Go 模板）

**被动观察者视角**：长期观察同一 Gateway 的出站流量，JA4 指纹只在 3-6 个值之间切换，聚类分析可快速识别。

**整改要求**：扩展到 30+ 模板，覆盖 Chrome/Firefox/Safari/Edge 各主要版本，每月从真实浏览器采集更新。

### [A-08] NPM 的 C 层仍保留 fixed MTU 模式，但运行时默认值已切到 Gaussian 🟡 Partial / Reframed

**位置**：`mirage-gateway/bpf/npm.c:68-72`

`npm.c` 仍保留 `NPM_MODE_FIXED_MTU` 实现；但按当前 Go 控制面默认值，`NewDefaultNPMConfig()` 已将 `PaddingMode` 设为 `NPMModeGaussian`，loader 也在使用该默认配置写入 map。

**当前结论**：

- "`默认 fixed MTU`" 这个判断已不再准确
- 但 `fixed MTU` 代码路径仍然存在，且尚未完成端到端抓包证明运行时不会回落到该模式

**整改要求**：
1. 保持运行时默认值为 `NPMModeGaussian`
2. 增加启动期/验收期断言，确保 map 实际值没有回落到 `fixed MTU`
3. 抓包确认默认包长分布不存在 1460 尖峰

### [A-09] B-DNA 只重写 SYN 包，后续数据包行为不一致 🟠 P1

**位置**：`mirage-gateway/bpf/bdna.c:128`

**整改要求**：对非 SYN 包也维护 Window Size 一致性（至少在连接前 N 个包内）。

### [A-10] Jitter-Lite 高斯采样实际是均匀分布 🟠 P2

**位置**：`mirage-gateway/bpf/common.h:247`

__u64 z = mean + (u1 % (stddev * 2)) - stddev;  // 均匀分布，不是高斯

**整改要求**：使用 Irwin-Hall 近似（4 个均匀分布求和再缩放）。

## Layer B：外部主动探测强特征

主动探测者向目标发包后，根据回包行为判定的特征。

### [B-01] Gateway 已部分接入公网数据面编排，但真实 QUIC/H3 bearer listener 仍缺失 🔴 P0

当前结论应拆成"已接线的部分"与"仍缺失的部分"：

1. 已接线：`NewChameleonListener()`、`Orchestrator.StartPassive()`、`DataPlane Injector`、`StealthControlPlane.ReceiveLoop()`
2. 仍缺失：生产态公网 `QUIC/H3 listener`、将 `HandshakeGuard/ProtocolDetector` 绑定到该公网数据面、DNS/ICMP/WebRTC bearers 的生产级启动链

因此，本条不应再表述为"生产入口没接"，而应表述为"生产入口只接了 WSS 被动链，默认公网 QUIC/H3 主路径仍不存在"。

拆成四个子项：

#### B-01a：Gateway 主入口未接入 Orchestrator/StealthCP ✅ Closed / Stale

**位置**：`mirage-gateway/cmd/gateway/main.go`

本子项已被代码关闭：

1. `ChameleonListener` 已创建并启动
2. `Orchestrator.StartPassive()` 已进入主链
3. `StealthControlPlane.ReceiveLoop()` 已拉起

后续不再将本项列为整改输入。

#### B-01b：生产态缺少 QUIC/H3 bearer listener，不能响应真实 H3 探测

**位置**：全仓库 `quic.ListenAddr` 只出现在测试入口 `loopback-test/main.go:41`。生产态 `main.go` 没有任何 QUIC/H3 数据面监听器。

**注意**：上一版文档写"启动 ChameleonListener 使 Gateway 能正确响应标准 HTTP/3 探测"，这与代码不符 — `chameleon_fallback.go:29` 的 ChameleonListener 是 WebSocket over TLS 降级监听（HTTP/1.1 + WebSocket Upgrade），不是 H3 listener。

**主动探测者视角**：
- 向 Gateway 端口发送标准 HTTP/3 请求，Gateway 无法响应（因为根本没有 QUIC listener）
- 向 Gateway 发送 QUIC Initial with ALPN=h3，无任何回包，即可判定非标准 HTTP/3 服务

**整改要求**：
1. 在 `main.go` 中创建生产态 QUIC/H3 bearer listener（`quic.ListenAddr` on 443/UDP）
2. 该 listener 必须能正确完成 QUIC 握手（ALPN=h3）并返回合法 HTTP/3 响应（如 403/404）
3. 将 Orchestrator 的数据面绑定到此 listener

#### B-01c：主动探测防护未绑定到未来公网数据面 listener

**位置**：`mirage-gateway/pkg/api/grpc_server.go:89`

当前 ProtocolDetector（line 604）/HandshakeGuard（line 603）只挂在 gRPC listener 上（内部控制面），不在未来公开数据面上。

**整改要求**：
1. 在 B-01b 创建的 QUIC/H3 bearer listener 上也接入主动探测防护
2. 对非法 QUIC Initial（错误 ALPN、异常 Transport Parameters）实施静默丢弃

### [B-02] 配额熔断导致流量瞬间归零 🔴 P0

**位置**：`mirage-gateway/bpf/jitter.c:30-42`

```c
if (*remaining == 0) {
    return TC_ACT_STOLEN;  // 流量瞬间归零
}
```

**主动探测者视角**：持续发送流量，观察到流量在某个精确字节数突然中断（不是渐进衰减），这是极强的行为特征。正常网络不会出现精确到字节的流量截断。

**整改要求**：实现渐进式降级——剩余配额低于阈值时逐步降低通过概率（如剩余 10% 时通过率 50%，剩余 1% 时通过率 10%）。

## 验收标准

修复完成后，必须通过以下验收测试才能判定"外部零特征消除"达标。

验收按两组独立执行：**默认 QUIC 模式**和 **WSS 降级模式**，不混在一起。

### V-01 默认 QUIC 模式 — 被动观察验收（pcap 对比基线）

> 抓包限定条件：Mirage 进程在受控负载下的出流（排除主机背景噪声）。

| 测试项 | 方法 | 通过标准 |
|--------|------|---------|
| QUIC ALPN | 抓取 QUIC Initial，解析 ALPN | 值为 `h3`，不含任何自定义字符串 |
| QUIC Transport Parameters | 抓取 QUIC Initial，解析 TP | `max_idle_timeout`、`initial_max_data`、`max_udp_payload_size` 与 Chrome 140+ 差异 < 5% |
| QUIC CID 行为 | 抓取 QUIC 连接，统计 CID 更新频率 | 与 Chrome 的 CID 更新频率差异 < 20% |
| JA4Q/QUIC 指纹 | 抓取 QUIC Initial，计算 JA4Q | 与目标浏览器（Chrome/Firefox）的 JA4Q 匹配 |
| 包长分布 | 抓取 10 分钟流量，绘制包长直方图 | 不出现单一 MTU 值的尖峰（如 1460），分布应为连续曲线 |
| IAT 分布 | 抓取 10 分钟流量，绘制 IAT 直方图 | 不呈现均匀分布，应近似正态或指数分布 |
| 协议出流 | 抓取 1 小时 Mirage 进程在受控负载下的出流 | 仅 QUIC（UDP 443）出流，不出现 ICMP/DNS tunnel 流量 |

### V-02 WSS 降级模式 — 被动观察验收

| 测试项 | 方法 | 通过标准 |
|--------|------|---------|
| TLS JA3/JA4 指纹 | 抓取 WSS 握手 TLS ClientHello，计算 JA3/JA4 | 与 Chrome 的 JA3/JA4 差异 < 3 个字段 |
| HTTP 头 | 抓取 WebSocket Upgrade 请求 | User-Agent/Accept/Accept-Language 与真实浏览器一致 |
| WebSocket Upgrade 语义 | 检查 Upgrade 握手 | 符合 RFC 6455，无自定义头 |

### V-03 主动探测验收

| 测试项 | 方法 | 通过标准 |
|--------|------|---------|
| HTTP/3 探测 | 向 Gateway 端口发送标准 HTTP/3 请求 | 返回合法 HTTP 响应（如 403/404），不暴露非标准行为 |
| ALPN 协商 | 发送 QUIC Initial with ALPN=h3 | 正常协商完成，不拒绝连接 |
| 配额耗尽行为 | 持续发送流量直到配额耗尽 | 流量渐进衰减（>5 秒过渡期），不出现瞬间归零 |
| 防御升级行为 | 逐步升级探测强度 | 防御参数变化不呈现离散阶梯（同等级内有随机偏移） |
| 域名切换行为 | 触发 G-Switch 转生 | 双发窗口时长不固定（80-200ms 范围），无精确 100ms 峰值 |
| fallback 行为 | 阻断 QUIC 后观察降级 | WSS 降级握手的 JA4 与 Chrome 一致 |

### V-04 统计学验收（长期观察）

| 测试项 | 方法 | 通过标准 |
|--------|------|---------|
| 周期性检测 | 对 24 小时流量做 FFT | 不出现固定频率峰值（如 5s、10s、60s 周期） |
| 时段边界 | 观察整点前后 30 分钟的延迟变化 | 延迟变化为渐变（sigmoid），不出现阶跃 |
| 指纹多样性 | 统计 1 周内的 JA4 指纹种类 | 至少出现 10 种不同指纹 |
