# Mirage V2 升级核心纪要：编排内核

## 一、文档性质

本文档用于完整保存本轮关于 Mirage V2 升级方向的核心讨论结果。

这不是普通模块说明，也不是单点优化建议，而是 Mirage 从“六协议协同系统”升级为“可运营、可回滚、可约束的编排系统”的上位设计纪要。

后续所有与 V2 相关的：

- 架构设计
- 状态模型
- 协议协同
- 资源预算
- 排期拆解
- 文档输出

都应默认以本文作为方向依据。

---

## 二、V2 升级的核心结论

Mirage V2 的重点，不再是继续堆协议，也不是继续增加局部对抗技巧。

V2 的真正升级目标是：

**把 Mirage 从“六协议能力集合”，升级为“围绕会话状态、画像快照、预算约束和原子提交运行的编排系统”。**

更具体地说：

- V1 解决的是“能不能连、能不能伪装、能不能活”
- V2 解决的是“这些能力能不能以一致、可控、可回滚的方式稳定运行”

因此，V2 的关键词不是：

- 更多协议
- 更多模板
- 更多参数

而是：

- 状态
- 提交
- 预算
- 回滚
- 编排

---

## 三、V2 为什么必须升级

当前 Mirage 已经具备六协议协同的基础：

- `G-Tunnel`
- `B-DNA`
- `NPM`
- `G-Switch`
- `Jitter-Lite`
- `VPC`

但当前系统的主要问题已经不再是“有没有能力”，而是“这些能力如何一起工作”。

本轮讨论确认了 Mirage 当前的四个结构性上限。

### 3.1 局部合理，但整体可能撕裂

当前传输、指纹、包长、时序、背景噪音虽然都能调，但它们之间仍然更像多个参数系统，而不是围绕同一个统一画像工作的单一系统。

结果就是：

- 某个握手像一种客户端
- 某种包长像另一类业务
- 某种时序又像第三类业务
- 背景噪音再像第四种网络环境

单点看都合理，合在一起就可能显得不自然。

### 3.2 生存能力仍偏单点切换

当前的 `G-Switch` 更接近：

- 入口失效
- 切域名
- 必要时联动部分重置

这仍然偏“局部生存能力”，而不是“全局生存姿态控制”。

### 3.3 策略引擎仍偏线性调参

当前系统已经能根据威胁等级提高：

- jitter
- padding
- noise

但这更像线性增强，而不是围绕目标函数和成本约束做真正决策。

### 3.4 缺少事务观和回滚能力

当前如果多个模块分别更新，就容易出现：

- 某一半配置生效
- 另一半未生效
- 进程中断时状态停在中间
- 会话和链路状态不同步

这会直接破坏：

- 稳定性
- 一致性
- 可解释性
- 可运营性

---

## 四、V2 的总体升级方向

本轮讨论最终收敛出的 V2 总方向是：

**从六协议协同系统，升级为三层编排系统。**

这三层分别是：

1. `Transport Fabric`
2. `Persona Engine`
3. `Survival Orchestrator`

---

## 五、三层编排系统

### 5.1 Transport Fabric

这一层对应 Mirage 当前的传输骨架，核心代表是 `G-Tunnel`。

它负责解决：

- 请求如何被装入隧道
- 数据如何被拆分、纠错、重组
- 当前走哪条承载路径
- 底层链路如何切换
- 降级时使用哪种承载方式

这一层关心的是：

- 连通性
- 传输质量
- 路径健康
- 底层切换成本

它不负责定义“看起来像谁”，只负责“数据能否稳定送达”。

### 5.2 Persona Engine

这一层对应当前的伪装系统集合：

- `B-DNA`
- `NPM`
- `Jitter-Lite`
- `VPC`

V2 的核心结论是：

这四个协议不应该再作为四套独立参数运行，而应该收敛成一个统一的 `Persona`。

一个 Persona 不是一个标签，而是一份完整画像快照，至少应包含：

- 握手/栈指纹配置
- 包长形态配置
- 时域特征配置
- 背景噪音配置
- 版本号
- Epoch
- 校验值

也就是说：

**对外不再是“开了四个协议”，而是“当前会话正在使用某个统一画像”。**

### 5.3 Survival Orchestrator

这一层对应当前 `G-Switch` 的升级方向。

V2 不再把生存问题理解为单一域名切换，而是理解为：

**系统在不同风险状态下，应该整体进入什么生存姿态。**

建议把生存姿态建模为状态机，例如：

- `Normal`
- `LowNoise`
- `Hardened`
- `Degraded`
- `Escape`
- `LastResort`

每种模式决定的，不只是入口切换，而是：

- 当前 transport 策略
- 当前 persona 是否冻结或切换
- 当前资源预算是否放宽
- 当前是否允许高成本维持
- 当前是否进入保守运行方式

## 六、V2 的四个核心机制

这是本轮讨论的真正核心。

### 6.1 三层状态模型

Mirage V2 必须显式建模三层状态：

#### Layer 1：Link State

描述物理或承载链路本身：

- 当前 transport 类型
- 当前 gateway / entry
- RTT / loss / jitter
- 当前链路是否降级
- 当前链路健康分

#### Layer 2：Session State

描述用户业务会话：

- `session_id`
- `user_id`
- `client_id`
- 当前服务等级
- 当前 persona
- 当前可用预算
- 当前优先级

#### Layer 3：Control State

描述系统如何进行编排和切换：

- 当前 `epoch`
- 当前 `persona_version`
- 当前 `route_generation`
- 最近一次切换原因
- 当前是否处于迁移中
- 当前是否存在 rollback marker

这三层拆开后，Mirage 才能做到：

- 链路切换不等于会话消失
- 会话策略不等于控制流程
- 状态变化可以被审计、解释和回滚

### 6.2 Persona Manifest 与原子切换

V2 的核心原则之一是：

**Persona 切换必须像事务一样切，而不是分散改多个 Map。**

建议做法是：

- 先在控制面组装完整 `Persona Manifest`
- 写入 shadow state
- 完成 checksum / version 校验
- 再通过一次原子 flip 切换生效版本

这样可以避免：

- TLS 指纹已经切换
- 但包长策略仍停留旧版本
- 时序和背景策略又落后半拍

也就是避免“特征撕裂”。

V2 这里的目标不是某个具体内核技巧，而是这 4 个结果：

- 原子切换
- 版本化
- 可回滚
- 可校验

### 6.3 双层预算模型

本轮讨论明确确认：

Mirage V2 的预算不能只停留在技术参数上，必须同时表达：

1. 系统内部真实资源成本
2. 对外服务等级和计费策略

建议拆成两层：

#### Internal Cost Model

描述系统内部真实消耗：

- 带宽预算
- 延迟预算
- 切换预算
- 域名损耗预算
- Gateway 负载预算

#### External SLA / Service Policy

描述用户可获得的服务级别：

- Standard
- Priority
- Hardened Access
- Dedicated Survival Window

这里的核心原则是：

**内部成本不直接裸映射成外部价格，而是先转成服务策略，再由服务策略决定是否允许消耗该预算。**

### 6.4 State Commit Engine

这是 V2 最重要的机制。

Mirage V2 必须把关键状态变更视为“提交事件”，而不是普通参数更新。

典型变更包括：

- persona switch
- transport migration
- gateway reassignment
- survival mode change
- billing / SLA mode shift

建议统一走标准提交流程：

1. `Prepare`
2. `Validate Constraint`
3. `Shadow Write`
4. `Flip`
5. `Acknowledge`
6. `Commit or Rollback`

这一机制的意义是：

- 防止半切换状态
- 防止控制面崩溃后节点变成死节点
- 防止状态漂移无法解释
- 让系统具备数据库级别的事务观

## 七、V2 的工程原则

为了防止 V2 又退回到“继续堆协议”的路线，本轮讨论还确认了以下原则。

### 7.1 先做状态与提交，再谈协议扩张

如果没有：

- 三层状态模型
- Persona Manifest
- State Commit Engine

那么再加协议，只会让系统更复杂，不会更成熟。

### 7.2 先做一致性，再做高级切换

V2 的第一目标不是切得更多，而是切得一致、切得可解释、切得能回滚。

### 7.3 预算必须约束策略

Mirage 不是无限资源系统。

任何：

- 加强 FEC
- 提高 padding
- 增加 noise
- 高频切换
- 域名逃逸

都应视为消耗预算的动作，而不是“有威胁就无脑加码”的动作。

### 7.4 生存姿态高于单点动作

V2 不应该把“切域名”“换通道”“换画像”分别看待，而应该把它们放进统一的 `Survival Mode` 状态机里。

### 7.5 控制语义必须存在，但承载方式服从现有可靠传输

本轮讨论明确否定了“新增显眼外部控制协议”的方向。

V2 后续应先定义：

- 控制事件类型
- 一致性要求
- 优先级要求
- 回放保护要求
- 确认与失败处理要求

即先定义控制语义层，而不是先讨论外部承载技巧。

## 八、V2 的实施优先级

本轮讨论最终确认的优先级如下。

### P0：必须先完成

#### P0-1：三层状态模型

- `Link State`
- `Session State`
- `Control State`

#### P0-2：Persona Manifest

- Persona 结构定义
- 版本号
- Epoch
- 校验值
- 原子切换语义

#### P0-3：State Commit Engine

- Prepare / Validate / Shadow / Flip / Ack / Rollback
- 回滚标记
- 提交完成标记
- 恢复时重放或撤销逻辑

### P1：在 P0 完成后推进

#### P1-1：Internal Cost Model

- 带宽预算
- 延迟预算
- 切换预算
- 入口消耗预算

#### P1-2：External SLA / Service Policy

- 服务等级
- 是否允许进入高成本生存模式
- 预算上限与核销规则

#### P1-3：Survival Mode 状态机

- `Normal`
- `LowNoise`
- `Hardened`
- `Degraded`
- `Escape`
- `LastResort`

### P2：在状态和提交流程稳定后推进

#### P2-1：控制语义层

- 事件定义
- 优先级
- 幂等语义
- 回放保护

#### P2-2：观测与审计

- 为什么切换
- 为什么回滚
- 当前运行在哪个 persona
- 当前会话挂在哪个 link

#### P2-3：协议删减或补强评估

到这一阶段，才有资格判断：

- 现有协议是否还需要增加
- 哪些协议该退居兜底
- 哪些路径不值得长期维护

---

## 九、V2 完成后的系统形态

如果本轮讨论确认的 V2 路线真正落地，Mirage 会发生质变。

它将不再只是：

- 一个协议组合系统
- 一个靠策略调参生存的工具
- 一个高强度但脆弱的技术堆栈

而会变成：

- 一个围绕会话运行的编排系统
- 一个能统一管理画像、传输和生存姿态的控制系统
- 一个能以提交和回滚机制维护一致性的高可用系统
- 一个既能控制技术行为，也能控制经营成本的运营系统

换句话说：

**Mirage V2 的目标不是“更花哨”，而是“更像一套真正可长期运行的基础设施”。**

---

## 十、V2 的一句话总结

本轮讨论最终确认的 V2 核心可以压缩成一句话：

**Mirage V2 不再继续堆叠协议能力，而是围绕状态模型、画像快照、预算约束和原子提交，建立一套真正可运营、可回滚、可解释的编排内核。**

这就是 Mirage V2 升级的核心方向。
