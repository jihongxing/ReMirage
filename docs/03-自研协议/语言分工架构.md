# 六大协议语言分工架构

## 一、核心原则

**C 做内核数据面，Go 做控制面**

- **C 语言**：纳秒级控制、零拷贝、内核态操作
- **Go 语言**：策略计算、API 调用、高层逻辑

---

## 二、语言分配表

| 协议 | 核心实现语言 | 理由 |
|------|-------------|------|
| **NPM** | C (eBPF XDP) | 网卡层零拷贝 Padding，避免用户态内存拷贝 |
| **B-DNA** | C (eBPF TC) | TCP 握手阶段修改 Header/Window Size，需要内核态操作 |
| **VPC** | C (eBPF TC) | 纳秒级定时器模拟光缆抖动，控制 skb->tstamp |
| **Jitter-Lite** | C (eBPF TC) | 控制 skb->tstamp 实现绝对精准 IAT，硬件 FQ 调度 |
| **G-Tunnel** | Go 控制 + C 数据面 | Go 负责路径调度，C 负责包拆分与 FEC（AVX-512） |
| **G-Switch** | Go (M.C.C.) | API 调用、数据库更新、Raft 一致性等高层逻辑 |
| **WebRTC** | Go (pion/webrtc) | DTLS+SCTP DataChannel，SDP 信令通过 WSS 交换 |
| **ICMP Tunnel** | C (eBPF TC) + Go 控制面 | C 内核态构造/截获 ICMP 包，Go 通过 eBPF Map + Ring Buffer 通信 |
| **DNS Tunnel** | Go (miekg/dns) | Base32 编码子域名查询 + 权威 DNS 服务器响应 |

## 四、通信机制

### 4.1 Go → C（控制指令）

```go
// Go 控制面
type EBPFController struct {
    maps map[string]*ebpf.Map
}

// 下发威胁等级到 C 数据面
func (ec *EBPFController) SetThreatLevel(level int) error {
    key := uint32(0)
    value := uint32(level)
    
    // 写入 eBPF Map
    return ec.maps["threat_level_map"].Put(key, value)
}

// 更新 NPM 填充配置
func (ec *EBPFController) UpdatePaddingConfig(cfg PaddingConfig) error {
    return ec.maps["padding_config_map"].Put(cfg.FlowID, cfg)
}
```

### 4.2 C → Go（数据上报）

```c
// C 数据面
struct threat_event {
    __u64 timestamp;
    __u32 threat_type;
    __u32 source_ip;
};

// Ring Buffer 上报威胁事件
struct {
    __uint(type, BPF_MAP_TYPE_RINGBUF);
    __uint(max_entries, 256 * 1024);
} threat_events SEC(".maps");

SEC("xdp")
int detect_threat(struct xdp_md *ctx) {
    // 检测到威胁
    struct threat_event *event = bpf_ringbuf_reserve(&threat_events, sizeof(*event), 0);
    if (!event)
        return XDP_PASS;
    
    event->timestamp = bpf_ktime_get_ns();
    event->threat_type = THREAT_ACTIVE_PROBING;
    event->source_ip = get_source_ip(ctx);
    
    bpf_ringbuf_submit(event, 0);
    return XDP_PASS;
}
```

```go
// Go 控制面接收
func (ec *EBPFController) MonitorThreats() {
    reader, _ := ringbuf.NewReader(ec.maps["threat_events"])
    
    for {
        record, _ := reader.Read()
        
        var event ThreatEvent
        binary.Read(bytes.NewReader(record.RawSample), binary.LittleEndian, &event)
        
        // 处理威胁事件
        ec.handleThreat(&event)
    }
}
```

## 七、关键技术点

### 7.1 eBPF Map 类型选择

| 场景 | Map 类型 | 理由 |
|------|---------|------|
| 配置下发 | HASH | 支持动态更新 |
| 事件上报 | RINGBUF | 高性能无锁队列 |
| 统计数据 | ARRAY | 固定索引快速访问 |
| 流量映射 | LRU_HASH | 自动淘汰旧条目 |

### 7.2 内存管理

**C 数据面**：
- 使用 eBPF 栈内存（512 字节限制）
- 大对象使用 Map 存储
- 避免动态内存分配

**Go 控制面**：
- 使用 sync.Pool 复用对象
- 避免频繁 GC
- 使用 mmap 共享内存

### 7.3 错误处理

**C 数据面**：
```c
// 失败时返回默认行为
if (!config) {
    return XDP_PASS; // 不处理，放行
}
```

**Go 控制面**：
```go
// 失败时降级
if err := ec.UpdateConfig(cfg); err != nil {
    log.Warn("更新配置失败，使用默认配置")
    ec.UseDefaultConfig()
}
```

## 九、编译流程

### 9.1 C 数据面编译

```bash
# 编译 eBPF 程序
clang -O2 -target bpf -c npm.c -o npm.o
clang -O2 -target bpf -c bdna.c -o bdna.o
clang -O2 -target bpf -c jitter.c -o jitter.o
clang -O2 -target bpf -c vpc.c -o vpc.o

# 生成 Go 绑定
go run github.com/cilium/ebpf/cmd/bpf2go -cc clang npm bpf/npm.c
```

### 9.2 Go 控制面编译

```bash
# 编译 Gateway
cd mirage-gateway
go build -o bin/mirage-gateway cmd/gateway/main.go

# 静态链接（生产环境）
CGO_ENABLED=0 go build -ldflags="-s -w" -o bin/mirage-gateway cmd/gateway/main.go
```

## 十、总结

### 10.1 分工优势

1. **性能最优**：C 内核态处理数据包，延迟 < 1ms
2. **开发高效**：Go 处理复杂逻辑，开发时间 -50%
3. **可维护性**：清晰的职责分离，易于调试
4. **资源占用**：CPU < 20%，内存 < 250MB

### 10.2 关键指标

| 指标 | 纯 Go 实现 | C + Go 混合 | 提升 |
|------|-----------|------------|------|
| 延迟 | 50ms | 15ms | 70% |
| CPU | 35% | 18% | 49% |
| 内存 | 500MB | 220MB | 56% |
| 吞吐 | 500 Mbps | 1.2 Gbps | 140% |
