# Mirage Project

基于内核加速与多路径传输能力构建的高可用网络基础设施

Mirage 是一个面向复杂网络环境的通信基础设施项目。它不只是一个代理程序，而是一整套由网关、控制中心、客户端、命令行工具和 SDK 组成的系统，目标是在网络不稳定、限制较多或对隐私和稳定性要求较高的场景中，提供更平稳、更隐蔽、也更容易恢复的连接能力。

简单来说，Mirage 想解决的不是“能不能连上”，而是“在复杂环境里，能不能长期稳定、尽量不显眼地连着”。

---

## 这是什么

传统网络代理工具通常更强调“把内容加密传出去”。这当然很重要，但在很多现实环境里，真正导致连接不稳定的，往往不是内容是否加密，而是通信方式本身是否容易被识别、干扰或限制。

Mirage 的思路是把这件事往前再做一步：

- 不只保护内容，还尽量降低连接本身的可识别性
- 不只建立单一路径，还尽量准备多种可切换的连接方式
- 不只追求能用，还强调稳定、恢复能力和统一管理

所以它更像是一套“网络基础设施”，而不是一个单点工具。

---

## Mirage 能做什么

### 更稳定

Mirage 会尽量避免把所有通信都押在单一路径或单一种连接方式上。当网络环境发生波动时，系统可以根据当前状态做调整，尽量保证业务不中断，或者把中断时间降到更短。

### 更不容易被识别

Mirage 关注的不只是数据内容，也包括流量在外部看起来的表现形式。它的目标不是“显得特殊”，而是尽量“不要太显眼”，从而降低因为通信特征过于固定而被识别和干扰的概率。

### 更容易恢复

在入口变化、网络质量下降或部分节点失效的情况下，Mirage 会尽量通过预设的恢复机制继续工作，减少人工频繁改配置、换节点、重建连接的成本。

### 更容易管理

Mirage 包含控制中心、Web 控制台、CLI 和 SDK，方便团队统一查看状态、管理节点、做诊断和接入上层系统，而不是把所有工作都压在手工维护上。

---

## 适合什么场景

Mirage 更适合以下这类场景：

- 对连接稳定性要求很高的跨区域通信
- 容易出现网络波动、限速或不确定性的环境
- 需要统一管理多个节点和客户端的团队化部署
- 对隐私、日志留存、运行痕迹较敏感的系统

如果你的需求只是本地临时代理或非常轻量的单机场景，那么 Mirage 可能会显得偏重；它更适合中长期、可运维、可扩展的部署方式。

## 项目组成

Mirage 当前仓库主要包含以下几个部分：

- `mirage-gateway/`
  网关侧核心程序，负责承接流量、执行策略并与控制中心协同工作。

- `mirage-os/`
  控制中心，提供统一管理、状态查看、策略下发和服务协同能力。

- `phantom-client/`
  客户端程序，负责接入、连接维护与异常恢复。

- `mirage-cli/`
  命令行工具，适合运维排查、状态查询和日常管理。

- `sdk/`
  多语言 SDK，方便把 Mirage 的能力集成到其他系统中。

- `deploy/`
  部署脚本、环境配置和运维资产。

- `docs/`
  更完整的架构、协议、实施与设计文档。

## 三个核心组件

Mirage 最关键的，不只是“有很多目录”，而是它由三个相互配合的核心组件组成：`Mirage OS`、`Mirage Gateway` 和 `Phantom Client`。如果把整个系统看成一张持续运行的网络，那么这三者分别负责“大脑”“骨干节点”和“终端入口”。

### Mirage OS

`Mirage OS` 是整个系统的控制中心。它的价值不在于直接承载业务流量，而在于把原本分散、难维护的节点和策略统一起来。

它主要承担这些能力：

- 统一管理节点、状态和配置
- 为网关提供调度、策略和协同能力
- 提供 Web 控制台、接口和命令行配套能力
- 帮助运维人员查看系统健康状态、连接状况和整体运行情况
- 作为整个网络的“管理平面”，让部署从单点工具变成可运营的系统

如果没有 `Mirage OS`，Mirage 更像是若干孤立节点；有了 `Mirage OS`，它才更接近一套可统一管理、可持续运行的基础设施。

### Mirage Gateway

`Mirage Gateway` 是 Mirage 真正在网络中工作的核心节点。它位于系统中间层，负责承接客户端连接、处理转发、执行策略，并在复杂网络环境中尽量维持通信的稳定和连续。

它可以理解为整个系统的“骨干与前线”：

- 负责流量接入与转发
- 根据当前网络状态做连接层面的调整
- 尽量降低链路被干扰后造成的直接中断
- 与控制中心保持协同，执行来自上层的策略和调度
- 在性能、稳定性和隐蔽性之间做平衡

简单说，`Gateway` 决定的是“这条连接在网络里怎么活下去、怎么跑得更稳”。

### Phantom Client

`Phantom Client` 是面向最终使用侧的客户端组件。它负责把用户设备或本地网络接入到 Mirage 体系中，并在连接建立之后尽量维持稳定、完成恢复和降低人工干预。

它的核心价值主要体现在：

- 负责用户侧接入
- 负责连接建立、保活和切换
- 在网络波动或入口变化时尽量自动恢复
- 降低用户频繁改配置、换入口、重新连接的成本
- 让整个系统不仅“后端强”，前端使用体验也尽量稳定

如果说 `Mirage OS` 更偏管理，`Mirage Gateway` 更偏网络节点，那么 `Phantom Client` 就更偏使用入口与终端体验。

### 三者如何配合

这三个组件不是并列摆放的，而是各自承担不同职责：

- `Mirage OS` 负责统一管理和协同
- `Mirage Gateway` 负责在网络中实际工作
- `Phantom Client` 负责把用户接入这套系统

它们组合起来，Mirage 才不是一个单独的代理程序，而是一套从管理、到传输、再到终端接入都相互衔接的完整系统。

```text
          控制中心
   管理 / 状态 / 策略 / 控制台
                │
                │
```

## 和普通代理工具有什么不同

| 维度 | 普通代理工具 | Mirage |
|------|--------------|--------|
| 关注点 | 建立一条可用连接 | 建立一套可持续运行的连接体系 |
| 连接方式 | 通常较单一 | 更强调多路径与恢复能力 |
| 管理方式 | 多依赖手工维护 | 提供控制中心、CLI、SDK |
| 部署思路 | 偏单点工具 | 偏基础设施和系统化部署 |
| 适用周期 | 临时或轻量使用 | 中长期、可运维场景 |

Mirage 不一定是“最简单”的方案，但它希望成为“更稳、更像基础设施”的方案。

## 项目特点

为了让 README 更适合对外阅读，这里只保留高层说明，不展开实现细节。你可以把 Mirage 的特点概括为下面几点：

- 强调稳定性：面对网络波动时更有韧性
- 强调隐蔽性：尽量降低过于固定的通信特征
- 强调恢复能力：节点变化或链路异常时更容易自愈
- 强调运维能力：适合团队化管理和长期运行
- 强调安全性：对运行时保护、敏感数据处理和最小痕迹部署有较多考虑

## 快速开始

如果你只是想快速把项目跑起来，可以按下面的方式理解：

### 1. 启动控制中心

```bash
cd mirage-os
docker compose up -d --build
```

### 2. 构建并启动网关

```bash
cd mirage-gateway
make all
sudo ./bin/mirage-gateway -config configs/gateway.yaml
```

### 3. 启动客户端

```bash
cd phantom-client
go run ./cmd/phantom -token <bootstrap_token>
```

### 4. 查看状态

```bash
cd mirage-cli
go run . status
```

如果你准备做正式部署，建议先阅读：

- `DEPLOYMENT.md`
- `mirage-os/DEPLOYMENT.md`
- `docs/`
